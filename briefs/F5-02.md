## Входы и факты
- Канон §4 T1 фиксирует четыре исхода `/create_order` (success / partial / rejected / duplicate),
  §4 T2 — «ровно 4» события 1С→CRM, §5 — ключи идемпотентности (INV-5). Канон правится только ADR.
- Волна 1 УЖЕ добыла (не переоткрывать, использовать как вход):
  · P-02 — эхо подтверждено для /cancel_reserve; /update_order статус не меняет; `order_completed`
    load-bearing (CRM потребляет); подавление эха уже реализовано на стороне CRM (анти-петля);
    диспозиция — документационный фикс, 0 кода 1С.
  · P-03/P-05 — межтиповой упорядоченности нет, реквизита версии документа в 1С нет.
  Значит по P-17 задача сводится к дельте: добавляет ли P-17 что-то сверх диспозиции P-02, или
  SUPERSEDED→P-02. Дельту доказывать цитатами, не рассуждением.
- OQ-03 (OPEN, defer post-MVP): целевой рефактор — транзакционная очередь исходящих. P-16 —
  источник этого вопроса. В этом брифе P-16 верифицируется ТОЛЬКО как текст ТЗ (описан ли
  синхронный HTTP в ПриЗаписи, ретраи, гарантия доставки, порядок). Очередь НЕ проектируется.
- К-7 — ключевое для P-26/P-27: это гипотезы о коробочном поведении ТКПТ. Если текст ТЗ не
  отвечает — это НЕ REFUTED, это ESCALATE → вопрос к List.kg. Отсутствие в ТЗ означает
  «подрядчик не обязан», а не «система так не делает».
- P-17 содержит предложение правила «повтор = 200, невозможное действие = 409». Это архитектурное
  решение о контракте, а не находка. Если правила нет в корпусе — фиксируй отсутствие цитатой и
  выноси в «требует ADR». Не объявляй его действующим.
- ADR-005: критерий Track B. Всё, что меняет контракт эндпоинтов по существу, — Track A.

## Шаги
1. Дословно выписать формулировки шести P-xx из source (с номерами строк).
2. Собрать фактуру по привязке выше; зафиксировать `файл@SHA:строки`.
3. Присвоить статус каждой P-xx: CONFIRMED-DOC | CONFIRMED-FACT | REFUTED | SUPERSEDED | ESCALATE.
4. Отдельным блоком — OQ-04: собрать ПОЛНЫЙ перечень исходящих событий 1С→CRM, встречающихся в
   корпусе, каждое с цитатой и указанием источника подписки (ПриЗаписи / ПослеПроведения / иное) и
   потребителя на стороне CRM. Сопоставить с каноном §4 T2. Вывод по расхождению НЕ формулировать
   как решение — только «канон говорит X, корпус даёт Y, цитаты такие-то» → ESCALATE.
5. По P-30 — заполнить таблицу исходов `/create_order` целиком по тексту: для каждого значения
   `status` — создаётся ли документ, ставится ли резерв, возвращается ли guid/номер, заполняется ли
   `missing[]`, что делает повторный вызов с тем же `CRM_Deal_ID` (INV-5). Пустые клетки оставлять
   пустыми с пометкой «в тексте нет» — не заполнять логикой.
6. Для CONFIRMED-DOC — черновик задачи подрядчику (только устранение противоречия/неопределённости
   текста). Для фактовых пробелов — черновики CQ к List.kg.

- Все шесть: статус + ≥1 цитата `файл@SHA:строки`; REFUTED — тоже с цитатой.
- Таблица исходов `/create_order` заполнена по тексту; каждая непустая клетка имеет цитату.
- Инвентарь событий 1С→CRM: перечень с цитатами + явное сопоставление с каноном §4 T2.
