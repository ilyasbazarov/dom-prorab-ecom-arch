## Входы и факты
- ADR-002: объект анализа — корпус D1–D6 как immutable-снапшоты. ADR-006: единица цитирования — md-фрагмент.
- ADR-004: P-xx мигрируют как UNVERIFIED; этот бриф выполняет миграцию+верификацию для шести.
- ADR-005 (критерий Track B): исправление НЕ меняет бизнес-инвариантов, только устраняет внутреннее
  противоречие текста. Проблема, нарушающая инвариант, — НЕ Track B → ESCALATE в Track A.
- К-1: без цитаты файл@SHA:строки статус CONFIRMED невозможен. Пересказ ≠ доказательство.
- К-7: CONFIRMED-DOC = противоречие/факт виден в тексте ТЗ. Отсутствие в ТЗ = «подрядчик не обязан»,
  а НЕ «система так не делает» → это НЕ REFUTED, это граница doc-оракула.
- Цитировать по SHA своей сессии; идентичность байт гарантируется sha256 в MANIFEST.

## Шаги
1. Для каждой P-xx: выписать формулировку из источника ДОСЛОВНО (фактура до суждения).
2. По MANIFEST определить вкладку(и), релевантные предмету проблемы; curl фрагмент(ы)@SHA.
3. Найти точные строки, подтверждающие/опровергающие противоречие; зафиксировать файл@SHA:строки.
4. Присвоить статус:
   - CONFIRMED-DOC — противоречие внутри текста ТЗ доказано цитатой;
   - REFUTED — текст непротиворечив / противоречие снято другой цитатой (тоже с файл@SHA:строки);
   - ESCALATE — для вывода требуется поведенческий факт 1С (List.kg) ИЛИ затронут бизнес-инвариант.
     Не форсировать, не догадываться.
5. Проверить критерий Track B (ADR-005) по каждой: инвариант не тронут. Иначе → ESCALATE (Track A).
6. Для CONFIRMED-DOC: черновик задачи подрядчику (адресат 1С/Bitrix/BI), scope — только приведение
   текста ТЗ в согласованность.
