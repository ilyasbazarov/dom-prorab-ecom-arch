## Входы и факты
- ADR-013 (ACCEPTED): `l0-canon/canonical_order_flow.md` — L0-канон. §3 содержит каноническую
  таблицу «стадия CRM → ЛК» (10 строк с провенансом). К-2: конфликт L0↔L3 трактуется как дефект
  документации подрядчика. Канон в этом брифе НЕ правится ни при каких находках.
- Канон уже разделяет две нехватки: «Требует уточнения (Нехватка) — partial» и
  «Уточнение состава — oos_at_picking (мутация)» — это прямо касается P-12.
- INV-2: 1С — мастер операционной истины; §3 канона закрепляет стадию сделки за CRM.
  Если в корпусе найдётся эндпоинт вида `/set_status` или прямая запись стадии из 1С — это
  не дефект формулировки, а нарушение владения → ESCALATE (Track A), не Track B.
- ADR-005 (критерий Track B): исправление не меняет бизнес-инвариантов, только устраняет
  внутреннее противоречие текста. Иначе — Track A.
- К-1: без цитаты `файл@SHA:строки` статус CONFIRMED невозможен.
- К-7: отсутствие в ТЗ = «подрядчик не обязан», а НЕ «система так не делает». Для P-08 реальный
  список `STATUS_ID` — факт портала, а не текста: текстом закрывается только рассинхрон источников.
- Канон §7: ЛК-трекинг отнесён к Phase 2 (MVP — WA/SMS). Для P-18 это влияет на scope
  исправления, но НЕ на статус верификации.
- OQ-02 (DEFERRED): платёжная механика pre-auth/charge/refund-SLA. Для P-23 это значит:
  «кто создаёт платёжную ссылку и возникает ли нативный заказ Bitrix» — верифицируется;
  всё, что упирается в механику эквайринга, помечается DEFER→OQ-02, не додумывается.

## Шаги
1. Для каждой P-xx выписать формулировку из problem_register.source.md ДОСЛОВНО (с номерами строк).
2. По MANIFEST подобрать вкладки (старт — привязка выше); curl фрагменты; найти строки за/против.
3. Присвоить статус: CONFIRMED-DOC | CONFIRMED-FACT | REFUTED | SUPERSEDED | ESCALATE.
4. Отдельной колонкой — сверка с каноном: «соответствует §3 (строка канона)» /
   «противоречит §3 (строка канона + цитата корпуса)» / «канон не покрывает».
5. Проверить критерий Track B (ADR-005) по каждой; нарушение инварианта → Track A.
6. Для CONFIRMED-DOC — черновик задачи подрядчику, scope = только приведение текста в согласованность
   с каноном §3 (номера строк канона в тексте задачи обязательны).
7. Для пробелов, не закрываемых текстом (в первую очередь фактический список стадий воронки по P-08)
   — черновик записи в Contractor Question Register (адресат: Bitrix-подрядчик; предмет:
   выгрузка `crm.dealcategory.stage.list` с боевого портала).

## Критерии приёмки
- Все пять: статус ∈ {CONFIRMED-DOC, CONFIRMED-FACT, REFUTED, SUPERSEDED, ESCALATE}, каждый —
  ≥1 цитата `файл@SHA:строки`. REFUTED тоже требует цитату (текст, снимающий противоречие).
- Все пять: вердикт против канона §3 с указанием строки канона.
- REGISTER_PATCHES для `l2-registers/problem_register.md`: пять строк в формате существующей
  таблицы (ID · Статус · Риск · Доказательства · Адресат · Track · Трассировка=F5-01), готовые к append.
- Ни одного статуса без цитаты; ни одной правки канона; любой пробел → CONTEXT GAP, стоп.
