## P-01. «HTTP 200 = резерв поставлен» — ТЗ противоречит само себе

**Уверенность:** 🟢 Подтверждено
**Кого касается:** Битрикс, 1С
**Статус:** решение принято, нужна правка ТЗ

**Суть.** В разделах «Общие архитектурные ограничения» и «Гарантии 1С» написано: «HTTP 200 = строго факт проведения документа в 1С с записью в регистр резервов [DEC-33]». В таблицах обработки ответов того же ТЗ указано, что при HTTP 200 статус в теле может быть `partial` или `rejected` — то есть заказа нет.

**Откуда растут ноги.** Аналитик подтвердил прямо, с указанием разделов и DEC-33. Цитата из его ответа: «Утверждение „HTTP 200 = резерв поставлен — ложь“ верно».

**Риск.** Если робот проверяет только HTTP-код — сделка уедет в «Ожидает сборки» при нехватке товара. Кладовщик получит задание на несуществующий заказ.

**Решение.** Триггер `PICKING` = `HTTP 200` И `body.status == "success"` И `body.purchase_order_guid != null`. Формулировку в ТЗ заменить на: «HTTP 200 — техническое подтверждение доставки запроса. Бизнес-результат определяется строго по полю `status` в теле ответа».

**Открыто:** реализовано ли уже по-старому. Вопрос Б24-4.

## P-02. Подавление вебхуков по `УФ_CRM_Deal_ID` отключит все события

**Уверенность:** 🟢 Подтверждено (что предложение сломано)
**Кого касается:** 1С
**Статус:** предложение, ещё не реализовано → правится бесплатно

**Суть.** Аналитик предложил подавлять исходящие вебхуки в 1С по наличию реквизита `УФ_CRM_Deal_ID`. Но этот реквизит заполнен у **каждого** e-com заказа. Подавление по нему заглушит все события: ручную отмену менеджером, отгрузку, нехватку при сборке.

**Откуда растут ноги.** Прямая цитата аналитика: «Использовать наличие этого ID (или явный флаг `ИсточникИзменения`) как условие для подавления вебхука в 1С». Первая половина фразы сломана, вторая — правильная. Вывод, что реквизит заполнен у всех e-com заказов, — из логики его назначения (связь с CRM), а не из цитаты. Но других вариантов нет: если бы он был не у всех, интеграция бы не работала.

**Причина ошибки.** `УФ_CRM_Deal_ID` — свойство **документа** (кем создан). Источник изменения — свойство **транзакции** (кто пишет прямо сейчас). Из первого второе не выводится.

**Решение.** `ПараметрыСеанса.ИсточникИзменения`, выставляется на входе HTTP-сервиса, читается при формировании события.

**Хорошая новость.** Аналитик сам написал: «Статус в ТЗ: сейчас это отсутствует». То есть подавление ещё не реализовано. Правим бесплатно.

## P-03. Правило версионности не работает

**Уверенность:** 🟢 Подтверждено
**Кого касается:** Битрикс
**Статус:** предложение, ещё не реализовано

**Суть.** Предложено: «игнорировать вебхук, если в базе уже есть запись с этим `event` и `version >= входящей`». Версия сравнивается в разрезе типа события. Контрпример:

```
Пришло: order_in_progress, doc_version=5 → применили, записали (in_progress / 5)
Пришло: order_cancelled, doc_version=3   → записи "cancelled" нет
                                         → правило не сработало
                                         → применили устаревшее событие
                                         → сделка откатилась в «Отменена»
```

**Откуда растут ноги.** Прямая цитата предложенного правила. Контрпример — детерминированная логика, ни от чего внешнего не зависит.

**Решение.** Курсор `last_applied_doc_version` **на сделку**, а не на пару (сделка, событие). Любое событие с `doc_version <= last_applied` отбрасывается независимо от типа.

## P-04. Ключ дедупликации `(CRM_Deal_ID, event)` проглотит законный повтор

**Уверенность:** 🟡 Высокая
**Кого касается:** Битрикс

**Суть.** Ключ уникальный. Второй законный `oos_at_picking` по той же сделке (кладовщик нашёл вторую недостачу после `/update_order`) будет молча отброшен как дубль.

**Откуда растут ноги.** Аналитик: «ключом идемпотентности в таблице `b_dom_processed_1c_events` является только пара `(CRM_Deal_ID, event)`». Уникальность ключа я **предположил** — если он не уникальный, то он и не ключ идемпотентности. Сценарий с двумя нехватками — моя реконструкция, но она реалистична: кладовщик собирает заказ по позициям.

**Решение.** Дедупликация по `event_id` (UUID от 1С). Упорядочивание по `doc_version`. Два разных механизма, не схлопывать.

**Проверить:** уникален ли ключ на самом деле. Вопрос Б24-6.

## P-05. `doc_version` в 1С не существует

**Уверенность:** 🟢 Подтверждено
**Кого касается:** 1С

**Суть.** Битрикс должен отбрасывать устаревшие события. Для этого нужен монотонный счётчик версии документа. В 1С его нет, и откуда он возьмётся — не сказано.

**Откуда растут ноги.** Прямая цитата аналитика: «в текущих источниках нет понятия версии документа, что является „дырой“ при проектировании асинхронного обмена».

**Решение.** Реквизит-счётчик, инкремент при каждой записи. Либо появляется бесплатно, если делаем очередь исходящих событий (см. P-16).

## P-06. Событие `order_confirmed` не описано в спецификации 1С

**Уверенность:** 🟡 Высокая
**Кого касается:** оба

**Суть.** `order_confirmed` фигурирует в ТЗ по Битриксу (триггер перехода в «Ожидает сборки», раздел про защиту от петли). В блоке событий 1С его нет.

**Откуда растут ноги.** Твои исходные вводные по Битриксу: «после ответа `success` на создание заказа **или получения вебхука `order_confirmed`**». Блок вводных по 1С перечисляет 4 события, потом добавился `check_printed` — итого 5, `order_confirmed` среди них нет.

**Чего я не знаю:** я видел только выжимку по 1С, а не полное ТЗ. Возможно, событие описано в разделе, которого у меня не было.

**Наше мнение.** Событие не нужно: переход в `PICKING` делается по синхронному ответу. Похоже на артефакт ранней редакции. Но пока оно в тексте — кто-нибудь его реализует.

**Проверить:** вопросы 1С-13, Б24-8. Плюс NotebookLM (см. ниже).

## P-07. Мастер статусов определён дважды и по-разному

**Уверенность:** 🟢 Подтверждено
**Кого касается:** оба
**Статус:** согласовано с бизнесом

**Суть.** В одном ответе: «мастер-системой для статусов является 1С». В другом: «мастером статусов является Bitrix24 CRM».

**Откуда растут ноги.** Две прямые цитаты из ответов аналитика (Q3 и Q6).

**Решение (согласовано).** «Статус» — не один объект:

| Артефакт | Владелец | Единственный писатель |
|---|---|---|
| Факты товародвижения (проведён / резерв / отгружен / отменён) | 1С | 1С |
| Стадия Сделки (воронка) | Битрикс24 | только Битрикс24 |
| Клиентский статус в ЛК | CMS | CMS |

**Формулировка для ТЗ:** 1С владеет событиями, Битрикс24 владеет стадией, CMS владеет формулировкой.

**Почему это важно.** Из фразы «1С — мастер статусов» логически следует эндпоинт вида `/set_status` или прямая запись стадии из 1С. Тогда у стадии два писателя и гарантированная гонка на отмене.

## P-12. Две нехватки схлопнуты в одну стадию (в одной из редакций)

**Уверенность:** 🟢 Подтверждено, но **проблема, возможно, уже решена в ТЗ**
**Кого касается:** Битрикс

**Суть.** Нехватка при создании (`partial` от `/create_order`, заказа в 1С нет) и нехватка при сборке (`oos_at_picking`, заказ есть) — разные состояния с разными выходами. В части источников это одна стадия `OOS_QUERY`.

**Откуда растут ноги.** Ответ на четвёртый вопрос: «`OOS_QUERY` — получен ответ `partial` при создании **или сборке**». Но ответ K5: в версии v1.1 «Слоя наблюдаемости» (CONTRA-2) старый «Частичный OOS» уже разделён на «Требует уточнения (Нехватка)» и «Уточнение состава».

**То есть:** проблема, скорее всего, уже решена в актуальной версии ТЗ, но **список стадий из ТЗ Чекаута этого не учитывает**. Это рассинхрон документов, а не ошибка проектирования.

**Решение.** Синхронизировать. `OOS_CREATE` + `OOS_PICKING`, `OOS_QUERY` вычеркнуть.

## P-14. Нехватка при сборке не правит учётный остаток

**Уверенность:** 🟠 Средняя
**Кого касается:** 1С, бизнес

**Почему я всё равно считаю это важным.** Если процесса нет, механизм воспроизводит сам себя: каждая необработанная нехватка гарантирует следующую.

Создаётся ли какой-либо документ коррекции, пересчёта или списания недостачи? Ставится ли задача сотруднику? Приведи раздел ТЗ и цитату. Если такого процесса в ТЗ нет — так и скажи.

## P-15. `/update_order` меняет состав — деньги не пересчитываются нигде

**Уверенность:** 🟠 Средняя

## P-28. Инверсия ломает смысл статуса «В работе»

**Уверенность:** 🔴 Гипотеза, **актуальна только если принимаем инверсию** (Б-1)
**Кого касается:** 1С, бизнес

**Суть.** Если резерв ставится до оплаты, а `/create_order` создаёт заказ сразу в «В работе», то на складе появится **неоплаченный заказ в статусе «можно собирать»**. Кладовщик его соберёт. Если клиент не оплатит за 30 минут — `/cancel_reserve` отменит уже собранный заказ.

**Возможное решение.** Статус **«Новый»** = «резерв поставлен, оплата не подтверждена». Отсюда же объясняется, почему у него нет исходящего события: он не должен доезжать до склада.

```
/create_order            → «Новый» + проведён + резерв   → CRM: AWAIT_PAYMENT
вебхук эквайринга (hold) → /confirm_payment              → «В работе» → CRM: PAID → PICKING
таймаут 30 мин           → /cancel_reserve               → «Отменён»
```

**Откуда растут ноги.** Логическое следствие инверсии. Обоснование Q3 («создаём в „В работе“, потому что вызываем только после оплаты») теряет посылку, но вывод остался.

**Чего я не знаю — критично:** **формирует ли проведение резерв при статусе «Новый»?** Если нет — схема не работает без доработки конфигурации. Вопрос 1С-4.

**Что понадобится:** новый эндпоинт `/confirm_payment` (или параметр в существующем). Сейчас его нет.

**Побочный риск:** резерв до оплаты = способ заморозить остаток. Нужны лимиты (вопрос Б-11).

## P-29. Автоотмена при `rejected` — бизнес-решение, принятое по умолчанию

**Уверенность:** 🟡 Высокая (что решение спорное), 🔴 гипотеза (что автоотмена реализована)
**Кого касается:** бизнес

**Суть.** `partial` (нет части) → менеджер, SLA 24ч. `rejected` (нет ничего) → автоотмена.

На практике: заказ на 1 позицию, её нет → автоотмена, клиент ушёл. Заказ на 20 позиций, нет одной → менеджер спасает.

Порог принятия решения — «сколько строк в корзине», а не «можем ли предложить аналог». Для DIY, где 1–3 позиции = основная масса заказов, это автоматический отказ самой массовой категории.

**Откуда растут ноги.** Цитата аналитика: «Статусы `partial` и `rejected` должны уводить сделку по совершенно другим веткам (в „Требует уточнения“ или „Отменена“)». Что именно `rejected` ведёт в «Отменена» автоматически — **моя интерпретация** этой фразы.

**Чего я не знаю.** Может, там менеджер. Может, ветка вообще не описана.
