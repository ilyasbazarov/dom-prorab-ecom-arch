## Общие архитектурные ограничения (обязательно к прочтению)

- **HTTP 200 от /create_order и /update_order = строго факт проведения документа** в 1С с записью в регистр резервов \[DEC-33\]. CRM не может считать резерв поставленным раньше.

- **Мутация заказа — полная перезапись ТЧ** через /update_order. Единственная точка инициации — кнопка «Изменения согласованы» в CRM \[DEC-34\].

- **Store_ID не меняется при мутации** даже если передан в payload \[DEC-34, CON-09\].

- **Резерв при oos_at_picking сохраняется** до решения менеджера \[DEC-PR-04\]. Снятие только через /cancel_reserve или /update_order.

- **Автооткат мутации не реализуется на MVP** \[DEC-35\].

- **Pre-check — UX-помощь, не финальная гарантия.** Финальная истина — ответ 1С \[COM-PR-02\].

- **Идемпотентность входящих вебхуков 1С → CRM:** таблица b_dom_processed_1c_events, ключ (CRM_Deal_ID, event). Создана в рамках Checkout TZ \[INF-CK-02\].

# ЧАСТЬ I — ВХОДЯЩИЕ ВЕБХУКИ ОТ 1С → CRM

## INF-PR-01 — Обработчик вебхука order_confirmed / order_cancelled (IC-PR-00)

> Вебхук 1С-РТ-03 перенесён из IC Discovery & Selection в этот IC \[COM-PR-01, COM-DS-05\].

**Endpoint CRM:** POST {CRM_Webhook_URL_OrderStatus} — **отдельный URL** от CRM_Webhook_Path_CheckPrinted \[INF01 р.4.3\]

**Входящий payload от 1С:**

{

"CRM_Deal_ID": "12345",

"event": "order_confirmed",

"purchase_order_guid": "a1b2c3d4-...",

"purchase_order_number": "ЗКЧ-000123",

"timestamp": "2024-01-15T14:30:00"

}

**Триггеры отправки от 1С (только e-com заказы с заполненным УФ_CRM_Deal_ID):**

| **Новый статус в 1С**     | **event**                                                                 |
|---------------------------|---------------------------------------------------------------------------|
| В работе / Ожидает сборки | order_confirmed                                                           |
| Отменён                   | order_cancelled                                                           |
| Нехватка при сборке       | oos_at_picking — **отдельный endpoint** CRM_Webhook_URL_OOS (см. Эпик 19) |

**AC обработчика:**

1.  Валидировать X-API-Key → если неверный: HTTP 401

2.  Проверить b_dom_processed_1c_events по ключу (CRM_Deal_ID, event)

    - Если дубль → HTTP 200 без действий (идемпотентность)

    - Если новый → записать, продолжить

3.  Маршрутизация по event:

| **event**       | **Действие CRM**                                                                                                                                                                                    |
|-----------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| order_confirmed | Найти сделку по CRM_Deal_ID. Сохранить purchase_order_guid в UF_PURCHASE_ORDER_GUID (если не был сохранён). Сделка → «В работе / Ожидает сборки»                                                    |
| order_cancelled | Найти сделку. **Проверить:** если сделка уже в статусе «Отменена» (CRM сам инициировал) → HTTP 200, без изменений (защита от петли). Иначе: сделка → «Отменена». Уведомление менеджеру              |
| order_completed | Найти сделку по CRM_Deal_ID. Проверить что сделка не в статусе «Успешно реализовано» (защита от дублей). Сделка → «Успешно реализовано». Триггер: WA-накладная (Эпик 23, LastMile TZ) \[COM-R3-05\] |

4.  Вернуть HTTP 200

> **Защита от петли:** CRM-робот сам вызывает /cancel_reserve → 1С отменяет заказ → 1С отправляет order_cancelled → CRM уже в статусе «Отменена» → игнорировать.

**DoD:**

- order_confirmed обновляет статус сделки

- order_cancelled проверяет текущий статус перед действием

- Повторный вебхук → HTTP 200 без изменений

- order_completed переводит сделку в «Успешно реализовано» и запускает WA-накладную; если сделка уже «Успешно реализовано» — HTTP 200 без повторных действий

- Обработчик не блокирует кассовую операцию в 1С

# ЧАСТЬ II — ЭПИКИ 16–21

## Эпик 16 — Hard Booking: гарантия резерва \[CRM\]

**User Story:** Как бизнес, я хочу чтобы оплаченный/подтверждённый заказ гарантированно заморозил остатки на складе, исключив человеческий фактор.

### Что вызывает в 1С

- POST /create_order (IC-PR-01 / IC-CK-03)

- Триггер: вебхук оплаты (delivery) ИЛИ кнопка «Подтверждён» менеджером (самовывоз)

### Требования к реализации \[CRM\]

> **Примечание:** Механизм вызова /create_order описан в Checkout TZ (Эпики 15.2, 15.3). Данный раздел добавляет полную матрицу обработки ответов и гарантии корректности.

**Полная матрица ответов /create_order → действия CRM:**

| **HTTP** | **status** | **Поля ответа**                                 | **Действие CRM**                                                                                                                                                |
|----------|------------|-------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 200      | success    | purchase_order_guid, purchase_order_number      | Сохранить purchase_order_guid в UF_PURCHASE_ORDER_GUID. WA клиенту (HSM-шаблон: заказ подтверждён, слот, склад). Сделка → «В работе / Ожидает сборки» (AC-16.1) |
| 200      | partial    | missing: \[{GUID, qty_ordered, qty_available}\] | **→ Эпик 17.1**                                                                                                                                                 |
| 200      | rejected   | reason: "insufficient_stock"                    | **→ Эпик 17.2**                                                                                                                                                 |
| 200      | duplicate  | purchase_order_guid, idempotent: true           | Лог. Сохранить purchase_order_guid если пустой. Без повторных действий (AC-16.3)                                                                                |
| 400      | error      | reason: "invalid_store_id", details             | Уведомление менеджеру: «Ошибка: некорректный склад. Детали: {details}». Сделка остаётся в текущем статусе                                                       |
| 500      | error      | reason: "posting_failed", details               | Уведомление менеджеру: «Ошибка 1С при создании резерва. Детали: {details}». Логировать. Сделка остаётся в текущем статусе                                       |

**Гарантии 1С (проверить при приёмке, не реализовывать на стороне CRM):**

- HTTP 200 = факт проведения с записью в регистр резервов \[DEC-33\]

- Ошибка проведения = HTTP 500 с явным reason: "posting_failed" \[1С-PR-01\]

- Store_ID валидируется до создания документа \[1С-PR-02\]

### DoD

- AC-16.1: success → purchase_order_guid сохранён, WA отправлен, статус обновлён

- AC-16.2: error: posting_failed → менеджер уведомлён, сделка не перешла в ошибочный статус

- AC-16.3: повторный вызов с тем же CRM_Deal_ID → duplicate, без дублирования действий

## Эпик 17 — OOS урегулирование при резервировании \[CRM\]

**User Story:** Как бизнес, я хочу автоматический процесс урегулирования если клиент купил онлайн то, что физически забрали со склада минуту назад.

### Что получает от 1С

- Ответы /create_order: partial, rejected с массивом missing\[\]

### Требования к реализации \[CRM\]

**AC-17.1 — Ответ partial (часть позиций OOS):**

1.  Сделка → статус «Требует уточнения (Нехватка)»

2.  Сохранить missing\[\] в UF_OOS_ITEMS (JSON)

3.  CRM-робот создаёт задачу менеджеру:

Тема: «Нехватка при резервировании — Сделка \#{CRM_Deal_ID}»

Текст задачи:

Заказ создан частично. Позиции с нехваткой:

{для каждого элемента missing\[\]}

• {Name_Clean} — заказано: {qty_ordered} {Unit_Base}, доступно: {qty_available}

Действия:

\[Связаться с клиентом и согласовать изменения\]

⚠️ Для card-pay заказа (доставка):

При согласии клиента на уменьшенный состав — сумма к возврату: {refund_sum} сом

При отказе клиента — полный возврат: {original_payment_amount} сом

Возврат выполнить через кабинет эквайринга (до реализации API)

4.  refund_sum рассчитывается автоматически: Σ(missing\[i\].qty_ordered × item_price) \[EDGE-15\]

5.  Для самовывоза строку о card-pay не выводить \[COM-PR-06\]

**AC-17.2 — Ответ rejected (все позиции OOS):**

1.  Сделка → «Отменена (OOS)»

2.  WA клиенту (HSM-шаблон): «К сожалению, товары из вашего заказа закончились на складе. Менеджер свяжется с вами» (EDGE-03)

3.  Для delivery/card-pay: задача менеджеру — «Выполнить полный возврат {original_payment_amount} сом через кабинет эквайринга»

**AC-17.3 — partial + card-pay (delivery):**

- Инструкция в задаче менеджеру рассчитывается CRM автоматически (описано в AC-17.1 выше)

**AC-17.4 — Эскалация: сделка в «Требует уточнения» без движения 2 ч:**

1.  CRM-робот (таймер: 2 ч с момента перехода в «Требует уточнения»)

2.  Задача переназначается на руководителя

3.  WA клиенту (HSM-шаблон): «Мы работаем над вашим заказом и свяжемся с вами в течение 1 часа»

### DoD

- AC-17.1: задача менеджеру с корректным составом missing\[\] и расчётом суммы возврата

- AC-17.2: сделка отменена, WA отправлен, задача по возврату создана (для delivery)

- AC-17.4: эскалация через 2 ч без ручного движения по сделке

## Эпик 18 — Транзакционные коммуникации \[CRM\]

**User Story:** Как клиент, я хочу получить подтверждение заказа в мессенджер, чтобы быть спокойным за деньги.

### Требования к реализации \[CRM\]

**Канал и fallback \[COM-PR-07\]:**

| **Тип сообщения**                             | **Таймаут WA**                   | **Fallback**             |
|-----------------------------------------------|----------------------------------|--------------------------|
| Транзакционные уведомления (заказ, OOS, слот) | **60 сек** на статус «Delivered» | SMS через локальный шлюз |
| OTP-авторизация                               | **45 сек**                       | SMS                      |

**Точки отправки транзакционных уведомлений:**

| **Триггер**                            | **HSM-шаблон**          | **Переменные**                                        | **AC**  |
|----------------------------------------|-------------------------|-------------------------------------------------------|---------|
| /create_order = success                | «Заказ подтверждён»     | order_id, total_sum, date, store_name / delivery_slot | AC-18.1 |
| /create_order = rejected               | «Заказ отменён (OOS)»   | order_id                                              | AC-17.2 |
| Сделка → «Требует уточнения» (partial) | «Есть вопрос по заказу» | order_id, manager_name                                | —       |
| Эскалация 2ч                           | «Работаем над заказом»  | order_id, eta_hours = 1                               | AC-17.4 |

**AC-18.2 — Fallback SMS при недоставке WA:**

- Если за 60 сек уведомление не получило статус «Delivered» от WA API → автоматически отправить SMS с аналогичным содержанием через SMS-шлюз

- Fallback работает на уровне CRM-робота (не CMS)

**AC-18.3 — Prerequisite \[CON-10\]:**

- Все HSM-шаблоны зарегистрированы в Meta Business Manager до запуска

- При незарегистрированном шаблоне WA-канал недоступен → все уведомления идут через SMS

### DoD

- WA-уведомление отправляется при каждом триггере из таблицы

- Fallback SMS срабатывает автоматически при недоставке WA за 60 сек

- Незарегистрированный HSM → SMS, без ошибки системы

## Эпик 19 — Мутация заказа \[CRM\]

**User Story:** Как менеджер, я хочу управлять изменениями заказа в CRM, а склад получать актуальный состав без рассинхрона.

### Что получает от 1С

- Вебхук oos_at_picking (IC-PR-03) — 1С → CRM при фиксации нехватки кладовщиком

- Ответы /update_order (IC-PR-04) — success, partial_oos, error

### Что вызывает в 1С

- POST /update_order (IC-PR-04)

### 19.1 — Входящий вебхук oos_at_picking от 1С \[CRM handler\]

**Endpoint CRM:** POST {CRM_Webhook_URL_OOS} — **отдельный URL** от CRM_Webhook_URL_OrderStatus

**Входящий payload:**

{

"CRM_Deal_ID": "12345",

"status": "oos_at_picking",

"missing": \[

{"GUID": "550e8400-...", "qty_ordered": 5, "qty_available": 2}

\],

"timestamp": "2024-01-15T14:30:00"

}

> qty_available = qty_ordered − Кол_во_Нехватки (вычисляет 1С). CRM видит сколько есть в наличии, не сколько отсутствует \[COM-PR-04, DEC-PR-07\]

**AC обработчика:**

1.  Валидировать X-API-Key

2.  Проверить b_dom_processed_1c_events по ключу (CRM_Deal_ID, "oos_at_picking") — идемпотентность

3.  Сохранить missing\[\] в UF_OOS_ITEMS карточки сделки

4.  Сделка → «Уточнение состава» (AC-19.1)

5.  **Резерв в 1С НЕ снимается** — сохраняется до решения менеджера \[DEC-PR-04\]

6.  Создать задачу менеджеру (см. ниже)

7.  Вернуть HTTP 200

**Задача менеджеру при oos_at_picking:**

Тема: «Нехватка при сборке — Сделка \#{CRM_Deal_ID}»

Позиции с нехваткой:

• {Name_Clean}: заказано {qty_ordered}, на складе {qty_available}

\[Изменения согласованы\] — кнопка инициирует pre-check + /update_order

### 19.2 — Цикл мутации: Pre-check → /update_order \[CRM\]

**Триггер:** менеджер нажимает «Изменения согласованы» в карточке сделки.

#### Шаг 1: Pre-check \[COM-PR-02, COM-PR-03\]

**Механизм:** нативный Bitrix PHP API — прямой запрос к таблице b_dom_stocks. **HTTP-вызов к 1С запрещён.**

// Для каждой новой позиции в составе сделки:

\$stockFree = BDomStocks::getStockFree(\$guid, \$dealStoreId);

// \$dealStoreId = UF_SOURCE_STORE_ID сделки — строго этот склад \[COM-PR-03\]

// ЗАПРЕЩЕНО: использовать Total_Stock_Free (агрегат по всем складам)

**Алгоритм:**

1.  Для каждой позиции нового состава: получить Stock_Free из b_dom_stocks WHERE guid = ? AND store_id = UF_SOURCE_STORE_ID

2.  Если Stock_Free \< qty_requested → добавить в список предупреждений

3.  Если есть предупреждения:

    - Показать менеджеру модальное окно: «Возможна нехватка по {N} позициям на складе {store_name}. Pre-check данные могут отставать до 5 мин. Продолжить?»

    - Кнопки: «Продолжить» / «Отмена»

    - При «Продолжить» → перейти к Шагу 2

4.  Если предупреждений нет → Шаг 2 автоматически

**Лаг данных:** ≤ 5 мин (Fast Track батч). Pre-check — UX-помощь, финальная истина — ответ 1С.

#### Шаг 2: Вызов /update_order (IC-PR-04)

**Payload CRM → 1С:**

{

"CRM_Deal_ID": "12345",

"items": \[

{"GUID": "550e8400-...", "Qty": 5, "Price": 600.00},

{"GUID": "660f9511-...", "Qty": 2, "Price": 1200.00}

\]

}

> Store_ID в payload **не передаётся** — 1С использует UF_SOURCE_STORE_ID исходного заказа \[DEC-34, CON-09\]

**Матрица ответов /update_order → действия CRM:**

| **HTTP** | **status**                  | **Действие CRM**                                                                                                                                                                                                                                                                                                                                                                                                                  |
|----------|-----------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 200      | success                     | Обновить состав сделки. Обновить OPPORTUNITY сделки: OPPORTUNITY = Σ(qty_applied × Price), где Price — из товарных строк сделки по GUID. Запустить финансовый флоу (Эпик 20). Сделка → «Ожидает сборки». UF_MUTATION_COUNT += 1 (AC-19.3)                                                                                                                                                                                         |
| 200      | partial_oos                 | Обновить состав до applied\[\]. Убрать missing\[\] из отображения. Обновить OPPORTUNITY сделки: OPPORTUNITY = Σ(applied\[i\].qty_applied × Price), где Price — из товарных строк сделки Битрикс24 по GUID (цена на момент создания/последней мутации заказа, не текущий каталог). Запустить финансовый флоу по applied-сумме. Задача менеджеру: «Согласовать оставшийся missing\[\] с клиентом». Цикл может повториться (AC-19.4) |
| 200      | partial_oos + applied: \[\] | **→ AC-19.4a: все позиции OOS** \[COM-PR-10\]                                                                                                                                                                                                                                                                                                                                                                                     |
| 200      | success, idempotent: true   | Лог. Без изменений состава                                                                                                                                                                                                                                                                                                                                                                                                        |
| 400      | error: items_empty          | Показать менеджеру: «Нельзя сохранить пустой состав. Для отмены заказа — нажмите «Отменить заказ»»                                                                                                                                                                                                                                                                                                                                |
| 409      | error: order_not_mutable    | Уведомить менеджера: «Заказ уже закрыт или отменён в 1С, изменения невозможны»                                                                                                                                                                                                                                                                                                                                                    |
| 500      | error: posting_failed       | Уведомить менеджера: «Ошибка 1С при мутации. Старый состав сохранён.» Логировать details                                                                                                                                                                                                                                                                                                                                          |

**AC-19.4a — applied: \[\] (все позиции OOS при мутации) \[COM-PR-10\]:**

1.  Сделка остаётся в «Уточнение состава»

2.  Задача менеджеру:

Тема: «Критично: новый состав полностью недоступен — Сделка \#{CRM_Deal_ID}»

Все позиции нового состава отсутствуют на складе {store_name}.

Выберите действие:

\[Отменить заказ\] → /cancel_reserve → резерв снят → сделка «Отменена»

\[Связаться с клиентом\] → сделка остаётся в «Уточнение состава»

3.  Автодействий без нажатия кнопки нет \[DEC-35\]

**AC-19.5 — Защита склада:**

- 1С игнорирует Store_ID из payload. На стороне CRM: Store_ID в payload /update_order **не передавать**

### DoD

- AC-19.1: вебхук oos_at_picking → сделка «Уточнение состава», задача менеджеру, резерв сохранён

- AC-19.2: кнопка «Изменения согласованы» → pre-check → модальное окно если нехватка

- AC-19.3: success → состав обновлён, UF_MUTATION_COUNT увеличен

- AC-19.4: partial_oos → состав по applied\[\], цикл мутации может повториться

> **⚠️ Важно про резервы:** после ответа partial_oos позиции из missing\[\] **не имеют резерва в 1С** — старый резерв снят полностью, новый поставлен только на applied\[\]. При повторном согласовании missing\[\] позиций остаток не гарантирован: другой заказ мог занять их за это время. Возможен повторный partial_oos.

- AC-19.4a: applied: \[\] → задача менеджеру с двумя кнопками

- AC-19.5: Store_ID в payload /update_order не передаётся

- AC-19.6: идемпотентный oos_at_picking → HTTP 200 без повторных задач

### Ошибки и исключения

| **Ситуация**                                             | **Поведение**                                                                                                                 |
|----------------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------|
| 1С недоступна при /update_order                          | Timeout → уведомление менеджеру. Сделка остаётся в текущем статусе                                                            |
| Менеджер нажал «Изменения согласованы» с пустым составом | Заблокировать на UI до вызова 1С: «Добавьте хотя бы одну позицию»                                                             |
| 1С вернула 400 crm_deal_id_required                      | Лог. Уведомление менеджеру: «Ошибка передачи ID сделки в 1С. Обратитесь в техподдержку»                                       |
| 1С вернула 400 guid_not_found (details: GUID)            | Уведомление менеджеру: «Товар {GUID} из состава не найден в 1С. Возможно, номенклатура была изменена — уточните у кладовщика» |
| 1С вернула 400 qty_invalid                               | Уведомление менеджеру: «Некорректное количество в одной из позиций. Проверьте состав заказа»                                  |

## Эпик 20 — Финансовый трекинг \[CRM\]

### 20.1 — Самовывоз (оплата на кассе): финансовых корректировок нет \[COM-PR-06\]

При мутации заказа с типом «Самовывоз»:

- Кассир при выдаче пробивает **итоговую сумму** после всех мутаций

- CRM не инициирует никаких финансовых операций

- нативное поле OPPORTUNITY сделки Битрикс24 обновляется CRM после каждого успешного /update_order (формула — см. PR-P2). Отображается в карточке как итоговая сумма заказа

- Никаких кнопок возврата или доплаты не создаётся

### 20.2 — Delivery (card-pay): Финансовые корректировки

> **⚠️ ОСОЗНАННЫЙ ГЭП \[COM-PR-08, CON-11\]**
>
> Весь блок финансовых корректировок для delivery заказов — **TBD до выбора эквайринга** (CON-11). Реализация не стартует до получения документации Бакай Банка / финтех-провайдера.
>
> **Зафиксированная архитектура для будущего патча:**

**Переменные:**

- original_payment_amount — зафиксирован в UF_ORIGINAL_PAYMENT_AMOUNT при подтверждении оплаты \[COM-PR-05\]

- OPPORTUNITY сделки (нативное поле Битрикс24) — обновляется CRM после каждой успешной мутации \[COM-R3-02\]

**Алгоритм выбора флоу:**

delta = OPPORTUNITY - UF_ORIGINAL_PAYMENT_AMOUNT

delta \< 0 → partial_refund (AC-20.1)

delta \> 0 → add_on_payment (AC-20.2)

delta = 0 → финансовых действий нет, заказ продолжает движение

**AC-20.1 — Partial Refund (delta \< 0): MVP-заглушка \[DEC-53\]**

1.  CRM вычисляет refund_amount = original_payment_amount − new_order_total

2.  Отображает менеджеру кнопку: **«Инициировать возврат \[{refund_amount} сом\]»**

3.  При нажатии — инструкция: «Выполнить возврат через кабинет эквайринга. Сумма: {refund_amount} сом. Основание: частичная отмена заказа {CRM_Deal_ID}»

4.  После подтверждения менеджером → зафиксировать в карточке: refund_amount, refund_confirmed_at, refund_confirmed_by

5.  **API-интеграция возврата заморожена** до документации эквайринга \[DEC-37\]

**AC-20.1.2 — Клиент отказывается от частичного заказа:**

1.  Полный refund = original_payment_amount

2.  CRM → 1С /cancel_reserve → резерв снят → сделка «Отменена»

**AC-20.2 — Add-on Payment (delta \> 0): ⚠️ Реализуемость TBD \[DEC-30\]**

1.  addon_amount = new_order_total − original_payment_amount

2.  Генерировать подчинённый счёт (привязка: сделка → счёт на дельту)

3.  WA клиенту: «По вашему заказу изменился состав. Доплата: {addon_amount} сом. \[Ссылка\]»

4.  Таймаут addon_payment_timeout (TBD) без оплаты → задача менеджеру с двумя вариантами:

    - «Отгрузить исходный состав» (без доплаты)

    - «Полная отмена» → /cancel_reserve

    - Автодействий нет \[EDGE-16\]

5.  Клиент оплатил → webhook → CRM фиксирует доплату. Единый фискальный чек на финальную сумму при закрытии \[DEC-36\]
