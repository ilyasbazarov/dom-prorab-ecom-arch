## Общие архитектурные ограничения (обязательно к прочтению)

- **Прямые вызовы CMS → 1С запрещены архитектурно.** Только CRM → 1С.

- **/create_order никогда не вызывается автоматически при оформлении.** Всегда есть промежуточное событие: оплата (доставка) или подтверждение менеджера (самовывоз) \[DEC-15, DEC-CK-05\].

- **Один заказ = один склад.** Межскладовое дробление запрещено \[CON-04\].

- **UF_SOURCE_STORE_ID фиксируется при создании заказа и никогда не меняется** \[CON-09\].

- **Стоимость доставки не входит в сумму заказа** и не проводится через кассу 1С \[CON-06\].

- **Идемпотентность вебхуков 1С → CRM** обеспечивается таблицей b_dom_processed_1c_events (ключ: order_id + event).

- **Идемпотентность вебхуков эквайринга → CRM** обеспечивается таблицей b_dom_processed_payments (ключ: payment_id).

- **72-часовой таймер автоотмены резерва** — CRM-робот Битрикс24. 1С не содержит регламентного задания для автоотмены \[DEC-32, DEC-CK-03\].

- **Проверка остатков при добавлении в корзину** — CMS-кэш Fast Track (таблица b_dom_stocks), не синхронный вызов 1С. Финальная гарантия — /create_order в 1С \[COM-CK-04\].

## Инфраструктурный блок — Новые таблицы Битрикс

### INF-CK-01 — Таблица b_dom_processed_payments

Назначение: идемпотентность вебхуков эквайринга. Исключает двойное списание при ретраях.

CREATE TABLE b_dom_processed_payments (

id INT AUTO_INCREMENT PRIMARY KEY,

payment_id VARCHAR(100) NOT NULL,

crm_deal_id VARCHAR(50) NOT NULL,

amount DECIMAL(12,2),

status VARCHAR(20),

processed_at DATETIME NOT NULL,

UNIQUE KEY uk_payment_id (payment_id),

INDEX idx_deal (crm_deal_id)

);

**Логика:** При получении вебхука от эквайринга — сначала проверить payment_id в этой таблице. Если найден → HTTP 200 без повторных действий. Если новый → записать и продолжить.

### INF-CK-02 — Таблица b_dom_processed_1c_events

Назначение: идемпотентность вебхуков 1С → CRM (в первую очередь check_printed).

CREATE TABLE b_dom_processed_1c_events (

id INT AUTO_INCREMENT PRIMARY KEY,

order_id VARCHAR(50) NOT NULL,

event VARCHAR(50) NOT NULL,

received_at DATETIME NOT NULL,

payload TEXT,

UNIQUE KEY uk_order_event (order_id, event),

INDEX idx_order (order_id)

);

**Логика:** При получении вебхука от 1С — проверить пару (CRM_Deal_ID, event). Если найдена → HTTP 200 без повторных действий.

## Эпик 8 — Фильтрация cross-sell по складу \[CMS\]

**User Story:** Как бизнес, я хочу рекомендовать только то, что физически лежит на том же складе, чтобы не дробить заказ клиента.

### Что получает от систем

- Cross_Sell_GUIDs (JSON-массив UUID4), Cross_Sell_Source, Sales_30d, Ecom_Status, Category_L2 — из bitrix_catalog.csv (Slow Track, BI, ежедневно)

- Stock_Free\[Store_ID\] — из таблицы b_dom_stocks (Fast Track, каждые 5 мин)

### Требования к реализации \[CMS\]

**Алгоритм рендера блока «С этим товаром покупают»:**

1.  Взять Cross_Sell_GUIDs текущего товара из инфоблока (поле UF_CROSS_SELL_GUIDS)

2.  Если Cross_Sell_GUIDs не пустой → использовать список; иначе → **fallback** (см. ниже)

3.  Для каждого GUID применить фильтры:

    - stock_free WHERE guid = ? AND store_id = window.activeStoreId \> 0 (запрос к b_dom_stocks)

    - Ecom_Status NOT IN ('Toxic_Online', 'Review_Required')

    - Ecom_Status = 'Add_On_Only' — **включать** (специально для cross-sell, ACTIVE=Y)

    - GUID ≠ текущий товар

4.  Результат = 0 товаров → **скрыть блок полностью** (пустой блок не отображается, EDGE-08)

5.  Иначе → рендерить блок

**Fallback (при пустом Cross_Sell_GUIDs или Cross_Sell_Source = 'category_fallback'):**

- Запрос: товары из той же Category_L2 текущего товара, где:

  - Stock_Free\[window.activeStoreId\] \> 0

  - GUID != текущий GUID

  - Ecom_Status NOT IN ('Toxic_Online', 'Review_Required')

- Сортировка: Sales_30d DESC, при равном — Stock_Free DESC

- Лимит: топ-3. Если доступно \< 3 — выводить сколько есть, блок не скрывать

### DoD

- AC-8.1: Клей с Stock_Free = 0 на window.activeStoreId не отображается в блоке

- AC-8.2: Если ни один товар из Cross_Sell_GUIDs не проходит фильтр → блок скрыт, DOM-элемент не рендерится

- AC-8.3: Товар с ACTIVE=N (Toxic_Online, Review_Required) не отображается в блоке (EDGE-09)

- AC-8.3 — Add_On_Only: товар с ACTIVE=Y и Ecom_Status='Add_On_Only' отображается в cross-sell

### Ошибки и исключения

| **Ситуация**                                                 | **Поведение**                                     |
|--------------------------------------------------------------|---------------------------------------------------|
| b_dom_stocks не содержит записи для \[GUID + activeStoreId\] | Трактовать как Stock_Free = 0 — товар не выводить |
| UF_CROSS_SELL_GUIDS в инфоблоке невалидный JSON              | Трактовать как пустой массив → fallback           |
| Category_L2 не заполнена, fallback невозможен                | Скрыть блок                                       |

## Эпик 9 — Умные рекомендации (Market Basket) \[CMS\]

**User Story:** Как клиент, я хочу видеть подходящие сопутствующие товары, чтобы не забыть купить расходники.

### Что получает от систем

- Cross_Sell_Source (apriori \| category_fallback) — из bitrix_catalog.csv. **Клиент UI не видит различия** — блок выглядит одинаково для обоих источников

- MVP стартует на category_fallback. Переключение на apriori — после ≥ 90 дней истории транзакций (Data Engineer, без изменений CMS-кода)

### Требования к реализации \[CMS\]

Логика рендера — идентична Эпику 8 (один компонент). Cross_Sell_Source хранится в UF_CROSS_SELL_SOURCE — используется только для аналитики BI, на UI не влияет.

**Мониторинг (AC-9.3):** Сторона BI. CMS не участвует.

**AC-9.4 (OOS между корзиной и чекаутом):** Финальная проверка актуальности остатков — в /create_order на стороне 1С. Блок cross-sell не отвечает за это. Если товар OOS в момент /create_order — 1С вернёт partial ответ, CRM урегулирует (EDGE-02, EDGE-10).

### DoD

- AC-9.1: Для товара с непустым Cross_Sell_GUIDs и доступным на складе товаром — блок выводит карточки

- AC-9.2: При пустом Cross_Sell_GUIDs — fallback по Category_L2, сортировка Sales_30d DESC

- Визуально: блок выглядит одинаково для apriori и category_fallback источников

## Эпик 10 — Пакетное добавление в корзину (Bundle) \[CMS\]

**User Story:** Как клиент, я хочу в один клик добавить основной товар и все выбранные сопутствующие.

### Требования к реализации \[CMS\]

**Компонент «Добавить всё в корзину»** (JS в шаблоне Аспро):

**Умолчания \[CON-05\]:**

- Количество сопутствующего товара по умолчанию = 1 Unit_Base

- Шаг инпута = 1 Unit_Base (аналогично Эпику 4 основного товара)

- Автоматический расчёт по справочнику достаточности (сколько клея на сколько плитки) — **post-MVP**

**Алгоритм «Добавить всё в корзину»:**

1.  Собрать список: основной товар + все сопутствующие с активным чекбоксом

2.  Для каждого сопутствующего: проверить b_dom_stocks где guid = ? AND store_id = window.activeStoreId

    - Stock_Free = 0 → пропустить + показать уведомление: «\[Название\] закончился на складе» (AC-10.4)

3.  Для каждого прошедшего: если GUID уже в корзине → qty += выбранное_qty (AC-10.2, без дубликата)

4.  Добавить все в корзину одним запросом

5.  Показать попап «Товары добавлены»

**Блокировка Add_On_Only при прямом добавлении \[COM-PRE-01, COM-CK-02\]:**

- Попытка добавить товар с Ecom_Status = 'Add_On_Only' **напрямую** (не через bundle) → блокировать + сообщение: «Этот товар доступен только в комплекте с другим товаром»

- Через bundle-компонент — добавляется без ограничений

### DoD

- AC-10.1: Пакетное добавление создаёт корректные позиции в корзине

- AC-10.2: Дубликат GUID → увеличение qty, не новая позиция

- AC-10.3: Снятый чекбокс → товар не добавляется

- AC-10.4: OOS-товар → пропускается с уведомлением, остальные добавляются

### Ошибки и исключения

| **Ситуация**                  | **Поведение**                                             |
|-------------------------------|-----------------------------------------------------------|
| Все сопутствующие OOS         | Добавляется только основной товар. Уведомление по каждому |
| Прямое добавление Add_On_Only | Блокировка. Сообщение                                     |

# ЧАСТЬ II — КОРЗИНА И ЧЕКАУТ (Эпики 11–15)

## Эпик 11 — Умная корзина (вес и объём) \[CMS\]

**User Story:** Как клиент, я хочу видеть общий вес и объём корзины, чтобы понять — влезет ли это в мою машину или сколько будет стоить доставка.

### Что получает от систем

- Weight_kg, Volume_m3, Weight_Source — из bitrix_catalog.csv (Slow Track, BI), хранятся в UF_WEIGHT_KG, UF_VOLUME_M3, UF_WEIGHT_SOURCE

### Требования к реализации \[CMS\]

**Расчёт (фронтенд JS):**

Total_Weight = Σ(Weight_kg × qty) // qty в Unit_Base

Total_Volume = Σ(Volume_m3 × qty)

Пересчёт при каждом изменении qty — без перезагрузки страницы (AC-11.4).

**Отображение под списком товаров корзины:**

Общий вес: {Total_Weight} кг \| Объём: {Total_Volume} м³

**Алерты по качеству данных:**

| **Условие**                                       | **Алерт**                                             |
|---------------------------------------------------|-------------------------------------------------------|
| Weight_kg = NULL или UF_WEIGHT_SOURCE отсутствует | «Вес некоторых товаров не учтён» (AC-11.2). Вклад = 0 |
| UF_WEIGHT_SOURCE = 'category_default'             | «Вес рассчитан приблизительно» (AC-11.3)              |

Если оба алерта активны → показывать только «Вес некоторых товаров не учтён» (более строгий).

### DoD

- Вес и объём отображаются корректно для AC-11.1

- При изменении qty — пересчёт мгновенный, без перезагрузки

- Алерты не перекрывают основной контент корзины

## Эпик 11a — Фиксация склада корзины \[CMS\]

> Отдельный раздел, т.к. логика фиксации склада — инфраструктурная, влияет на всё поведение корзины.

**Правило \[COM-PRE-09, CON-04\]:** Склад корзины (cart_store_id) фиксируется при добавлении первого товара. При попытке добавить товар, остаток которого принадлежит другому складу — блокировать.

> Технически: все товары в корзине привязаны к одному Store_ID. window.activeStoreId на момент добавления первого товара становится cart_store_id корзины.

**Алгоритм добавления товара в корзину:**

1.  Если корзина пуста → зафиксировать cart_store_id = window.activeStoreId

2.  Если корзина не пуста И window.activeStoreId != cart_store_id → **предупреждение:** «Ваша корзина привязана к магазину \[Название cart_store\]. Переключитесь на этот магазин или оформите текущую корзину отдельно»

    - **Не добавлять товар автоматически**

    - Кнопки: «Продолжить с \[cart_store\]» / «Очистить корзину и выбрать \[activeStore\]»

3.  Если window.activeStoreId == cart_store_id → добавить товар обычным образом

**Хранение:** cart_store_id в сессии пользователя (Битрикс session). При очистке корзины — cart_store_id сбрасывается.

### DoD

- Первый товар фиксирует склад

- Товар с другого склада — предупреждение, автодобавления нет

- Очистка корзины сбрасывает фиксацию склада

## Эпик 12 — OTP-авторизация \[CMS + CRM\]

**User Story:** Как клиент, я хочу войти по номеру телефона без пароля, чтобы чекаут занял секунды.

### Что получает от систем

- Нет входящих данных от 1С

### Что вызывает в 1С

- Ничего. OTP не связан с 1С

### Требования к реализации \[CMS\]

**Триггер:** Клиент нажимает «Оформить заказ» → проверка авторизации → если не авторизован → OTP-форма.

**Форма:** поле «Номер телефона» (+996...) → кнопка «Получить код».

**Rate limiting \[server-side\]:**

- 5 запросов OTP с одного номера за 60 минут → блокировка до истечения окна (AC-12.5)

- Хранить счётчик в кэше Битрикс (managed cache) с TTL 60 мин

**Прогрессивные задержки повторного запроса:**

- Повтор 1: 60 сек

- Повтор 2: 120 сек

- Повтор 3: 300 сек

- После 3-го повтора: блокировка 10 мин (AC-12.4)

**OTP-формат:** 4-значный цифровой код. Генерировать server-side в Битриксе, хранить в сессии с TTL 5 мин.

**Создание пользователя (AC-12.3):**

- Успешная верификация → поиск пользователя Битрикс по телефону

- Найден → залогинить

- Не найден → создать «теневой» профиль: LOGIN = PHONE, PASSWORD = random_hash, группа «Клиенты»

### Требования к реализации \[CRM\]

**OTP-отправка оркестрируется CRM \[COM-PRE-06\].** CMS формирует код и вызывает CRM-эндпоинт для отправки.

**Канал и fallback:**

| **Шаг** | **Действие**                                                                               |
|---------|--------------------------------------------------------------------------------------------|
| 1       | CMS → CRM: «Отправить OTP {code} на номер {phone}»                                         |
| 2       | CRM → WhatsApp Business API (HSM-шаблон «OTP», pre-approved)                               |
| 3       | Таймаут 45 сек без доставки WA → CRM автоматически fallback → SMS-провайдер с тем же кодом |
| 4       | CRM → CMS: результат отправки                                                              |

**Prerequisite \[CON-10\]:** HSM-шаблон «OTP» зарегистрирован в Meta Business Manager до запуска.

### DoD

- AC-12.1: OTP в WA за 45 сек

- AC-12.2: Fallback SMS при недоставке WA

- AC-12.3: Новый номер → теневой профиль

- AC-12.4: Прогрессивные задержки

- AC-12.5: Rate limit 5 запросов / 60 мин

### Ошибки и исключения

| **Ситуация**            | **Поведение**                                                       |
|-------------------------|---------------------------------------------------------------------|
| WA и SMS оба недоступны | «Не удалось отправить код, попробуйте позже»                        |
| Неверный OTP введён     | Счётчик попыток + 1. После 5 неверных — форма блокируется на 10 мин |
| Истёк TTL кода (5 мин)  | «Код истёк. Запросите новый»                                        |

## Эпик 13 — Логистика: расчёт стоимости доставки \[CMS\]

**User Story:** Как клиент, я хочу знать примерную стоимость доставки до ввода оплаты.

### Что получает от систем

- Total_Weight, Total_Volume из корзины (Эпик 11)

- is_oversize — из UF_IS_OVERSIZE инфоблока товаров

- Яндекс.Карты API — дистанция по адресу (внешний API)

- Тарифная сетка — хранится в настройках CMS (не 1С, не BI)

### Требования к реализации \[CMS\]

**Тип получения — переключатель на чекауте:**

- «Доставка» / «Самовывоз» — влияет на весь дальнейший флоу

**При выборе «Доставка»:**

1.  Поле адреса с интеграцией Яндекс.Карты (подсказки, pin на карте)

2.  При вводе адреса → PHP-скрипт запрашивает дистанцию через Яндекс.Карты API

3.  Определить зону:

| **Зона**             | **Условие**                  | **Действие**                                    |
|----------------------|------------------------------|-------------------------------------------------|
| Бишкек               | Адрес в городе               | Рассчитать тариф автоматически (AC-13.1)        |
| Чуйская область      | За чертой Бишкека, в области | «Стоимость доставки уточнит менеджер» (AC-13.3) |
| За пределами Чуйской | —                            | «Доставка в этот регион недоступна» (AC-13.4)   |
| Яндекс API ошибка    | —                            | Поведение как Чуйская (EDGE-12, AC-13.5)        |

#### Требования к реализации \[CMS\]

**Шаг чекаута «Оплата»:** При выборе «Самовывоз» — единственный доступный способ: «Оплата на кассе». Карта и QR не предлагаются.

**После подтверждения формы клиентом (IC-CK-01) — CMS создаёт сделку в CRM:**

POST Bitrix24 REST API: crm.deal.add

{

"CRM_Deal_ID": "{auto}",

"CLIENT_PHONE": "+996XXXXXXXXX",

"UF_SOURCE_STORE_ID": "{cart_store_id}",

"DELIVERY_TYPE": "pickup",

"DATE_PICKUP": "2024-01-20",

"TIME_SLOT": "14:00-18:00",

"ITEMS": \[

{"GUID": "uuid4", "qty": 11, "Price_Ecom": 1000.00, "Name_Clean": "..."}

\],

"TOTAL_WEIGHT": 220.0,

"TOTAL_VOLUME": 0.15

}

**Определение типа клиента (UF_CLIENT_TYPE):**

При формировании payload crm.deal.add CMS определяет тип контрагента:

- Клиент заполнил реквизиты юрлица (наименование компании / ИНН) в форме чекаута → UF_CLIENT_TYPE = "legal"

- Во всех остальных случаях → UF_CLIENT_TYPE = "physical" (дефолт)

**После создания сделки → CMS показывает клиенту:** «Заказ принят! Менеджер свяжется с вами для подтверждения»

#### Требования к реализации \[CRM\]

**Статус при создании сделки:** «Ожидает подтверждения». /create_order не вызывается. Резерв в 1С не ставится (AC-15.2.1).

**Робот CRM при переходе в «Ожидает подтверждения»:**

- Задача менеджеру: «Связаться с клиентом {phone} и подтвердить заказ {deal_id}»

- Дедлайн задачи: 2 часа

**Кнопка «Подтверждён» в карточке сделки:**

- Менеджер нажимает после подтверждения клиентом → CRM вызывает POST /create_order в 1С (IC-CK-03):

{

"CRM_Deal_ID": "12345",

"Client_Phone": "+996XXXXXXXXX",

"Store_ID": "store_kokzhar_02",

"Items": \[

{"GUID": "uuid4", "qty": 11, "Price_Ecom": 1000.00}

\]

}

**Обработка ответов /create_order:**

| **HTTP**  | **status** | **Действие CRM**                                                                                                                                                                                                       |
|-----------|------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 200       | success    | Сохранить purchase_order_guid в UF_PURCHASE_ORDER_GUID. Сохранить purchase_order_number в UF_1C_ORDER_NUMBER. WA клиенту: «Ваш заказ подтверждён, ждём вас \[Дата\]». Сделка → «В работе / Ожидает сборки» (AC-15.2.3) |
| 200       | partial    | Часть позиций OOS. Сделка → «Требует уточнения». Задача менеджеру: уточнить с клиентом нехватку. EDGE-02                                                                                                               |
| 200       | rejected   | Все позиции OOS. Сделка → «Отменена». WA клиенту: уведомление об отмене                                                                                                                                                |
| 200       | duplicate  | Логировать. Прочитать purchase_order_guid из ответа, сохранить в карточку                                                                                                                                              |
| 400 / 500 | —          | Логировать в лог CRM. Уведомление менеджеру: «Ошибка создания заказа в 1С»                                                                                                                                             |

**Если клиент не ответил / не подтвердил (AC-15.2.4):**

- Сделка остаётся в «Ожидает подтверждения»

- /create_order не вызывается

- Дальнейший флоу — операционный SOP менеджера

- AC-15.2.5: При получении ответа success от /create_order — purchase_order_number сохранён в UF_1C_ORDER_NUMBER сделки

### 15.3 — Автоотмена резерва 72 часа \[CRM\]

**User Story:** Как бизнес, я хочу автоматически размораживать товар, за которым не приехали.

#### Требования к реализации \[CRM\]

**CRM-робот «Автоотмена 72ч»:**

- Триггер: сделка переходит в статус «Ожидает выдачи» + резерв в 1С стоит (= получен purchase_order_guid)

- Таймер: 72 часа с момента создания резерва

- **Сброс таймера:** CRM получил вебхук check_printed от 1С (IC-CK-05) → таймер сбрасывается, автоотмена не срабатывает (AC-15.3.2)

**Действие по истечении таймера (AC-15.3.1):**

CRM вызывает POST /cancel_reserve в 1С (IC-CK-04):

{

"CRM_Deal_ID": "12345"

}

**Обработка ответов /cancel_reserve:**

| **HTTP** | **status**        | **Действие CRM**                                                              |
|----------|-------------------|-------------------------------------------------------------------------------|
| 200      | cancelled         | Резерв снят. Витрина обновится через Fast Track (≤5 мин). Сделка → «Отменена» |
| 200      | already_cancelled | Логировать. Без повторных действий (AC-15.3.3)                                |
| 200      | already_closed    | Логировать (клиент получил товар). Сделка → «Выполнена»                       |
| 404      | order_not_found   | Логировать. Уведомление менеджеру                                             |
