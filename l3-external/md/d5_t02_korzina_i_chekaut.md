## Общие архитектурные ограничения (обязательно к прочтению)

- Прямые синхронные вызовы с сайта в 1С — **архитектурно запрещены**

- Мастер остатков и резервов — **1С**. Никакая другая система не изменяет резервы напрямую

- Один заказ = один склад. Межскладовое дробление запрещено \[CON-04\]

- UF_SOURCE_STORE_ID фиксируется при создании заказа и никогда не меняется \[CON-09\]

- Все HTTP-эндпоинты **идемпотентны по умолчанию**

- После любого изменения остатков — затронутые пары \[GUID, Store_ID\] пишутся в Очередь_Fast_Track (стандартный 5-минутный батч, без отдельного push-механизма)

## Флоу заказов — обязательно понимать перед реализацией

| **Тип заказа**                   | **Последовательность**                                                                                                                                                               |
|----------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Доставка (онлайн-оплата)**     | Клиент оформляет → CRM получает сделку → CRM ждёт вебхук об оплате от эквайринга → только после подтверждения оплаты CRM вызывает /create_order → 1С ставит резерв \[DEC-15\]        |
| **Самовывоз (оплата наличными)** | Клиент оформляет → CRM получает сделку → менеджер связывается с клиентом, подтверждает заказ → менеджер нажимает «Подтверждён» в CRM → CRM вызывает /create_order → 1С ставит резерв |

> **Итог:** /create_order никогда не вызывается автоматически при оформлении. Всегда есть промежуточное событие (оплата или подтверждение менеджера).

## 1С-ИНФ-01 (ДОПОЛНЕНИЕ) — Реквизит УФ_CRM_Deal_ID в документе «Чек ККМ»

**Инициатива / Эпик:** Корзина и Чекаут / Инфраструктура

**Описание задачи:** Дополнение к задаче 1С-ИНФ-01 из блока Discovery & Selection. Добавить реквизит УФ_CRM_Deal_ID в документ «Чек ККМ» для технической связки с Битрикс24 (требуется для 1С-CK-03), а также вывести его на печатную форму чека для покупателя.

> **Связь с другими задачами:** реквизит УФ_CRM_Deal_ID в «Заказе покупателя» создаётся в основной задаче 1С-ИНФ-01 (блок Discovery & Selection). Данное дополнение добавляет его в «Чек ККМ» и настраивает автозаполнение.

#### Техническая реализация в 1С

**Добавить в документ «Чек ККМ»:**

| **Реквизит**   | **Тип**    | **По умолчанию** | **Назначение**                |
|----------------|------------|------------------|-------------------------------|
| УФ_CRM_Deal_ID | Строка(50) | Пустая строка    | Связь с CRM-сделкой Битрикс24 |

**Автозаполнение при создании на основании «Заказа покупателя»:**

Добавить в обработчик события ОбработкаЗаполнения документа «Чек ККМ»:

Если ОснованиеЗаполнения \<\> Неопределено

И ТипЗнч(ОснованиеЗаполнения) = Тип("ДокументСсылка.ЗаказПокупателя") Тогда

ЭтотОбъект.УФ_CRM_Deal_ID = ОснованиеЗаполнения.УФ_CRM_Deal_ID;

КонецЕсли;

**Вывод на печатную форму чека (MVP):**

Если в «Чеке ККМ» заполнено УФ_CRM_Deal_ID (не пустая строка) — добавить строку в печатную форму:

Заказ сайта: {УФ_CRM_Deal_ID}

Располагать после шапки чека, до таблицы с товарами. Если поле пустое — строку не выводить.

#### Критерии приёмки (DoD)

- Реквизит УФ_CRM_Deal_ID присутствует в документе «Чек ККМ»

- При создании чека на основании «Заказа покупателя» с заполненным УФ_CRM_Deal_ID — поле автоматически копируется в чек

- При создании чека вручную (без основания) — поле остаётся пустым, не ошибка

- На печатной форме строка «Заказ сайта: X» присутствует только если поле заполнено

#### Зависимости

- Реквизит УФ_CRM_Deal_ID в «Заказе покупателя» должен быть создан (основная задача 1С-ИНФ-01, блок Discovery & Selection) до реализации этого дополнения

## 1С-CK-01 — Логирование входящих запросов /create_order

**Инициатива / Эпик:** Корзина и Чекаут / Платёжный гейтвей

**Связь с предыдущими задачами:** Дополнение к 1С-РТ-01 (HTTP-сервис /create_order, реализован в блоке Discovery & Selection). Идемпотентность по CRM_Deal_ID уже описана в 1С-РТ-01 и здесь не переопределяется. Текущая задача добавляет только механизм явного логирования каждого входящего запроса — в том числе дублей.

> **Почему отдельная задача:** Требования к логированию транзакционно значимых запросов (резервирование, оплата) — самостоятельная приёмочная проверка. Смешивание с 1С-РТ-01 создаёт риск что при сдаче РТ-01 логирование будет пропущено как «и так понятно».

**User Story:** Как бизнес, я хочу исключить пустые прогоны машин и двойные резервы. Для этого мне нужно видеть в логах каждый запрос на создание заказа — включая повторные (дубли) — чтобы при спорных ситуациях можно было восстановить хронологию.

**Описание задачи:** Добавить в эндпоинт /create_order логирование каждого входящего запроса до выполнения любой бизнес-логики. Логировать как первичные запросы, так и повторные (дубли по CRM_Deal_ID).

#### Техническая реализация в 1С

**Где добавить:** В начало обработчика HTTP-сервиса /create_order, до проверки идемпотентности и до любых операций с данными.

**Регистр сведений Лог_Запросов_CreateOrder (создать):**

| **Поле**           | **Тип**    | **Описание**                                           |
|--------------------|------------|--------------------------------------------------------|
| DateTime           | Дата+Время | Время получения запроса                                |
| CRM_Deal_ID        | Строка(50) | ID сделки из payload                                   |
| Store_ID           | Строка(50) | Склад из payload                                       |
| Client_Phone       | Строка(20) | Телефон из payload                                     |
| Is_Duplicate       | Булево     | Истина если CRM_Deal_ID уже существует                 |
| HTTP_Response_Code | Число      | Код ответа который был возвращён                       |
| Response_Status    | Строка(20) | success / partial / rejected / duplicate / error       |
| Items_Count        | Число      | Количество позиций в Items                             |
| Request_Body_Hash  | Строка(64) | SHA-256 от тела запроса (для дедупликации при анализе) |

**Алгоритм логирования:**

// Шаг 0 — ЛОГИРОВАНИЕ (выполняется всегда, до любой логики)

ЗаписьЛога = РегистрыСведений.Лог_Запросов_CreateOrder.СоздатьМенеджерЗаписи();

ЗаписьЛога.DateTime = ТекущаяДатаВремя();

ЗаписьЛога.CRM_Deal_ID = Payload.CRM_Deal_ID;

ЗаписьЛога.Store_ID = Payload.Store_ID;

ЗаписьЛога.Client_Phone = Payload.Client_Phone;

ЗаписьЛога.Items_Count = Payload.Items.Количество();

ЗаписьЛога.Request_Body_Hash = ХэшСтроки(ТелоЗапроса); // SHA-256

// Is_Duplicate, HTTP_Response_Code, Response_Status — заполнить после

// выполнения логики и до возврата ответа клиенту.

Поля Is_Duplicate, HTTP_Response_Code, Response_Status — заполняются после выполнения логики и до возврата ответа клиенту.

**Дублировать в журнал регистрации 1С:**

- Уровень **Информация** для каждого запроса: "create_order вызов \| CRM_Deal_ID={x} \| Store={y} \| Items={n}"

- Уровень **Информация** для дублей: "create_order ДУБЛЬ \| CRM_Deal_ID={x} \| purchase_order_guid={guid} \| purchase_order_number={number}"

**Очистка лога:** Записи старше 90 дней удалять автоматически (добавить в регламентное задание «НочнаяВыгрузкаSlowTrack» или отдельное задание по расписанию).

#### Критерии приёмки (DoD)

- Каждый POST на /create_order фиксируется в Лог_Запросов_CreateOrder

- Повторный запрос с тем же CRM_Deal_ID фиксируется с Is_Duplicate = Истина

- HTTP_Response_Code и Response_Status заполнены корректно для всех сценариев: 200 success, 200 duplicate, partial, rejected, 400, 500

- Логирование не замедляет ответ эндпоинта (запись в регистр не блокирует возврат ответа)

#### Ошибки и исключения

| **Ситуация**                          | **Поведение**                                                                   |
|---------------------------------------|---------------------------------------------------------------------------------|
| Ошибка записи в лог-регистр           | WARN в журнал регистрации. Основная логика /create_order **НЕ прерывается**     |
| Payload не парсится (невалидный JSON) | Логировать с пустыми полями и Response_Status = "parse_error", вернуть HTTP 400 |

#### Зависимости

- Задача 1С-РТ-01 (основной эндпоинт /create_order) должна быть реализована

## 1С-CK-02 — HTTP-сервис /cancel_reserve

**Инициатива / Эпик:** Корзина и Чекаут / Платёжный гейтвей

**User Story:** Как бизнес, я хочу автоматически размораживать товар, за которым не приехали (или заказ отменён), чтобы продать его другим клиентам. Как клиент с самовывозом — хочу знать что если я не приеду в течение 72 часов, бронь автоматически снимется.

**Описание задачи:** Создать HTTP-эндпоинт /cancel_reserve, который принимает CRM_Deal_ID, находит соответствующий «Заказ покупателя» и отменяет его с освобождением резерва. Инициатором вызова **всегда является CRM Битрикс24** (робот подрядчика по Битриксу) — 1С не содержит регламентного задания для автоотмены по таймауту \[DEC-32\].

#### Техническая реализация в 1С

**Эндпоинт:**

| **Параметр**   | **Значение**                                |
|----------------|---------------------------------------------|
| URL            | /cancel_reserve                             |
| Метод          | POST                                        |
| Аутентификация | Заголовок X-API-Key: {константа CRM_ApiKey} |
| Content-Type   | application/json                            |

**Входной payload:**

{

"CRM_Deal_ID": "12345"

}

**Валидация:**

| **Поле**    | **Проверка**      | **Ошибка**                                  |
|-------------|-------------------|---------------------------------------------|
| CRM_Deal_ID | Не пустое, строка | HTTP 400, {"error": "CRM_Deal_ID required"} |
| X-API-Key   | Верный ключ       | HTTP 401                                    |

**Алгоритм (пошагово):**

**Шаг 1.** Найти «Заказ покупателя» по реквизиту УФ_CRM_Deal_ID = CRM_Deal_ID:

- Если не найден → HTTP 404, {"error": "order_not_found", "CRM_Deal_ID": "12345"}

**Шаг 2.** Идемпотентность — проверить статус найденного заказа:

| **Текущий статус заказа**         | **Действие**      | **Ответ**                                                           |
|-----------------------------------|-------------------|---------------------------------------------------------------------|
| Уже отменён / снят с проведения   | Не трогать        | HTTP 200, {"status": "already_cancelled"}                           |
| Закрыт («Выполнен», «Реализован») | Не трогать        | HTTP 200, {"status": "already_closed", "reason": "order_completed"} |
| «В работе» / «Новый» / активный   | Продолжить отмену | —                                                                   |

**Шаг 3.** Отмена резерва (для активных заказов):

НачатьТранзакцию()

ДокументОбъект = НайденныйЗаказ.ПолучитьОбъект()

ДокументОбъект.ОтменитьПроведение()

// Автоматически снимает записи в регистре ЗаказыПокупателей (резерв)

// Проверить что Stock_Reserved уменьшился

ЗафиксироватьТранзакцию()

**Шаг 4.** После успешной отмены — обновить Fast Track:

Для каждой строки табличной части «Товары» отменённого заказа записать в Очередь_Fast_Track: GUID, Store_ID, DateTime = ТекущаяДатаВремя(), Обработан = Ложь. Стандартный 5-минутный батч подхватит и обновит витрину \[DEC-CK-04\].

**Шаг 5.** Вернуть ответ:

{

"status": "cancelled",

"purchase_order_guid": "a1b2c3d4-e29b-41d4-a716-446655440000",

"purchase_order_number": "ЗКЧ-000123",

"items_released": \[

{"GUID": "550e8400...", "qty_released": 3}

\]

}

**Логирование:**

- Каждый вызов: уровень **Информация**, "cancel_reserve \| CRM_Deal_ID={x} \| Статус={result}"

- При отмене активного заказа: "Резерв снят \| GUID={guid} Store={store} Qty={qty}"

#### Критерии приёмки (DoD)

- После вызова: Stock_Reserved по каждой позиции отменённого заказа уменьшился на соответствующий Qty

- Stock_Free по каждой позиции увеличился на соответствующий Qty

- Повторный вызов с тем же CRM_Deal_ID (заказ уже отменён) → HTTP 200, already_cancelled без ошибки

- Вызов для завершённого заказа → HTTP 200, already_closed без попытки отмены

- Пары \[GUID, Store_ID\] добавлены в Очередь_Fast_Track — в течение 5 минут витрина показывает обновлённый Stock_Free

- Запрос без X-API-Key → HTTP 401

#### Ошибки и исключения

| **Ситуация**                    | **HTTP-код** | **Ответ**                                                 |
|---------------------------------|--------------|-----------------------------------------------------------|
| CRM_Deal_ID не найден           | 404          | {"error": "order_not_found"}                              |
| Неверный API-ключ               | 401          | —                                                         |
| Ошибка транзакции 1С при отмене | 500          | {"error": "internal_error", "message": "..."}             |
| Заказ уже отменён               | 200          | {"status": "already_cancelled"}                           |
| Заказ закрыт (выполнен)         | 200          | {"status": "already_closed", "reason": "order_completed"} |

#### Зависимости

- Автоотмена через 72ч по \[DEC-32\] — инициируется CRM-роботом Битрикс24, не регламентным заданием 1С

- Реквизит УФ_CRM_Deal_ID в «Заказе покупателя» — задача 1С-ИНФ-01 (блок Discovery & Selection)

## 1С-CK-03 — Вебхук в CRM при пробитии чека

**Инициатива / Эпик:** Корзина и Чекаут / Платёжный гейтвей

**User Story:** Как менеджер CRM, я хочу мгновенно видеть в Битрикс24 что клиент оплатил и получил товар на кассе (чек пробит), чтобы автоматически переводить сделку в статус «Успешно реализовано» без ручного контроля.

**Описание задачи:** При проведении «Чека ККМ» в 1С, связанного с «Заказом покупателя» (e-com заказ), отправить вебхук в Битрикс24 с событием check_printed.

### ⚠️ БЛОКИРУЮЩИЙ ВОПРОС — ответить до начала реализации

Реализация зависит от ответов List.kg на следующие вопросы о конфигурации:

| **\#** | **Вопрос**                                                                                                                                                                         |
|--------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1      | В вашей конфигурации «Чек ККМ» создаётся кассиром через «Создать на основании» из «Заказа покупателя» — или кассир создаёт чек вручную и подбирает товары самостоятельно?          |
| 2      | Есть ли в вашей конфигурации реквизит ДокументОснование в «Чеке ККМ»? Заполняется ли он автоматически при создании через «Создать на основании»?                                   |
| 3      | Готовы ли добавить кастомный реквизит УФ_CRM_Deal_ID (строка, 50 символов) в «Чек ККМ» с автозаполнением при создании на основании «Заказа покупателя»? (см. 1С-ИНФ-01 ДОПОЛНЕНИЕ) |

## 1С-CK-04 — Автозаполнение УФ_Purchase_Order_GUID при проведении «Чека ККМ»

**Инициатива / Эпик:** Корзина и Чекаут / Платёжный гейтвей

> ✅ **NEW в v2.0. \[FIX-1C-01\]** Задача отсутствовала в v1.0. Добавлена по результатам аудита цепочки идемпотентности /close_order → /create_return для физлиц.

**User Story:** Как архитектор, я хочу гарантировать что «Чек ККМ», созданный через /close_order, всегда содержит обратную ссылку на «Заказ покупателя» в виде УФ_Purchase_Order_GUID — чтобы идемпотентность /close_order и последующий вызов /create_return работали корректно для физлиц.

**Описание задачи:** При проведении «Чека ККМ», созданного на основании «Заказа покупателя», автоматически заполнить реквизит УФ_Purchase_Order_GUID = ЗаказОснование.УникальныйИдентификатор(). Реализация: подписка ПослеПроведения «Чека ККМ».

> **Контекст и мотивация:**

- /close_order для физлиц создаёт «Чек ККМ» и записывает в него УФ_Purchase_Order_GUID напрямую (см. 1С-LM-01, блок Last Mile)

- Однако если «Чек ККМ» создаётся кассиром вручную (через «Создать на основании») **без** вызова /close_order, поле УФ_Purchase_Order_GUID останется пустым

- Это ломает идемпотентность: повторный вызов /close_order не найдёт существующий чек и создаст дубль

- Данная подписка закрывает этот gap: **в любом случае** после проведения чека с основанием-заказом поле будет заполнено

- Аналог для юрлиц: 1С-LM-02 (подписка ПослеПроведения «Реализации товаров», блок Last Mile)

#### Техническая реализация в 1С

**Триггер:** Подписка на событие ПослеПроведения документа «Чек ККМ».

> **Рекомендация по реализации:** Если задача 1С-CK-03 уже реализована и подписка ПослеПроведения «Чека ККМ» существует — добавить логику УФ_Purchase_Order_GUID в ту же процедуру **перед** отправкой вебхука. Это сократит количество отдельных подписок на один документ.
