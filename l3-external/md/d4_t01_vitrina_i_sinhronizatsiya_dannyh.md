## Общие архитектурные ограничения (обязательно к прочтению)

- **Прямые вызовы CMS → 1С запрещены архитектурно.** Только CRM → 1С.

- **1С — мастер цен, остатков и документов.** CMS хранит только то, что получила от 1С или BI.

- Два трека данных из 1С: **Slow Track** (FTP, 02:00) и **Fast Track** (REST, каждые 5 мин).

- Price_Promo — строго JSON null (не 0, не "", не строка "null") при отсутствии акции.

- Stock_Free = MAX(0, Physical - Reserved) — никогда не отрицательный (гарантирует 1С).

- ACTIVE управляется **только Slow Track**. Fast Track управляет только Stock_Free и ценами — **никогда ACTIVE**.

- GUID товара (XML_ID в инфоблоке) — первичный ключ для матчинга во всех системах. UUID4, строка без фигурных скобок.

## ИНФРАСТРУКТУРНЫЙ БЛОК

### INF-DS-01 — Slow Track consumer: опрос FTP и импорт bitrix_catalog.csv

**Назначение:** CMS забирает ночной каталог из BI-пайплайна и обновляет инфоблок.

**Механизм доставки:**

| **Параметр**          | **Значение**                                                                                     |
|-----------------------|--------------------------------------------------------------------------------------------------|
| FTP-сервер            | Общий с 1С Slow Track (выдаёт List.kg)                                                           |
| Путь файла (BI пишет) | /bi_catalog/YYYY-MM-DD/bitrix_catalog.csv                                                        |
| Сигнал готовности     | /bi_catalog/YYYY-MM-DD/\_DONE.flag — записывается BI строго **после** успешной записи файла      |
| FTP-аккаунт CMS       | cms_reader — read-only, путь /bi_catalog/ (создаёт List.kg)                                      |
| Расписание опроса     | Каждые 30 мин с 04:00. При обнаружении \_DONE.flag — немедленный импорт                          |
| Хранение              | 7 дней. CMS читает только актуальный пакет (сравнивает дату с последним успешно импортированным) |
| Atomicity             | BI пишет во временное имя → переименовывает → пишет флаг. CMS не читает файл без флага           |

**Алгоритм импорта (AC):**

1.  Проверить наличие \_DONE.flag в папке текущей даты

2.  Если флаг есть и дата папки \> last_imported_date → начать импорт

3.  Разобрать bitrix_catalog.csv (UTF-8 BOM, разделитель ;)

4.  Для каждой строки: найти элемент инфоблока по XML_ID = GUID

    - Если найден → обновить поля (см. маппинг ниже)

    - Если не найден → создать новый элемент

5.  Применить правила управления ACTIVE (см. INF-DS-01a ниже)

6.  Зафиксировать last_imported_date = текущая дата

7.  При ошибке парсинга любой строки: логировать в системный лог Bitrix с GUID строки, **продолжить** (не прерывать весь импорт). Повторить строку на следующем цикле невозможно — логировать для ручного разбора

8.  При ошибке чтения FTP: логировать, повторить на следующем цикле опроса. Флаг не удалять

### INF-DS-01a — Правила управления ACTIVE \[CMS\]

**Приоритет применяется строго сверху вниз:**

| **Приоритет** | **Условие**                    | **ACTIVE**    | **Поведение**                                         |
|---------------|--------------------------------|---------------|-------------------------------------------------------|
| 1 (высший)    | Is_Toxic_Offline = 1           | **N**         | Принудительно, независимо от Ecom_Status              |
| 2             | Ecom_Status = Toxic_Online     | **N**         | Скрыт из каталога и поиска                            |
| 3             | Ecom_Status = Review_Required  | **N**         | Скрыт до ручного перевода администратором             |
| 4             | Ecom_Status = Stand_Alone      | **Y**         | Полноценная карточка                                  |
| 5             | Ecom_Status = Add_On_Only      | **Y**         | Карточка есть, кнопка корзины заблокирована (AC ниже) |
| —             | Stock_Free = 0 (из Fast Track) | **НЕ МЕНЯЕТ** | Карточка остаётся ACTIVE=Y → режим «Под заказ»        |

**Правило EDGE-04:** Если ACTIVE = N → все входящие Fast Track батчи по этому GUID **игнорировать** (не обновлять цены и остатки).

**AC для Add_On_Only \[COM-PRE-01, COM-CK-02\]:**

- Элемент инфоблока: ACTIVE = Y

- При самостоятельном добавлении в корзину: показывать сообщение «Этот товар доступен только в комплекте с другим товаром» — кнопка «В корзину» заблокирована

- Cross-sell/bundle компонент загружает Add_On_Only-товары явным GUID-запросом, игнорируя фильтр ACTIVE

### INF-DS-02 — Fast Track endpoint POST /update_fast_track \[CMS\]

**Назначение:** Приём батчей обновлений цен и остатков от 1С каждые 5 минут.

**Спецификация эндпоинта:**

| **Параметр**    | **Значение**                                                                    |
|-----------------|---------------------------------------------------------------------------------|
| URL             | /update_fast_track (кастомный REST API Битрикс)                                 |
| Метод           | POST                                                                            |
| Content-Type    | application/json                                                                |
| Аутентификация  | Заголовок X-Api-Key: {secret} + IP-whitelist офисного сервера 1С \[COM-PRE-04\] |
| Идемпотентность | Повторный батч с теми же данными = безопасная перезапись                        |

**Формат входящего JSON:**

\[

{

"GUID": "e2a3b1f4-5c6d-7e8f-9a0b-1c2d3e4f5a6b",

"Store_ID": "store_belovodsk_01",

"Stock_Free": 14,

"Price_Retail": 1200.00,

"Price_B2B": 1050.00,

"Price_Ecom": 1150.00,

"Price_Promo": 990.00

},

{

"GUID": "e2a3b1f4-5c6d-7e8f-9a0b-1c2d3e4f5a6b",

"Store_ID": "store_kokzhar_02",

"Stock_Free": 0,

"Price_Retail": 1200.00,

"Price_B2B": 1050.00,

"Price_Ecom": 1150.00,

"Price_Promo": null

}

\]

> Один товар на N складах = **N объектов** в массиве (по ключу \[GUID, Store_ID\]).

**Алгоритм обработки (AC):**

1.  Валидировать X-Api-Key. Если неверный → HTTP 401

2.  Для каждого объекта в массиве: a. Найти элемент инфоблока по XML_ID = GUID b. Если не найден **или** ACTIVE = N → **пропустить объект** (EDGE-04), логировать как INFO c. Обновить stock_free в таблице b_dom_stocks по ключу \[guid, store_id\] (upsert) d. Пересчитать Total_Stock_Free = SUM(stock_free) из b_dom_stocks WHERE guid = ? e. Обновить цены в инфоблоке:

    - Price_Promo != null AND Price_Promo \> 0 → установить Price_Promo как активную акционную, Price_Ecom — зачёркнутая

    - Price_Promo = null → только Price_Ecom, акционная цена снять

    - Price_Promo = 0 → **аномалия** \[COM-PRE-02\], логировать как WARN ("Fast Track anomaly: Price_Promo=0 for GUID={guid}"), трактовать как null

3.  Вернуть HTTP 200 после обработки **всего** батча

4.  При ошибке обработки батча → HTTP 500. 1С повторит в следующем цикле

**Ответ:**

HTTP 200 OK

{"status": "ok", "processed": N, "skipped": M}

### INF-DS-03 — Механизм window.activeStoreId \[CMS\]

**Назначение:** Глобальный JS-контекст активного склада. Используется в карточке товара, калькуляторе и cross-sell.

**Правила \[COM-PRE-03\]:**

| **Ситуация**                           | **Значение activeStoreId**                                          |
|----------------------------------------|---------------------------------------------------------------------|
| Первый визит                           | Константа STORE_DEFAULT_ID (главный склад, настраивается в админке) |
| Пользователь выбрал склад              | Значение из cookie: active_store_id (TTL: 30 дней)                  |
| Cookie повреждена / значение невалидно | Fallback на STORE_DEFAULT_ID                                        |

**Компонент «Переключатель склада»** (шапка сайта):

- Отображает название текущего активного склада

- При клике: выпадающий список всех активных складов

- Список складов: берётся из справочника UF_STORES (заполняется при деплое, список Store_ID → название)

- При выборе: обновить cookie active_store_id, обновить window.activeStoreId, перерендерить блоки остатков и cross-sell без перезагрузки страницы

**Хранение остатков по складам в Битрикс:**

Использовать кастомную таблицу b_dom_stocks (не JSON-поле инфоблока). Решение масштабируется на сеть дарксторов без изменения кода.

CREATE TABLE b_dom_stocks (

id INT AUTO_INCREMENT PRIMARY KEY,

guid VARCHAR(36) NOT NULL,

store_id VARCHAR(50) NOT NULL,

stock_free INT NOT NULL DEFAULT 0,

updated_at DATETIME NOT NULL,

UNIQUE KEY uk_guid_store (guid, store_id),

INDEX idx_guid (guid)

);

**Запись при обработке Fast Track батча (upsert):**

INSERT INTO b_dom_stocks (guid, store_id, stock_free, updated_at)

VALUES (?, ?, ?, NOW())

ON DUPLICATE KEY UPDATE stock_free = VALUES(stock_free), updated_at = NOW();

**Чтение в карточке товара:**

SELECT store_id, stock_free FROM b_dom_stocks WHERE guid = ?

Total_Stock_Free = SUM(stock_free) по guid — агрегировать в PHP при выборке. Хранить отдельно в CATALOG_QUANTITY не нужно — считать на лету из таблицы.

## ЭПИК 1 — Поиск по нормальным названиям \[CMS\]

**User Story:** Как клиент, я хочу находить товары по понятным названиям («Гипсокартон») и аббревиатурам («ГКЛ»), чтобы быстро найти нужное.

### Что получает от систем

- **Slow Track (BI → CMS):** Name_Clean (отображаемое), Raw_Name (скрытое), Action_Flag из bitrix_catalog.csv

### Требования к реализации \[CMS\]

**Индексирование:**

- Name_Clean — основное отображаемое название. Индексируется поиском

- Raw_Name — **скрытое свойство** UF_RAW_NAME. Клиенту не отображается. **Обязательно индексируется поиском** (для матча по аббревиатурам типа «ГКЛ»)

- Поиск: игнорировать регистр и окончания (настройки компонента поиска Аспро / встроенного поиска Битрикс)

**Обработка Name_Clean = NULL:**

- BI дублирует Raw_Name в Name_Clean при невозможности сформировать — это логика BI, не CMS

- CMS обязана отображать Name_Clean всегда. Если при импорте Name_Clean пустой (нарушение контракта) → использовать Raw_Name, зафиксировать в лог

**Action_Flag = "Manual_Rename":**

- В **админке** (список элементов инфоблока) добавить колонку Action_Flag с цветовой пометкой

- Администратор видит какие товары требуют ручной правки Name_Clean

- На **витрине** Action_Flag не отображается

### DoD

- Поиск по Name_Clean возвращает корректный результат (без учёта регистра)

- Поиск по значению из Raw_Name (аббревиатура) возвращает карточку с Name_Clean

- В adminке колонка Action_Flag видна и фильтруема

- Товары без Name_Clean (экстренный сценарий) отображаются с Raw_Name и не падают

### Ошибки и исключения

| **Ситуация**            | **Поведение**                                   |
|-------------------------|-------------------------------------------------|
| Name_Clean пустой в CSV | Использовать Raw_Name, логировать GUID          |
| Raw_Name тоже пустой    | Использовать GUID как заглушку, логировать      |
| FTP недоступен          | Продолжать работу на данных предыдущего импорта |

## ЭПИК 2 — Визуальные подсказки в поиске \[CMS\]

**User Story:** Как клиент, я хочу видеть фото и цену в выпадающем поиске, чтобы ориентироваться не заходя в карточку.

### Что получает от систем

- **Fast Track:** Price_Ecom, Price_Promo

- **CMS (мастер фото):** фотографии товаров (загружаются менеджером, 1С не источник фото)

### Требования к реализации \[CMS\]

**AJAX-поиск (компонент Аспро или кастомный):**

- Триггер: пользователь ввёл ≥2 символов, задержка 500–800мс

- В выпадающем списке для каждого результата: миниатюра, Name_Clean, цена

- **Логика отображения цены:**

  - Price_Promo != null → зачёркнутая Price_Ecom + акционная Price_Promo

  - Price_Promo = null → только Price_Ecom

  - Price_Ecom = 0 или отсутствует → строка «Цена по запросу» (не «0 сом»)

- **Фото:** если фото нет → placeholder (иконка категории или дефолтное изображение). Битая ссылка недопустима

- Клик на результат → переход в карточку товара

### DoD

- Подсказка появляется ≤800мс после ввода

- Фото и цена отображаются корректно для всех AC (см. USM Эпик 2)

- Нет «0 сом» и битых img-ссылок

- Акционная цена отображается только при Price_Promo != null AND Price_Promo \> 0

### Ошибки и исключения

| **Ситуация**         | **Поведение**                                         |
|----------------------|-------------------------------------------------------|
| Price_Promo = 0      | Трактовать как null \[COM-PRE-02\]. Только Price_Ecom |
| Фото отсутствует     | Placeholder. Не ломать вёрстку                        |
| 0 результатов поиска | «Ничего не найдено». Предложить категории             |

## ЭПИК 3 — Рабочий фильтр \[CMS\]

**User Story:** Как клиент, я хочу сужать выбор фильтрами по характеристикам, чтобы не листать весь каталог.

### Что получает от систем

- **Slow Track (BI → CMS):** Attr_Brand, Attr\_\* из bitrix_catalog.csv

### Требования к реализации \[CMS\]

**Свойства фильтра:**

- Каждое поле Attr\_\* из bitrix_catalog.csv → свойство инфоблока типа «Список»

- CMS **не нормализует** значения атрибутов — нормализация на стороне BI

- При импорте: пустая строка в ячейке → свойство не заполняется (не перезаписывать существующее значение пустотой)

- Значение Attr_Brand — отдельный фильтр «Бренд»

**Поведение фильтра:**

- Фасетный фильтр (AJAX, без перезагрузки страницы)

- Счётчик «Найдено X товаров» обновляется при каждом изменении фильтра

- Дублей значений в списке фильтра нет (Битрикс дедуплицирует автоматически через свойства типа «Список»)

- Комбинация фильтров с 0 результатов → сообщение «Товаров не найдено» + кнопка «Сбросить фильтры»

> ⚠️ **OQ-BC-03a:** Полный список Attr\_\* колонок финализирует Data Engineer. Реализацию фильтра выполнить динамически: читать список колонок из заголовка файла при импорте, создавать свойства автоматически. Это не блокирует остальную реализацию.

### DoD

- Фильтр по Attr_Brand работает

- При выборе значения список товаров обновляется без перезагрузки

- Сброс фильтров работает

- Пустая строка в Attr\_\* не создаёт пустое значение в списке фильтра

## ЭПИК 4 — Понятное ценообразование \[CMS\]

**User Story:** Как клиент, я хочу видеть цену за м² и за упаковку и покупать кратно упаковкам, потому что склад не продаёт полкоробки.

### Что получает от систем

- **Slow Track (BI → CMS):** Unit_Base, Unit_Alt, Ratio из bitrix_catalog.csv

- **Fast Track:** Price_Ecom — базовая цена за Unit_Base

### Требования к реализации \[CMS\]

**Логика отображения цены в карточке:**

| **Условие**                       | **Отображение**                                                                                                    |
|-----------------------------------|--------------------------------------------------------------------------------------------------------------------|
| Unit_Alt не пустой AND Ratio \> 0 | Переключатель «Unit_Base / Unit_Alt». При выборе Unit_Alt: Price_m2 = Price_Ecom / Ratio (вычислять клиентским JS) |
| Ratio = NULL или Ratio \<= 0      | Переключатель не выводить. Только Price_Ecom. Action_Flag = Ratio_Error в adminке                                  |
| Unit_Alt пустой                   | Переключатель не выводить                                                                                          |

**Шаг покупки:**

- Шаг поля «Количество» = 1 Unit_Base. Дробные значения недоступны

- Корзина = Qty × Price_Ecom (в Unit_Base)

### DoD

- Переключатель единиц работает (примеры: упак ↔ м², шт ↔ пог.м)

- Price_m2 вычисляется клиентским JS корректно

- Дробный ввод количества заблокирован

- При Ratio_Error переключатель скрыт, карточка работает с Price_Ecom

## ЭПИК 5 — Калькулятор потребности \[CMS\]

**User Story:** Как клиент, я хочу ввести площадь и получить готовое количество упаковок для заказа.

### Что получает от систем

- **Slow Track (BI → CMS):** Ratio, Unit_Base из bitrix_catalog.csv (зависит от Эпика 4)

- **Fast Track:** Stock_Free\[activeStoreId\]

### Требования к реализации \[CMS\]

**Показывать только для категорий:** плитка, ламинат, паркет (настраиваемый список категорий в константах)

**Формула:**

- Стандарт: ceil(area / Ratio)

- Чекбокс «+10% на отходы» (по умолчанию включён): ceil(area / Ratio × 1.1)

**AC:**

| **\#** | **Условие**                                     | **Результат**                                                                       |
|--------|-------------------------------------------------|-------------------------------------------------------------------------------------|
| 5.1    | Ratio = 1.44, клиент ввёл 15 м²                 | ceil(15 / 1.44) = 11 упак. Подставить в поле количества, пересчитать сумму          |
| 5.2    | Расчётное кол-во \> Stock_Free\[activeStoreId\] | Предупреждение: «На складе доступно X уп. Добавить X уп. и связаться с менеджером?» |
| 5.3    | Ratio = NULL или Ratio \<= 0                    | Калькулятор скрыть                                                                  |

**window.activeStoreId для проверки остатка:** читать из b_dom_stocks WHERE guid = ? AND store_id = activeStoreId.

### DoD

- Калькулятор отображается только для разрешённых категорий

- Формула с поправкой на отходы работает корректно

- Предупреждение при превышении остатка активного склада

## ЭПИК 6.1 — Актуальный остаток по складам \[CMS\]

**User Story:** Как клиент, я хочу знать в каком магазине товар есть прямо сейчас.

### Что получает от систем

- **Fast Track:** {GUID, Store_ID, Stock_Free, ...} — батч каждые 5 мин

### Требования к реализации \[CMS\]

**Статус товара на витрине (каталог и карточка):**

| **Условие**           | **Отображение**                                                         |
|-----------------------|-------------------------------------------------------------------------|
| Total_Stock_Free \> 0 | Зелёная плашка «В наличии»                                              |
| Total_Stock_Free = 0  | Статус «Под заказ / Сообщить о поступлении». Блок «Где купить» — скрыть |

**Попап «Где купить»** (кнопка на карточке, только при Total_Stock_Free \> 0):

- Список строк по каждому складу: «Название склада — В наличии (X шт)» / «Название склада — Нет в наличии»

- Данные: SELECT store_id, stock_free FROM b_dom_stocks WHERE guid = ?

- Названия складов: словарь Store_ID → Название из настроек (UF_STORES)

### DoD

- Статус «В наличии» / «Под заказ» обновляется в пределах 5 мин после изменения в 1С

- Попап отображает корректные данные по каждому складу

- Попап скрыт при Total_Stock_Free = 0

## ЭПИК 6.2 — Защита витрины от отрицательных остатков \[CMS\]

**User Story:** Как e-com директор, я хочу транслировать только гарантированно свободные остатки.

### Гарантии на стороне 1С (информационно):

- Stock_Free = MAX(0, Physical - Reserved) — 1С **никогда** не отправляет отрицательное значение

- При аномалии (Physical \< Reserved) — WARN в журнал 1С, в батч уходит Stock_Free = 0

### Требования к реализации \[CMS\]:

- Принимать Stock_Free как есть из батча — дополнительных проверок на отрицательность не нужно (гарантирует 1С)

- Stock_Free = 0 по всем складам → статус «Под заказ» (не ACTIVE=N)

- Не хранить отрицательные значения в b_dom_stocks. Если по какой-то причине пришло отрицательное (нарушение контракта) → принудительно хранить 0, логировать аномалию

### DoD

- В таблице b_dom_stocks нет отрицательных значений stock_free

- Total_Stock_Free никогда не отрицательный

## ЭПИК 7.1 — Управление видимостью: OOS и токсичные \[CMS\]

**User Story:** Как маркетолог, я хочу сохранять карточки временно OOS-товаров для SEO, но автоматически скрывать токсичные.

### Что получает от систем

- **Slow Track (BI → CMS):** Is_Toxic_Offline, Ecom_Status из bitrix_catalog.csv

- **Fast Track:** Stock_Free (управляет только остатком, не ACTIVE)

### Требования к реализации \[CMS\]

**Правила ACTIVE:** применять строго по таблице из INF-DS-01a (приоритет 1–5).

**Ключевые сценарии:**

| **\#** | **Условие**                                    | **Поведение**                                                                                     |
|--------|------------------------------------------------|---------------------------------------------------------------------------------------------------|
| 7.1.1  | ACTIVE=Y, Stock_Free=0 (Fast Track)            | Карточка видима. Кнопка «В корзину» скрыта. Блок: «Под заказ» + кнопка «Сообщить о поступлении»   |
| 7.1.2  | Is_Toxic_Offline=1 (Slow Track, ночной импорт) | ACTIVE=N. Скрыт из каталога и поиска                                                              |
| 7.1.3  | ACTIVE=N, Fast Track пытается обновить остаток | Батч по этому GUID **игнорируется** (EDGE-04). Карточка остаётся скрытой до следующего Slow Track |
| 7.1.4  | Ecom_Status = Review_Required                  | ACTIVE=N. Ручной перевод в Stand_Alone администратором в adminке                                  |

### DoD

- ACTIVE управляется только Slow Track

- Fast Track не может изменить ACTIVE

- Карточки с Stock_Free=0 видимы (ACTIVE=Y), кнопка корзины скрыта

- Токсичные скрыты (Is_Toxic_Offline=1 → ACTIVE=N) до следующего Slow Track с 0

## ЭПИК 7.2 — Лист ожидания → CRM \[CMS + CRM\]

**User Story:** Как РОП, я хочу чтобы заявки «Под заказ» мгновенно падали менеджерам в CRM с задачей предложить аналог.

### Что получает от систем

- Пользователь заполняет форму «Сообщить о поступлении» на карточке OOS-товара

### Требования к реализации \[CMS\]

**Форма «Сообщить о поступлении»:**

- Отображать только при ACTIVE=Y AND Total_Stock_Free = 0

- Поля: телефон (обязательное), имя (опциональное)

- При отправке: CMS передаёт в CRM через REST API Битрикс24 payload:

> {
>
> "phone": "+996XXXXXXXXX",
>
> "GUID": "uuid4",
>
> "Name_Clean": "Название товара",
>
> "active_store_id": "store_belovodsk_01"
>
> }

- После успешной отправки: показать клиенту «Мы уведомим вас по номеру +996XXXXXXX когда товар появится в наличии»

### Требования к реализации \[CRM\]

**Воронка «Лист ожидания»:**

| **\#**        | **Условие**                                       | **Действие CRM**                                                                                                                                                      |
|---------------|---------------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 7.2.1         | Нет открытой Сделки с \[phone + GUID\]            | Создать Сделку: поля phone, Name_Clean, GUID, UF_ACTIVE_STORE_ID, UF_LAST_REQUEST_DATE = now(). Робот назначает задачу менеджеру: «Предложить аналог по {Name_Clean}» |
| 7.2.2 (дедуп) | Открытая Сделка с \[phone + GUID\] уже существует | Новую Сделку **не создавать**. Только обновить UF_LAST_REQUEST_DATE = now(). Робот задачу **не повторяет** (EDGE-07)                                                  |

**Тип CRM-сущности:** Сделка (не Лид). Воронка: «Лист ожидания».

### DoD

- Форма отображается только на OOS-карточках

- Сделка создаётся при первой заявке

- При повторной заявке с тем же телефоном на тот же GUID — только обновление даты, без новой Сделки

- Менеджер получает задачу только при создании новой Сделки

- Клиент видит подтверждение с его номером телефона

### Ошибки и исключения

| **Ситуация**                  | **Поведение**                                                                                       |
|-------------------------------|-----------------------------------------------------------------------------------------------------|
| REST API Битрикс24 недоступен | Логировать, показать клиенту «Произошла ошибка, попробуйте позже». Не хранить заявку на стороне CMS |
| Некорректный формат телефона  | Валидация на фронтенде. Показать подсказку формата                                                  |
| GUID товара не передан        | Логировать ошибку, форму скрыть (нет смысла без GUID)                                               |
