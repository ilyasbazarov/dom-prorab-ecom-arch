## Общие архитектурные ограничения

- Вебхук Яндекса delivered = триггер закрытия документов для доставки. Ручного подтверждения нет \[DEC-LM-07\]

- Фото-пруф доставки от водителя — в чате Яндекс.GO, вне систем Дом Прораба \[CON-12 закрыт, FIX-1C-08\]

- Страховка Яндекс.GO в Бишкеке недоступна. declared_value передаётся для фискальных целей \[CON-13\]

- Клиентский трекинг — Яндекс автоматически отправляет SMS по телефону получателя. Без IT-интеграции со стороны Дом Прораба \[CON-14\]

- Обратная логистика для рекламаций — ручное оформление через корпоративный кабинет Яндекс.GO. API-интеграции нет \[CON-17\]

- closing_doc_guid = GUID закрывающего документа (Чек ККМ или Реализация). Используется в /create_return и /create_rko

- return_id = ID смарт-процесса «Возвраты» в Битрикс24. Ключ идемпотентности /create_return и /create_rko

## Эпик 22 — Маршрутизация и трекинг доставки \[CRM\]

**User Story:** Как клиент — хочу знать где едет моя машина. Как бизнес — хочу автоматически создавать заявки Яндекс без ручного ввода.

### 22.1 — Создание заявки Яндекс (IC-LM-01)

**Триггер:** кнопка «Сборка завершена» в карточке сделки при UF_DELIVERY_TYPE = "delivery" — CRM-робот переводит сделку в статус «Сборка завершена» и запускает создание заявки Яндекс \[COM-R3-01, DEC-B2Q3\].

**CRM-робот вызывает POST /b2b/cargo/integration/v2/claims/create:**

{

"callback_properties": {

"callback_url": "https://crm.domprorab.kg/api/yandex-delivery/webhook?token={secret}"

},

"items": \[{/\* состав из товарных строк сделки \*/}\],

"route_points": \[

{

"address": "{адрес склада по UF_SOURCE_STORE_ID}",

"type": "source",

"contact": {"name": "Дом Прораба", "phone": "{телефон склада}"}

},

{

"address": "{адрес доставки из сделки}",

"type": "destination",

"contact": {"name": "{имя клиента}", "phone": "{UF_CLIENT_PHONE}"}

}

\],

"transport_type": "{расчётный \| van_xcargo}",

"declared_value": "{OPPORTUNITY сделки — нативное поле суммы Битрикс24}",

"cargo_options": {"size_check": true}

}

**Правила формирования payload:**

| **Поле**                              | **Логика**                                                                                                                    |
|---------------------------------------|-------------------------------------------------------------------------------------------------------------------------------|
| declared_value                        | Нативное поле OPPORTUNITY сделки Битрикс24. Автоматически \[DEC-43, COM-R3-02\]                                               |
| transport_type                        | Если хотя бы один товар UF_IS_OVERSIZE = true → van_xcargo (XL). Иначе — по UF_TOTAL_VOLUME из справочника тарифов \[DEC-28\] |
| Время подачи                          | DATE_PICKUP - delivery_claim_lead_time_hours (дефолт 2ч) \[DEC-40\]                                                           |
| callback_url.token                    | Секрет для аутентификации входящих вебхуков Яндекса. Хранится в константах CRM                                                |
| Телефон клиента в destination.contact | Яндекс автоматически отправляет SMS-трекинг клиенту \[CON-14\]                                                                |

**После успешного ответа Яндекса:**

- Сохранить claim_id → UF_YANDEX_CLAIM_ID сделки \[COM-LM-03\]

- Сделка → «Ожидает водителя»

**Перенос и отмена (AC-22.5, 22.6):**

| **Условие**              | **Действие CRM**                               |
|--------------------------|------------------------------------------------|
| Водитель ещё не назначен | claims/edit — изменить параметры заявки        |
| Водитель уже назначен    | claims/cancel + новый claims/create \[DEC-42\] |

### 22.2 — Приём вебхуков от Яндекса (IC-LM-02) \[CMS endpoint\]

**Endpoint CMS:** POST /api/yandex-delivery/webhook?token={secret}

> Endpoint регистрируется на стороне **CMS** (Битрикс). Принимает HTTP-запросы от Яндекс и передаёт в CRM.

**Аутентификация:** проверить token из query string. Несовпадение → HTTP 401, игнорировать.

**Идемпотентность:** таблица b_dom_yandex_webhooks, ключ (claim_id, status).

**⚠️ OQ-LM-01:** IP-allowlist Яндекса — уточнить у аккаунт-менеджера Яндекс при подключении к B2B API.

**Маршрутизация по статусу:**

| **Яндекс status** | **Действие CRM**                                                                                                                        |
|-------------------|-----------------------------------------------------------------------------------------------------------------------------------------|
| driver_assigned   | Найти сделку по UF_YANDEX_CLAIM_ID. Вызвать claims/info → получить version. Вызвать claims/accept (AC-22.3). Сделка → «Передан курьеру» |
| delivering        | Обновить статус ЛК Phase 2. Вызовов 1С нет                                                                                              |
| delivered         | Вызвать **IC-LM-05** (/close_order). Триггер WA-накладной (IC-LM-07, Эпик 23)                                                           |
| cancelled         | Алерт менеджеру. Задача: «Перезаказать доставку или связаться с клиентом»                                                               |
| driver_not_found  | Алерт менеджеру: «Яндекс не нашёл водителя для заявки {claim_id}»                                                                       |

**DoD:**

- Webhook принимается и маршрутизируется по всем статусам

- Повторный webhook с тем же (claim_id, status) → HTTP 200 без действий

- Token-аутентификация работает: неверный token → 401

### 22.3 — Подтверждение заявки (claims/accept)

**Триггер:** Яндекс driver_assigned → CRM получил claim_id.

**Последовательность:**

1.  GET /b2b/cargo/integration/v2/claims/info?claim_id={id} → получить актуальную version

2.  POST /b2b/cargo/integration/v2/claims/accept → {"claim_id": "...", "version": N}

3.  При успехе: сделка → «Передан курьеру»

4.  При ошибке: алерт менеджеру

## Эпик 23 — WA-накладная при доставке (IC-LM-07) \[CRM\]

**Триггер:** статус сделки → «Успешно реализовано» (через delivered от Яндекса или check_printed для самовывоза физлиц).

**CRM-робот отправляет WA (HSM-шаблон order_completed_invoice):**

Заказ №{UF_1C_ORDER_NUMBER} доставлен!

{для каждой товарной строки сделки:}

• {Name_Clean} × {qty} {Unit_Base} — {sum} сом

Итого: {OPPORTUNITY} сом

Спасибо, что выбрали Дом Прораба!

**Важно:** используется **финальный состав** товарных строк сделки на момент отправки (все мутации учтены, AC-23.2).

**Fallback \[COM-PR-07\]:** WA не доставлен за 60 сек → SMS с сокращённым вариантом через SMS-шлюз (AC-23.3).

**DoD:**

- WA отправляется автоматически при переходе сделки в «Успешно реализовано»

- Состав — финальный, после всех мутаций

- SMS-fallback через 60 сек при недоставке WA

## Эпик 24 — Закрытие заказа: самовывоз \[CRM\]

### 24.1 — check_printed маршрутизация (IC-LM-04)

> **Обновляет placeholder COM-PRE-11 из Checkout TZ.** CRM-маршрутизация независима от выбора сценария А/Б на стороне 1С — payload check_printed одинаков в обоих сценариях.

**Endpoint CRM:** POST {CRM_Webhook_Path_CheckPrinted} (отдельный от CRM_Webhook_URL_OrderStatus)

**Входящий payload от 1С:**

{

"CRM_Deal_ID": "12345",

"event": "check_printed",

"receipt_id": "КЧКМ-000456",

"timestamp": "2024-01-15T14:30:00"

}

**AC обработчика:**

1.  Валидировать X-API-Key

2.  Проверить b_dom_processed_1c_events по ключу (CRM_Deal_ID, "check_printed") — идемпотентность

3.  Найти сделку по CRM_Deal_ID

4.  Маршрутизация по UF_DELIVERY_TYPE + UF_CLIENT_TYPE:

UF_DELIVERY_TYPE = "pickup" AND UF_CLIENT_TYPE = "physical":

→ order_completed флоу:

CRM вызывает /close_order(UF_PURCHASE_ORDER_GUID) → IC-LM-05

1С: idempotent: true (Чек уже существует) → возвращает closing_doc_guid

CRM сохраняет closing_doc_guid + closing_doc_number + closing_doc_type в сделку

Сделка → «Успешно реализовано»

Триггер: WA-накладная (Эпик 23)

Сбросить таймер робота «Автоотмена 72ч» (AC-15.3.2 из Checkout TZ)

UF_DELIVERY_TYPE = "pickup" AND UF_CLIENT_TYPE = "legal":

→ Логировать receipt_id в UF_CLOSING_DOC_NUMBER карточки (информационно)

→ order_completed придёт отдельным вебхуком от 1С-LM-02 (IC-LM-06)

→ Сбросить таймер «Автоотмена 72ч»

UF_DELIVERY_TYPE = "delivery":

→ Сбросить таймер «Автоотмена 72ч»

→ order_completed придёт от Яндекс delivered (IC-LM-02)

→ Никаких вызовов 1С здесь

5.  Вернуть HTTP 200

**DoD:**

- Физлицо самовывоз → /close_order вызван, сделка закрыта, WA-накладная отправлена

- Юрлицо самовывоз → receipt_id сохранён, ждём order_completed

- Доставка → таймер сброшен, без вызовов 1С

- Идемпотентный повторный вебхук → HTTP 200 без действий

### 24.2 — Закрытие юрлицо самовывоз: /close_order (IC-LM-05a)

**Триггер:** менеджер нажимает кнопку **«Товар выдан клиенту»** в карточке сделки (статус «Ожидает выдачи»).

> Для юрлиц кассового чека нет — кнопка менеджера является триггером вместо check_printed.

**CRM вызывает POST /close_order:**

{"purchase_order_guid": "{UF_PURCHASE_ORDER_GUID}"}

**Ожидаемый ответ (1С):** создаётся «Реализация товаров»:

{

"status": "ok",

"closing_doc_guid": "b2c3d4e5-...",

"closing_doc_number": "РТ-000456",

"closing_doc_type": "realization",

"idempotent": false

}

После получения ответа: сохранить все три поля в сделку. Сделка → «Ожидает вебхука order_completed» (промежуточный статус).

### 24.3 — Обработчик вебхука order_completed от 1С-LM-02 (IC-LM-06)

**Endpoint CRM:** POST {CRM_Webhook_URL_OrderStatus} — тот же URL что и для order_confirmed / order_cancelled; маршрутизация по полю event \[COM-R3-05\]

**Входящий payload:**

{

"CRM_Deal_ID": "12345",

"purchase_order_guid": "a1b2c3d4-...",

"event": "order_completed",

"timestamp": "..."

}

> Ветка event = "order_completed" добавлена в обработчик INF-PR-01 Processing TZ. Отдельный endpoint CRM_Webhook_URL_OrderCompleted не создаётся \[COM-R3-05\].

**AC:**

1.  Валидировать X-API-Key

2.  Идемпотентность: b_dom_processed_1c_events по ключу (CRM_Deal_ID, "order_completed")

3.  Сделка → «Успешно реализовано» 3a. Если UF_1C_ORDER_NUMBER сделки пустой — проверить payload на наличие purchase_order_number; если есть — сохранить в UF_1C_ORDER_NUMBER

4.  Триггер: WA-накладная (Эпик 23)

5.  HTTP 200

## Эпик 25 — Закрытие заказа: доставка \[CRM\]

**Триггер:** вебхук Яндекса delivered → IC-LM-02 → вызов /close_order.

### 25.1 — /close_order при доставке (IC-LM-05)

**CRM вызывает POST /close_order:**

{"purchase_order_guid": "{UF_PURCHASE_ORDER_GUID}"}

**Матрица ответов:**

| **HTTP** | **Условие**                          | **Действие CRM**                                                                                                                                         |
|----------|--------------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------------------|
| 200      | status: ok, тип check (физлицо)      | Сохранить closing_doc_guid/number/type. Сделка → «Успешно реализовано». WA-накладная \[AC-25.2\]                                                         |
| 200      | status: ok, тип realization (юрлицо) | Сохранить все три поля. Сделка → «Успешно реализовано». WA-накладная \[AC-25.3, 25.4\]                                                                   |
| 200      | idempotent: true                     | Прочитать closing_doc_guid из ответа, сохранить. Без повторных действий \[AC-25.6\]                                                                      |
| 404      | order_not_found                      | Алерт менеджеру. Сделка остаётся в «Передан курьеру»                                                                                                     |
| **409**  | order_cancelled                      | Алерт менеджеру. Логировать. **Не трактовать как ошибку 1С** \[DEC-LM-05\]                                                                               |
| 500      | posting_failed                       | CRM-робот: 3 ретрая с интервалом 5 мин, каждая ошибка логируется. После 3-й неудачи: сделка → «Требует ручного закрытия» + алерт менеджеру \[COM-R3-04\] |

> **HTTP 409 обрабатывается отдельно от 500** — это конфликт состояния ресурса, не системная ошибка 1С.

**AC-25.5 — Webhook delivered не пришёл:**

- Алерт менеджеру если сделка в «Передан курьеру» \> 24ч без изменения

- Ручное закрытие через кнопку «Закрыть вручную» → тот же /close_order вызов

**DoD:**

- delivered → /close_order вызван автоматически

- Все три поля closing_doc из ответа сохранены

- 409 обработан отдельной логикой от 500

- Алерт при отсутствии webhook \> 24ч

# ЧАСТЬ II — ВОЗВРАТЫ (Эпики 26–29)

## Инфраструктура: Смарт-процесс «Возвраты» \[CRM\]

> **\[CON-18\]** Строго отдельная сущность от «Рекламаций». Объединение, наследование или вложенность в «Сделку» как дочерней — контрактный провал.

**Воронка статусов:**

| **Статус**                     | **Кто переводит**                                      |
|--------------------------------|--------------------------------------------------------|
| «Новый»                        | CRM (создание менеджером)                              |
| «В обработке»                  | Менеджер (верификация)                                 |
| «Принято складом»              | Менеджер (физическая приёмка)                          |
| «Ожидает финансового возврата» | CRM (после /create_return)                             |
| «Успешно закрыт»               | CRM (после /create_rko или подтверждения безналичного) |

**Обязательные поля смарт-процесса «Возвраты»:**

| **Поле**               | **Тип**                  | **Откуда**                                | **Назначение**                                                                                       |
|------------------------|--------------------------|-------------------------------------------|------------------------------------------------------------------------------------------------------|
| return_id              | Auto (ID смарт-процесса) | Битрикс24 (автоматически)                 | Ключ идемпотентности /create_return и /create_rko                                                    |
| return_doc_id          | String(36)               | Ответ /create_return (поле return_doc_id) | GUID документа «Возврат товаров от клиента» в 1С — сохранить в карточку; для трассировки и поддержки |
| UF_LINKED_DEAL_ID      | Deal ref                 | Менеджер при создании                     | Связь с исходной сделкой                                                                             |
| UF_CLOSING_DOC_GUID    | String(36)               | Из карточки исходной сделки               | Передаётся в /create_return и /create_rko                                                            |
| UF_RETURN_ITEMS        | Text (JSON)              | Менеджер                                  | Состав возврата \[{guid, qty}\]                                                                      |
| UF_RETURN_REASON       | Text                     | Менеджер                                  | Причина возврата (свободный текст)                                                                   |

## Эпик 28 — Финансовый возврат \[CRM\]

### 28.1 — Безналичный возврат (карта/онлайн)

> **⚠️ ОСОЗНАННЫЙ ГЭП \[COM-LM-05, CON-11\]**
>
> API-интеграция возврата заморожена до выбора эквайринга. MVP: кнопка-инструкция.

**CRM action (MVP) \[DEC-53\]:**

1.  Рассчитать refund_amount = Σ(qty × Price_Ecom) по возвращаемым позициям

2.  Показать кнопку: **«Вернуть {refund_amount} сом»**

3.  При нажатии → инструкция: «Выполнить возврат через кабинет эквайринга. Сумма: {refund_amount} сом. Основание: возврат по заказу {UF_1C_ORDER_NUMBER}»

4.  Менеджер подтверждает → зафиксировать UF_REFUND_CONFIRMED_AT, UF_REFUND_CONFIRMED_BY

5.  WA клиенту (HSM-шаблон return_online_done): «Возврат {refund_amount} сом оформлен. Деньги поступят в течение 3–10 дней» (AC-28.1.2)

6.  Смарт-процесс → «Успешно закрыт». Триггер: NPS (Эпик 29)

### 28.2 — Наличный возврат (самовывоз, оплата на кассе) \[IC-RT-03\]

**Триггер:** менеджер нажимает кнопку **«Выплачено наличными»** в карточке смарт-процесса.

**Расчёт суммы возврата (UF_REFUND_AMOUNT):**

При заполнении состава возврата (UF_RETURN_ITEMS) CRM автоматически рассчитывает:

UF_REFUND_AMOUNT = Σ(qty_returned_i × sale_price_i)

Где:

- qty_returned_i — количество возвращаемых единиц по позиции i из UF_RETURN_ITEMS

- sale_price_i — цена позиции из **товарных строк исходной Сделки** (UF_LINKED_DEAL_ID), **не** текущий Price_Ecom из каталога

> **Почему цена из сделки, а не из каталога:** цена в каталоге могла измениться после продажи. Возврат оформляется по цене, которую клиент фактически заплатил \[COM-R3-03\].

Пересчёт выполняется при каждом изменении UF_RETURN_ITEMS. Поле UF_REFUND_AMOUNT доступно для просмотра в карточке смарт-процесса.

**CRM вызывает POST /create_rko:**

{

"return_id": "42",

"amount": {UF_REFUND_AMOUNT},

"closing_doc_guid": "{UF_CLOSING_DOC_GUID из смарт-процесса}"

}

> amount = поле UF_REFUND_AMOUNT из карточки смарт-процесса «Возвраты». Формула расчёта — см. выше. Значение amount ≤ 0 недопустимо — 1С вернёт HTTP 400.
>
> **Порядок операций**: /create_return должен быть успешно выполнен до /create_rko. CRM гарантирует это: кнопка «Выплачено наличными» активна только после перехода в «Ожидает финансового возврата».

**Матрица ответов:**

| **HTTP** | **Условие**                    | **Действие CRM**                                                                        |
|----------|--------------------------------|-----------------------------------------------------------------------------------------|
| 200      | Успех или idempotent: true     | Смарт-процесс → «Успешно закрыт». Триггер: NPS (AC-28.2.1, 28.2.2)                      |
| 500      | /create_return не был выполнен | Уведомление: «Нарушен порядок операций. Сначала проведите приёмку на склад» (AC-28.2.3) |
| 400      | amount ≤ 0                     | Уведомление менеджеру                                                                   |

**DoD:**

- Кнопка «Выплачено наличными» активна только после «Принято складом»

- После 200 → смарт-процесс «Успешно закрыт»

- Идемпотентность: повторное нажатие → idempotent: true без второго РКО

## Эпик 29 — NPS и пост-возврат \[CRM\]

**User Story:** Как собственник, я хочу знать насколько лёгким был возврат.

### Требования к реализации \[CRM + Wazzup\]

**Триггер:** смарт-процесс «Возвраты» → «Успешно закрыт» + таймер **4 часа**.

**Шаг 1 — отправка NPS-запроса:**

CRM-робот через Wazzup отправляет WA (HSM-шаблон return_nps_request): «Как всё прошло? Оцените сервис возврата от 1 до 5 (ответьте цифрой)»

**Шаг 2 — обработка ответа клиента \[Wazzup incoming messages, AC-29.2\]:**

- Обработчик входящих сообщений Wazzup матчит ответ «1»–«5» по телефону + открытый смарт-процесс «Возвраты»

- Окно матчинга: **48 часов** с момента отправки запроса

- Оценка фиксируется в поле UF_NPS_SCORE карточки смарт-процесса

**Шаг 3 — ответная реакция по оценке:**

| **Оценка**        | **Действие**                                                                                    |
|-------------------|-------------------------------------------------------------------------------------------------|
| 5                 | WA: «Спасибо! Рады что смогли помочь» \[AC-29.3\]                                               |
| 1–3               | WA: «Жаль, что так вышло. Передадим руководству». Задача менеджеру: «Реакция на негативный NPS» |
| 4                 | WA: «Спасибо за оценку!»                                                                        |
| Нет ответа за 48ч | NPS = null, смарт-процесс остаётся «Успешно закрыт» без изменений                               |

> **Промокод «Добор материала»** при NPS=5 — **post-MVP** \[DEC-52\]. На MVP только благодарственное WA.

**Prerequisite:** Wazzup (или аналог) подключён к Битрикс24 и настроен на приём + отправку WA до тестирования.

**DoD:**

- NPS-запрос отправляется через 4ч после «Успешно закрыт»

- Ответ «1»–«5» матчится и фиксируется в карточке

- NPS = 5 → благодарственное WA

- NPS = 1–3 → задача менеджеру
