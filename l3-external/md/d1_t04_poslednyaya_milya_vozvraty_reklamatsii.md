### IC-LM-01: delivery_claim_create — Создание заявки Яндекс (Эпик 22)

| **Параметр**  | **Значение**                                                    |
|---------------|-----------------------------------------------------------------|
| **Событие**   | Сборка завершена, нужно организовать доставку                   |
| **Инициатор** | \[CRM\] CRM-робот при смене статуса сделки → «Сборка завершена» |
| **Откуда**    | Битрикс24 CRM                                                   |
| **Куда**      | \[YA\] POST /b2b/cargo/integration/v2/claims/create             |

**Payload (CRM → Яндекс B2B API):**

{

"callback_properties": {

"callback_url": "https://crm.domprorab.kg/api/yandex-delivery/webhook?token=XXXXXXXX"

},

"items": \[/\* состав заказа из товарных строк сделки \*/\],

"route_points": \[

{

"address": "Адрес склада UF_SOURCE_STORE_ID",

"type": "source",

"contact": { "name": "Дом Прораба", "phone": "+996XXXXXXXXX" }

},

{

"address": "Адрес клиента из сделки",

"type": "destination",

"contact": { "name": "Имя клиента", "phone": "Телефон клиента" }

}

\],

"transport_type": "van",

"cargo_type": "van_xcargo",

"declared_value": 12350.00,

"cargo_options": { "size_check": true }

}

**Ключевые правила формирования payload:**

- declared_value = итоговая сумма сделки CRM, автоматически (DEC-43)

- transport_type / кузов: если is_oversize = true хотя бы у одного товара в заказе → принудительно XL (van_xcargo), иначе расчётный (DEC-28, AC 22.2)

- Телефон клиента в destination.contact.phone — Яндекс автоматически отправляет клиенту SMS со ссылкой на трекинг (CON-14). Без IT-интеграции с нашей стороны

- Время подачи заявки: delivery_claim_lead_time_hours (дефолт 2ч) до начала слота (DEC-40)

- callback_url: содержит ?token= — secret для аутентификации входящих вебхуков Яндекса

**Ожидаемый ответ (Яндекс → CRM):**

- Получить claim_id (ID заявки в системе Яндекса)

- Сохранить в карточке сделки: UF_YANDEX_CLAIM_ID = claim_id \[COM-LM-03\]

- Статус сделки → «Ожидает водителя»

**Перенос и отмена слота (AC 22.5, 22.6):**

- Водитель ещё не назначен → CRM вызывает claims/edit для изменения параметров

- Водитель уже назначен → CRM вызывает claims/cancel + новый claims/create (DEC-42)

**Страховка (AC 22.7):** Яндекс.GO страховка недоступна в Бишкеке (CON-13). declared_value передаётся для фискальных целей.

### IC-LM-02: yandex_webhook_received — Приём вебхука от Яндекса (Эпик 22, 25)

| **Параметр**  | **Значение**                                             |
|---------------|----------------------------------------------------------|
| **Событие**   | Изменение статуса заявки на стороне Яндекс               |
| **Инициатор** | \[YA\] Яндекс B2B API                                    |
| **Откуда**    | Яндекс B2B API                                           |
| **Куда**      | \[CRM\] POST /api/yandex-delivery/webhook?token=XXXXXXXX |

**Входящий payload от Яндекса:**

{

"claim_id": "yandex_claim_id_12345",

"status": "driver_assigned",

"updated_ts": "2024-01-15T14:30:00Z"

}

**Маршрутизация по status:**

| **Яндекс status** | **Действие CRM**                                                                                                                     |
|-------------------|--------------------------------------------------------------------------------------------------------------------------------------|
| driver_assigned   | Найти сделку по UF_YANDEX_CLAIM_ID. Вызвать claims/accept (IC-LM-03). Статус сделки → «Передан курьеру». ЛК Phase 2: обновить статус |
| delivering        | Опционально: обновить статус ЛК (Phase 2). Никаких вызовов 1С                                                                        |
| delivered         | Вызвать /close_order (IC-LM-06). Запустить WA-накладную (IC-LM-07)                                                                   |
| cancelled         | Алерт менеджеру. Задача: «Перезаказать доставку или связаться с клиентом»                                                            |
| driver_not_found  | Алерт менеджеру: «Яндекс не нашёл водителя»                                                                                          |

**Auth:** Проверить token в query string. Несовпадение → HTTP 401, запрос игнорировать.  
**Идемпотентность:** ключ claim_id + status — повторный вебхук → HTTP 200 без действий.  
**⚠️ OQ-LM-01:** IP-allowlist Яндекса уточнить при подключении.

### IC-LM-04: check_printed_routing — Маршрутизация check_printed (Эпики 24, 25) \[обновление IC-CK-05\]

| **Параметр**  | **Значение**                                   |
|---------------|------------------------------------------------|
| **Событие**   | 1С проводит Чек ККМ для e-com заказа           |
| **Инициатор** | \[1С\] подписка на ПослеПроведения «Чека ККМ»  |
| **Откуда**    | 1С:УТ                                          |
| **Куда**      | \[CRM\] endpoint CRM_Webhook_Path_CheckPrinted |

**Payload (1С → CRM):** *(без изменений vs IC-CK-05)*

{

"CRM_Deal_ID": "12345",

"event": "check_printed",

"receipt_id": "КЧКМ-000456",

"timestamp": "2024-01-15T14:30:00"

}

**⚠️ Prerequisite \[FIX-1C-01\]:** При пробитии Чека ККМ кассиром на основании Заказа покупателя — 1С **автоматически заполняет** УФ_Purchase_Order_GUID = ЗаказОснование.УникальныйИдентификатор(). Без этого идемпотентность /close_order ломается.

**Маршрутизация в CRM \[COM-LM-01\]:**

При получении event = "check_printed":

Найти сделку по CRM_Deal_ID

Если тип_получения = «Самовывоз» И тип_клиента = «Физлицо»:

→ обработать как order_completed:

CRM вызывает /close_order(purchase_order_guid) → IC-LM-05

1С: idempotent: true (Чек уже есть) → возвращает closing_doc_guid

CRM сохраняет closing_doc_guid. Сделка → «Успешно реализовано»

Триггер: WA-накладная (IC-LM-07)

Если тип_получения = «Доставка»:

→ штатная логика check_printed (сброс таймера 72ч, статус чека)

→ order_completed придёт отдельно от Яндекс delivered (IC-LM-02)

Если тип_клиента = «Юрлицо» + Самовывоз:

→ только логировать receipt_id в карточке

→ order_completed придёт через 1С-LM-02 webhook (IC-LM-05a)

**Идемпотентность:** b_dom_processed_1c_events, ключ CRM_Deal_ID + event. Повторный вебхук → HTTP 200 без действий.

| **Параметр**  | **Значение**                                                                                                                                                                                                          |
|---------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Триггеры**  | \(A\) Физлицо самовывоз: после check_printed → order_completed \[IC-LM-04\]. (B) Юрлицо самовывоз: CRM вызывает сразу при переводе в «Ожидает выдачи». (C) Доставка (физ/юр): после delivered от Яндекса \[IC-LM-02\] |
| **Инициатор** | \[CRM\] CRM-робот                                                                                                                                                                                                     |
| **Откуда**    | Битрикс24 CRM                                                                                                                                                                                                         |
| **Куда**      | \[1С\] POST /close_order                                                                                                                                                                                              |

**Payload (CRM → 1С):**

{

"purchase_order_guid": "a1b2c3d4-e29b-41d4-a716-446655440000"

}

> ⚠️ Применить Патч v1.1: поле purchase_order_id → purchase_order_guid \[FIX-1C-06\].

**Ответы 1С → CRM:**

| **HTTP** | **status**            | **Поля**                                               | **Действие CRM**                                                            |
|----------|-----------------------|--------------------------------------------------------|-----------------------------------------------------------------------------|
| 200      | ok, idempotent: false | closing_doc_guid, closing_doc_number, closing_doc_type | Сохранить все три поля в сделке (DEC-LM-01). Сделка → «Успешно реализовано» |
| 200      | ok, idempotent: true  | closing_doc_guid, closing_doc_number, closing_doc_type | Сохранить поля (если ещё не сохранены). Без повторных действий              |
| 404      | error                 | message: "order not found"                             | Алерт менеджеру                                                             |
| 409      | error                 | reason: "order_cancelled"                              | Алерт менеджеру. Заказ отменён — закрытие невозможно \[FIX-1C-02\]          |
| 500      | error                 | message                                                | Алерт менеджеру + ретрай через 5 мин                                        |

**1С определяет тип клиента самостоятельно** из реквизита контрагента ЮрФизЛицо (DEC-LM-02). CRM тип не передаёт.

**Fast Track после:** 1С пишет затронутые \[GUID, Store_ID\] в Очередь_Fast_Track. Витрина обновляется ≤5 мин (AC 24.4, 25.5).

### IC-RT-04: online_refund_manual — Безналичный возврат (Эпик 28.1)

| **Параметр**  | **Значение**                                                       |
|---------------|--------------------------------------------------------------------|
| **Статус**    | ⚠️ MVP: ручной возврат через кабинет эквайринга (DEC-53, OQ-CK-01) |
| **Инициатор** | \[CRM\] кнопка «Вернуть сумму \[X сом\]»                           |

**CRM action (MVP):**

1.  Рассчитать refund_amount = Σ(qty × Price_Ecom) по возвращаемым позициям

2.  Показать менеджеру инструкцию: «Выполнить возврат через кабинет эквайринга. Сумма: \[refund_amount\] сом»

3.  WA клиенту (HSM-шаблон return_online_done, DEC-45): «Возврат \[X\] сом оформлен. Деньги поступят в течение 3–10 дней» (AC 28.1.2)

4.  Смарт-процесс «Возвраты» → «Успешно закрыт». Триггер: NPS (IC-RT-05)

### IC-RT-05: return_nps — NPS после возврата (Эпик 29)

| **Параметр**  | **Значение**                                                   |
|---------------|----------------------------------------------------------------|
| **Триггер**   | Смарт-процесс «Возвраты» → «Успешно закрыт» + 4 часа (AC 29.1) |
| **Инициатор** | \[CRM\] CRM-робот                                              |
| **Куда**      | WA Business API → клиент                                       |

**WA-сообщение (HSM-шаблон return_nps_request, DEC-51):  
**«Как всё прошло? Оцените сервис возврата от 1 до 5»

**Обработка ответа клиента (AC 29.2):**

- Обработчик матчит входящий текст «1»–«5» по телефону + открытый смарт-процесс «Возвраты». Окно матчинга: 48 ч

- Оценка фиксируется в карточке смарт-процесса

- NPS = 5 → благодарственное WA. Промокод «Добор материала» — post-MVP (DEC-52)

### IC-CL-01: claim_created — Инициация рекламации в CRM (Эпик 30)

**Внутренний AC \[CRM\] — без интеграционного вызова:**

| **AC** | **Условие**                                    | **Действие**                                                                                          |
|--------|------------------------------------------------|-------------------------------------------------------------------------------------------------------|
| 30.1   | Жалоба на брак                                 | Менеджер создаёт тикет в воронке «Рекламации». Счётчик SLA: 2ч на принятие решения по фото            |
| 30.2   | SLA нарушен (2ч без смены статуса)             | Алерт руководителю: «Рекламация \[ID\] без ответа более 2 ч»                                          |
| 30.3   | Попытка создать «Рекламацию» внутри «Возврата» | Блокировать — запрещено (CON-18). Рекламация — отдельный смарт-процесс с отдельной привязкой к Сделке |

**Медиа (фото/видео):** хранится в карточке смарт-процесса «Рекламации» (CON-16).

### IC-CL-02: defect_isolation — Изоляция брака в 1С (Эпик 31)

> **Нет нового API-вызова из CRM.** Два документа создаются вручную кладовщиком в 1С в одной сессии (SOP, DEC-54).

**SOP-последовательность (строго соблюдать):**

Шаг 1: «Возврат товаров от клиента»

Store_From_ID = \_RETURN_IN

Store_To_ID = Склад_источника (UF_SOURCE_STORE_ID заказа)

Результат: Stock_Physical++ на складе

Шаг 2: «Списание товаров» (немедленно, в той же сессии)

Store_From_ID = Склад_источника

Store_To_ID = \_WRITEOFF

Writeoff_Reason = обязательный enum:

delivery_damage \| warehouse_damage \| manufacturer_defect

Результат: Stock_Physical-- со склада, \_WRITEOFF фиксирует выбытие

**1С-CL-01 обеспечивает:** без выбора Writeoff_Reason из справочника — документ «Списание» не проводится (DEC-56). Свободный текст запрещён.

**SOP-плашка в форме «Списания» (DEC-54, DEC-55):  
**«⚠️ При рекламации: сначала проведите «Возврат товаров от клиента», затем немедленно проведите это Списание в той же сессии.»

**Гарантийный обмен (AC 31.2.1, DEC-55):**

- Новая «Реализация» **не создаётся**

- Только «Возврат от покупателя» + «Списание»

- Новый товар передаётся физически на основании управленческого решения менеджера

- Убыток отражается в BI-блоке «Убытки от брака», не искажает ABC и GMROI

**Fast Track после «Списания»:** затронутые \[GUID, Store_ID\] → Очередь_Fast_Track. Витрина обновляется ≤5 мин.

### IC-CL-03: reverse_logistics_claims — Обратная логистика рекламаций (Эпик 32)

> **Нет IT-интеграции.** API Яндекс.GO не используется (CON-17).

**Процесс (полностью ручной):**

- Логист вызывает Яндекс.Грузовой через корпоративный кабинет

- Маршрут: «База → Клиент (выгрузка нового) → База (приём брака)»

- Оплата с корп. счёта. Сумму вносит логист вручную в карточку рекламации

- После подтверждения в кабинете Яндекс.GO → менеджер вручную переводит смарт-процесс «Рекламации» в «Успешно закрыт»

- Вина производитель/клиент → обратная логистика на стороне клиента (AC 32.3)

- Косты списываются в статью «Убытки/Рекламации» в P&L

### IC-CL-04: defect_analytics_bi — Аналитика брака (Эпик 33)

> **Нет вызовов из \[CRM\] или \[CMS\].** Полностью зона \[BI\] + \[1С\].

**Источник данных:** Movements.csv (Slow Track T-1), строки Store_To_ID = \_WRITEOFF с типизированным Writeoff_Reason.

**BI строит два независимых среза (DEC-58):**

- По Attr_Brand (бренд производителя)

- По Supplier_Name (поставщик-контрагент)

**AC 33.1.2 (legacy):** Строки с нетипизированной причиной (до внедрения 1С-CL-01) → отдельная строка «Причина не указана» + WARN. Основной отчёт не ломается.

**Is_Toxic_Offline write-back (AC 33.2, DEC-57):**

| **Шаг** | **Действие**                                                                           | **Статус**                                |
|---------|----------------------------------------------------------------------------------------|-------------------------------------------|
| 1       | BI анализирует \_WRITEOFF строки по GUID                                               | ✅                                        |
| 2       | Если % брака \> toxic_offline_threshold → Is_Toxic_Offline = 1                         | ⚠️ OQ-CL-02: порог TBD                    |
| 3       | BI записывает обратно в 1С через write-back эндпоинт                                   | ⚠️ Задача List.kg в очереди \[COM-DS-04\] |
| 4       | Следующий Slow Track → Is_Toxic_Offline = 1 в Catalog.csv → Битрикс скрывает с витрины | ✅                                        |

**Пока toxic_offline_threshold не установлен (CON-19):** автоматическая маркировка не срабатывает. Аналитика строится, маркировка — ручное решение.
