## Архитектурные ограничения, актуальные для этого блока

- purchase_order_guid = GUID документа «Заказ покупателя» (до отгрузки)

- closing_doc_guid = GUID закрывающего документа (после отгрузки): «Чек ККМ» для физлица, «Реализация товаров» для юрлица

- CRM_Deal_ID = внутренний ID сделки Битрикс24

- Тип клиента определяется 1С из реквизита контрагента «Заказа покупателя» — не передаётся в payload

- Все HTTP-эндпоинты **идемпотентны по умолчанию**

- После любого изменения остатков — затронутые пары \[GUID, Store_ID\] пишутся в Очередь_Fast_Track (5-минутный батч)

- Исходящие вебхуки: **3 попытки × 30 сек**, при провале → DLQ + WARN + кнопка «Переотправить» \[COMMIT-24\]

- ✅ **\[FIX-1C-08 / CON-12 закрыт\]** Факт доставки подтверждается вебхуком Яндекса delivered. Фото-пруф остаётся в чате Яндекс.GO — вне систем компании, хранения в 1С или CRM не требует

> **Разделение смарт-процессов:** «Возвраты» и «Рекламации» — **два отдельных** смарт-процесса Битрикс24. Объединение в один процесс — контрактный провал. Разделение отражается в именовании: return_id принадлежит смарт-процессу «Возвраты», claim_id (если появится) — смарт-процессу «Рекламации».

## 1С-LM-01 — HTTP-сервис /close_order

**Инициатива / Эпик:** Последняя миля / Закрытие заказа

**User Story:** Как склад, я хочу выдать товар и автоматически закрыть все хвосты по документам — создать закрывающий документ, снять резерв, списать остаток — без ручных операций в 1С.

**Описание задачи:** Создать HTTP-эндпоинт /close_order. Принимает GUID «Заказа покупателя», определяет тип клиента из реквизита контрагента, создаёт соответствующий закрывающий документ (физлицо → «Чек ККМ», юрлицо → «Реализация товаров»), снимает резерв, списывает остаток. Возвращает closing_doc_guid и closing_doc_number — они потребуются CRM для последующих вызовов /create_return. Идемпотентен по purchase_order_guid. Вызывается двумя триггерами:

| **Тип получения** | **Триггер**                                                          |
|-------------------|----------------------------------------------------------------------|
| Самовывоз         | CRM-роботом через этот эндпоинт после проведения «Чека ККМ» кассиром |
| Доставка          | CRM-роботом (CRM-LM-07) при получении вебхука delivered от Яндекса   |

#### Техническая реализация в 1С

**Эндпоинт:**

| **Параметр**   | **Значение**     |
|----------------|------------------|
| URL            | /close_order     |
| Метод          | POST             |
| Аутентификация | X-API-Key        |
| Content-Type   | application/json |

**✅ \[v1.1\] Входной payload:**

{

"purchase_order_guid": "a1b2c3d4-e29b-41d4-a716-446655440000"

}

> ✅ **Патч v1.1:** Поле переименовано с purchase_order_id → **purchase_order_guid**. purchase_order_guid — GUID документа «Заказ покупателя» в 1С. Не номер документа, не CRM_Deal_ID.

**Валидация:**

| **Поле**            | **Проверка**                               | **Ошибка**                                                              |
|---------------------|--------------------------------------------|-------------------------------------------------------------------------|
| purchase_order_guid | Не пустое, валидный UUID4                  | HTTP 400, {"status": "error", "reason": "purchase_order_guid_required"} |
| X-API-Key           | Верный ключ                                | HTTP 401                                                                |
| Документ найден     | «Заказ покупателя» с таким GUID существует | HTTP 404, {"status": "error", "reason": "order_not_found"}              |

**Идемпотентность:**

До создания закрывающего документа — проверить: существует ли уже документ («Чек ККМ» **или** «Реализация товаров») с реквизитом УФ_Purchase_Order_GUID = purchase_order_guid. Поиск вести в **обоих** типах документов.

Если найден → вернуть HTTP 200 без создания нового документа:

{

"status": "ok",

"closing_doc_guid": "b2c3d4e5-...",

"closing_doc_number": "РТ-000456",

"closing_doc_type": "realization",

"idempotent": true

}

**Определение типа клиента (из «Заказа покупателя»):**

ЗаказОбъект = НайтиЗаказПоGUID(purchase_order_guid);

ТипКлиента = ЗаказОбъект.Контрагент.ЮрФизЛицо;

// Значения: "ФизическоеЛицо" / "ЮридическоеЛицо"

**Алгоритм (пошагово, в транзакции):**

НачатьТранзакцию()

Попытка

// 2. Проверить статус

// ✅ \[FIX-1C-02\] Отменённый заказ → HTTP 409, не HTTP 500

Если ЗаказОбъект.Статус = "Отменён" Тогда

ОтменитьТранзакцию();

Возврат HTTP_409({

"status": "error",

"reason": "order_cancelled",

КонецЕсли;

Иначе

> **Важно для CRM (подрядчик):** closing_doc_guid и closing_doc_type необходимо сохранить в полях сделки. closing_doc_guid используется как значение поля closing_doc_guid при последующем вызове /create_return если клиент инициирует возврат.

**Логирование:**

- Каждый вызов: **Информация**, "close_order \| purchase_order_guid={x} \| type={check\|realization} \| result={status}"

- При идемпотентном вызове: "close_order ДУБЛЬ \| purchase_order_guid={x} \| closing_doc_guid={y}"

- При неудачном НайтиПоКоду(): **Ошибка**, "close_order: склад не найден \| Store_ID={x} \| purchase_order_guid={y}"

#### Критерии приёмки (DoD)

- Физлицо: создан и проведён «Чек ККМ», Stock_Physical уменьшился, резерв снят

- Юрлицо: создана и проведена «Реализация товаров», Stock_Physical уменьшился, резерв снят

- ✅ **\[v1.1\]** Входной payload принимает purchase_order_guid (не purchase_order_id)

- ✅ **\[v1.1 / FIX-1C-05\]** Юрлицо: склад устанавливается через НайтиПоКоду(). Прямое присвоение строки УФ_SOURCE_STORE_ID к полю Склад — провал приёмки

- Ответ содержит closing_doc_guid, closing_doc_number, closing_doc_type

- Повторный вызов с тем же purchase_order_guid → HTTP 200, idempotent: true, новый документ не создаётся (поиск ведётся по обоим типам документов)

- ✅ **\[FIX-1C-02\]** Вызов для отменённого заказа → **HTTP 409**, reason: "order_cancelled" (не HTTP 500)

- Fast Track обновлён по всем позициям

#### Ошибки и исключения

| **Ситуация**                                          | **HTTP** | **Ответ**                                  |
|-------------------------------------------------------|----------|--------------------------------------------|
| purchase_order_guid не найден                         | 404      | reason: "order_not_found"                  |
| ✅ **\[FIX-1C-02\]** Заказ отменён                    | **409**  | reason: "order_cancelled"                  |
| ✅ **\[FIX-1C-05\]** Склад не найден по НайтиПоКоду() | 500      | reason: "posting_failed", текст с Store_ID |
| Закрывающий документ уже существует                   | 200      | idempotent: true                           |
| Ошибка проведения 1С                                  | 500      | reason: "posting_failed", текст ошибки     |
| Неверный API-ключ                                     | 401      | —                                          |

> **Разница FIX-1C-02 vs HTTP 500:** HTTP 409 («Конфликт») семантически корректен — клиент прислал валидный запрос, но сервер не может его выполнить из-за текущего состояния ресурса. HTTP 500 зарезервирован строго для непредвиденных ошибок 1С. CRM должна обрабатывать 409 отдельно от 500 (задача подрядчика по Битриксу, CRM-LM-08).

#### Зависимости

- УФ_Purchase_Order_GUID добавить в документы «Чек ККМ» **и** «Реализация товаров» — задача 1С-ИНФ-01 (консолидируется в конце этого блока)

- УФ_SOURCE_STORE_ID в «Заказе покупателя» — задача 1С-ИНФ-01 (блок Discovery & Selection)

- УФ_CRM_Deal_ID в «Заказе покупателя» и «Чеке ККМ» — задача 1С-ИНФ-01 (блоки Discovery & Selection и Checkout)

- CRM-LM-07 (подрядчик CRM) — вызывает /close_order для доставок по вебхуку delivered от Яндекса

- CRM-LM-08 (подрядчик CRM) — обработка HTTP 409

## 1С-LM-02 — Исходящий вебхук при самовывозе юрлиц

**Инициатива / Эпик:** Последняя миля / Закрытие заказа

**User Story:** Как менеджер CRM, я хочу мгновенно видеть что юрлицо получило товар самовывозом — через вебхук при проведении «Реализации» в 1С — чтобы автоматически переводить сделку в «Успешно реализовано».

**Описание задачи:** При проведении «Реализации товаров» для e-com заказов юрлиц с типом получения «Самовывоз» — отправить вебхук в Битрикс24 с событием order_completed.

> **Область действия (важно):** Физлица самовывоз — **не** покрываются этой задачей. Для физлиц /close_order создаёт «Чек ККМ», и событие check_printed (задача 1С-CK-03, блок Checkout) обрабатывается CRM как order_completed для физлиц самовывоза (задача CRM-LM-XX, подрядчик). Разделение намеренное — не объединять логику.

#### Техническая реализация в 1С

**Триггер:** Подписка на событие ПослеПроведения документа «Реализация товаров и услуг».

**Фильтр — только e-com самовывоз юрлиц (четыре условия, все обязательны):**

Процедура РеализацияТоваров_ПослеПроведения(Источник, Отказ)

// Условие 1: Реализация создана через /close_order

// (есть реквизит УФ_Purchase_Order_GUID, не пустой)

Если ПустаяСтрока(Источник.УФ_Purchase_Order_GUID) Тогда

Возврат; // не e-com реализация — игнорировать

КонецЕсли;

// Условие 2: Тип получения = Самовывоз

// (доставка закрывается через CRM-LM-07 + вебхук Яндекса, не здесь)

ЗаказОснование = НайтиЗаказПоGUID(Источник.УФ_Purchase_Order_GUID);

Если ЗаказОснование.УФ_Тип_Получения \<\> "Самовывоз" Тогда

Возврат;

КонецЕсли;

// Условие 3: Юрлицо

// (физлицо самовывоз покрывается через check_printed → CRM-LM-XX)

Если ЗаказОснование.Контрагент.ЮрФизЛицо \<\> "ЮридическоеЛицо" Тогда

Возврат;

КонецЕсли;

// Условие 4: Есть CRM_Deal_ID

CRM_Deal_ID = ЗаказОснование.УФ_CRM_Deal_ID;

Если ПустаяСтрока(CRM_Deal_ID) Тогда

Возврат;

КонецЕсли;

Purchase_Order_GUID = Источник.УФ_Purchase_Order_GUID;

ОтправитьВебхукOrderCompleted(CRM_Deal_ID, Purchase_Order_GUID);

КонецПроцедуры

> **Примечание:** Имя реквизита УФ_Тип_Получения в «Заказе покупателя» — уточнить у подрядчика CRM (Александр). Если реквизит называется иначе — скорректировать и зафиксировать в 1С-ИНФ-01.

**Payload вебхука:**

{

"CRM_Deal_ID": "12345",

"purchase_order_guid": "a1b2c3d4-e29b-41d4-a716-446655440000",

"event": "order_completed",

"timestamp": "2024-01-15T14:30:00"

}

| **Параметр** | **Значение**                             |
|--------------|------------------------------------------|
| URL          | Константа CRM_Webhook_URL_OrderCompleted |
| Заголовок    | X-API-Key: {константа CRM_ApiKey}        |

**Retry и DLQ \[COMMIT-24\]:**

3 попытки × 30 сек. После третьей неудачи:

- Записать в регистр Очередь_Ошибок: {purchase_order_guid, CRM_Deal_ID, event: "order_completed", timestamp, attempts: 3, last_error}

- Записать **WARN** в журнал: "LM-02 DLQ \| CRM_Deal_ID={x} \| purchase_order_guid={y} \| error={z}"

**Кнопка «Переотправить» (обработка для администратора):** Показывает записи из Очередь_Ошибок с event = "order_completed". При нажатии — повторная отправка. При успехе — удалить из DLQ.

> **Критически важно:** Ошибка отправки вебхука **не отменяет** проведение «Реализации». Кассовая и складская операция выполнена — только коммуникация с CRM отложена.

#### Критерии приёмки (DoD)

- При проведении e-com «Реализации» (юрлицо, самовывоз) — Битрикс24 получает вебхук order_completed

- При проведении «Реализации» для **физлица** — вебхук **НЕ** отправляется

- При проведении «Реализации» с типом получения **«Доставка»** — вебхук **НЕ** отправляется

- При проведении розничной «Реализации» (без УФ_Purchase_Order_GUID) — вебхук **НЕ** отправляется

- После 3 неудачных попыток — запись в Очередь_Ошибок, WARN в журнале

- Кнопка «Переотправить» работает корректно

- Ошибка отправки не влияет на проведение документа

#### Ошибки и исключения

| **Ситуация**           | **Поведение**                                                         |
|------------------------|-----------------------------------------------------------------------|
| Битрикс недоступен     | Retry × 3, затем DLQ + WARN                                           |
| HTTP ≠ 200 от Битрикса | Retry × 3, затем DLQ + WARN                                           |
| Физлицо самовывоз      | Вебхук не отправляется — покрыто CRM-LM-XX через check_printed        |
| Доставка               | Вебхук не отправляется — покрыто CRM-LM-07 + вебхук Яндекса delivered |

#### Зависимости

- УФ_Purchase_Order_GUID в «Реализации товаров» — задача 1С-ИНФ-01 (конец этого блока)

- УФ_Тип_Получения в «Заказе покупателя» — уточнить имя у подрядчика CRM

- URL и API-ключ CRM — предоставляет подрядчик CRM

- CRM-LM-06 (подрядчик CRM) — принимающий эндпоинт order_completed

- CRM-LM-XX (подрядчик CRM) — обработка физлиц через check_printed

## БЛОК: ВОЗВРАТЫ (Returns & Buyback)

## 1С-RT-01 — HTTP-сервис /create_return

**Инициатива / Эпик:** Возвраты / Складская приёмка и учёт

**User Story:** Как менеджер склада, я хочу чтобы при физической приёмке возвращённого товара 1С автоматически восстанавливала остатки и создавала «Возврат от покупателя» — без ручного ввода документов.

- Тест-кейс: провести «Списание» с причиной warehouse_damage → в Movements.csv за T-1 строка с Reason = "warehouse_damage"

- При legacy-данных — строка выгружается с WARN в журнале, без ошибки

#### Зависимости

- **1С-CL-01** должна быть реализована и принята до тестирования

- Data Engineer (BI-CL-01, BI-CL-02) — потребители этого поля
