## IC-PR-00: 1С-РТ-03 — Вебхук смены статуса заказа \[перенесён из D&S\]

| **Параметр**  | **Значение**                                              |
|---------------|-----------------------------------------------------------|
| **Событие**   | Изменение статуса «Заказа покупателя» в 1С                |
| **Инициатор** | \[1С\] подписка на ПриЗаписи документа «Заказ покупателя» |
| **Откуда**    | 1С:УТ                                                     |
| **Куда**      | \[CRM\] Битрикс24 — endpoint CRM_Webhook_URL_OrderStatus  |

**Триггеры отправки (только e-com заказы с заполненным УФ_CRM_Deal_ID):**

| **Новый статус 1С**       | **event в payload**                            |
|---------------------------|------------------------------------------------|
| В работе / Ожидает сборки | order_confirmed                                |
| Отменён                   | order_cancelled                                |
| Нехватка при сборке       | oos_at_picking (отдельный endpoint → IC-PR-03) |

**Payload (1С → CRM):**

{

"CRM_Deal_ID": "12345",

"event": "order_confirmed",

"purchase_order_guid": "a1b2c3d4-...",

"purchase_order_number": "ЗКЧ-000123",

"timestamp": "2024-01-15T14:30:00"

}

**Ожидаемый ответ:** HTTP 200 от CRM. При ошибке → WARN в журнал 1С, документ не прерывается. Идемпотентность на стороне CRM: ключ CRM_Deal_ID + event в b_dom_processed_1c_events.

## IC-PR-01: create_order_reserve — Hard Booking в 1С (Эпик 16)

> Дополняет IC-CK-03 из Эпиков 8–15. Здесь зафиксированы гарантии 1С-стороны (1С-PR-01, 1С-PR-02).

| **Параметр**  | **Значение**                                                                     |
|---------------|----------------------------------------------------------------------------------|
| **Событие**   | CRM вызывает /create_order для постановки резерва                                |
| **Инициатор** | \[CRM\] — после payment confirmed (delivery) ИЛИ менеджер «Подтверждён» (pickup) |
| **Откуда**    | Битрикс24 CRM                                                                    |
| **Куда**      | \[1С\] POST /create_order                                                        |

**Гарантии 1С (1С-PR-01, 1С-PR-02) — обязательны к проверке при приёмке:**

| **Гарантия**                 | **Требование**                                                                                                                   |
|------------------------------|----------------------------------------------------------------------------------------------------------------------------------|
| HTTP 200 = факт проведения   | 200 возвращается строго **после** Записать(Проведение) + контрольный запрос к регистру ЗаказыПокупателей (резерв ≠ 0) \[DEC-33\] |
| Нет «глотания» исключений    | Запрещён паттерн Попытка/КонецПопытки без блока Исключение                                                                       |
| Store_ID валидируется первым | До создания документа: Store_ID не пустой + найден в справочнике + не помечен на удаление                                        |

**Форматы ответов:**

| **HTTP** | **status** | **Поля**                                        | **Действие CRM**                                                         |
|----------|------------|-------------------------------------------------|--------------------------------------------------------------------------|
| 200      | success    | purchase_order_guid, purchase_order_number      | Сохранить purchase_order_guid в сделке. WA клиенту: «Заказ подтверждён»  |
| 200      | partial    | missing: \[{GUID, qty_ordered, qty_available}\] | Сделка → «Требует уточнения (Нехватка)». Задача менеджеру. \[COM-PR-04\] |
| 200      | rejected   | reason: "insufficient_stock"                    | Сделка → «Отменена (OOS)». WA клиенту (EDGE-03)                          |
| 200      | duplicate  | purchase_order_guid, idempotent: true           | Лог, без повторных действий \[DEC-15\]                                   |
| 400      | error      | reason: "invalid_store_id", details             | CRM логирует, уведомляет менеджера                                       |
| 500      | error      | reason: "posting_failed", details               | CRM логирует, уведомляет менеджера                                       |

## IC-PR-02: oos_escalation — Эскалация нехватки (Эпик 17)

**Внутренние AC \[CRM\] — без новых интеграционных вызовов:**

| **AC** | **Триггер**                                  | **Действие CRM**                                                                                                                                                                                              |
|--------|----------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 17.1   | /create_order → partial                      | Сделка → «Требует уточнения». Задача менеджеру: «Срочно связаться, предложить замену». missing\[\] отображается в задаче                                                                                      |
| 17.2   | /create_order → rejected                     | Сделка → «Отменена (OOS)». WA клиенту (EDGE-03). Для card-pay: полный refund через кабинет эквайринга                                                                                                         |
| 17.3   | partial + card-pay заказ                     | В задачу менеджеру: «При согласии — partial_refund на Σ(missing.qty_ordered × Price). При отказе — полный refund». Сумма рассчитывается CRM автоматически: Σ(missing\[i\].qty_ordered × item_price) (EDGE-15) |
| 17.4   | Сделка в «Требует уточнения» без движения 2ч | Эскалация задачи на руководителя + WA клиенту: «Мы работаем над заказом, свяжемся в течение N часов»                                                                                                          |

## IC-PR-03: oos_at_picking_webhook — OOS при физической сборке (Эпик 19)

| **Параметр**  | **Значение**                                                                      |
|---------------|-----------------------------------------------------------------------------------|
| **Событие**   | Кладовщик фиксирует нехватку при сборке                                           |
| **Инициатор** | \[1С\] подписка на ПриЗаписи «Заказа покупателя» (статус → «Нехватка_При_Сборке») |
| **Откуда**    | 1С:УТ                                                                             |
| **Куда**      | \[CRM\] Битрикс24 — endpoint CRM_Webhook_URL_OOS (отдельный от OrderStatus)       |

**Условие отправки:** УФ_CRM_Deal_ID не пустой + статус изменился на «Нехватка_При_Сборке» впервые (повторное сохранение без смены статуса → вебхук не уходит).

**Payload (1С → CRM):**

{

"CRM_Deal_ID": "12345",

"status": "oos_at_picking",

"missing": \[

{

"GUID": "550e8400-...",

"qty_ordered": 5,

"qty_available": 2

}

\],

"timestamp": "2024-01-15T14:30:00"

}

> qty_available вычисляется 1С: qty_ordered - Кол_во_Нехватки из УФ_OOS_Позиции. \[COM-PR-04\]

**Ожидаемый ответ (CRM action):**

1.  Идемпотентность: ключ CRM_Deal_ID + "oos_at_picking" в b_dom_processed_1c_events

2.  Сделка → «Уточнение состава»

3.  Резерв в 1С **НЕ снимается** — сохраняется до решения менеджера \[DEC-PR-04\]

4.  Задача менеджеру с составом missing\[\] и кнопкой «Изменения согласованы»

5.  Вернуть HTTP 200

**При ошибке/недоступности CRM:** WARN в журнал 1С. Документ сохраняется, регистр Расхождения_При_Сборке пишется в любом случае.

## IC-PR-04: mutation_cycle — Цикл мутации заказа (Эпик 19)

### Шаг 1: Pre-check \[CRM internal, COM-PR-02/03\]

| **Параметр**  | **Значение**                                                                         |
|---------------|--------------------------------------------------------------------------------------|
| **Инициатор** | \[CRM\] — при нажатии менеджером «Изменения согласованы»                             |
| **Механизм**  | Вариант A: прямой запрос через нативный Bitrix PHP API к данным CMS (не HTTP, не 1С) |

**⚠️ Критично:** CRM-робот читает Stock_Free строго в разрезе \[GUID + UF_SOURCE_STORE_ID\].  
Total_Stock_Free (агрегат по всем складам) **запрещён** — pre-check теряет смысл. \[COM-PR-03\]

**Алгоритм pre-check:**

Для каждой новой позиции в составе сделки:

Stock_Free = читать из CMS данных Fast Track по \[GUID + UF_SOURCE_STORE_ID\]

Если Stock_Free \< qty_requested:

Добавить в список предупреждений менеджеру

Если есть предупреждения:

Показать менеджеру: «Возможна нехватка по \[N\] позициям. Продолжить?»

Менеджер подтверждает → перейти к Шагу 2

Иначе:

Перейти к Шагу 2 автоматически

**Лаг данных:** ≤5 мин (Fast Track батч). Pre-check — UX-помощь, **не** финальная гарантия.

### Шаг 2: /update_order \[CRM → 1С\]

| **Параметр**  | **Значение**              |
|---------------|---------------------------|
| **Инициатор** | \[CRM\] — после pre-check |
| **Откуда**    | Битрикс24 CRM             |
| **Куда**      | \[1С\] POST /update_order |

**Payload (CRM → 1С):**

{

"CRM_Deal_ID": "12345",

"items": \[

{"GUID": "550e8400-...", "Qty": 5, "Price": 600.00},

{"GUID": "660f9511-...", "Qty": 2, "Price": 1200.00}

\]

}

> Store_ID в payload не передаётся — 1С использует UF_SOURCE_STORE_ID исходного заказа \[DEC-34, CON-09\].

**Ответы и действия CRM:**

| **status**                     | **Действие CRM**                                                                                                                                                                                             |
|--------------------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| success                        | Обновить состав сделки. Пересчитать сумму. Запустить финансовый флоу (IC-PR-05). Сделка → «Ожидает сборки»                                                                                                   |
| partial_oos                    | Обновить состав до applied\[\]. Убрать missing\[\]. Пересчитать сумму. Задача менеджеру: «Согласовать missing\[\] с клиентом». Запустить финансовый флоу по applied-сумме (IC-PR-05). Цикл может повториться |
| success, idempotent: true      | Лог. Без изменений                                                                                                                                                                                           |
| error: items_empty (400)       | Не разрешать пустой состав. Менеджер должен выбрать отмену заказа явно                                                                                                                                       |
| error: order_not_mutable (409) | Уведомить менеджера: заказ уже закрыт/отменён                                                                                                                                                                |
| error: posting_failed (500)    | Лог. Уведомить менеджера. Старый состав сохранён в 1С (транзакция откачена)                                                                                                                                  |

**Пример partial_oos (COM-PR-04):**

{

"status": "partial_oos",

"applied": \[{"GUID": "550e8400-...", "qty_applied": 5}\],

"missing": \[{"GUID": "660f9511-...", "qty_ordered": 2, "qty_available": 0}\]

}

**Fast Track после мутации (DEC-PR-05):** 1С пишет в Очередь_Fast_Track пары \[GUID, Store_ID\] из **обоих** наборов — старый состав (резерв снят) + новый applied\[\] (резерв поставлен). Витрина обновляется в течение ≤5 мин.

## IC-PR-05: financial_correction — Финансовые корректировки после мутации (Эпик 20)

**Применяется:** только для **delivery (card-pay) заказов** \[COM-PR-06\]  
**Не применяется:** самовывоз — кассир пробивает итоговую сумму при выдаче

**Переменные:**

- original_payment_amount — зафиксирован при IC-CK-02 (payment confirmed), **не изменяется** при мутациях \[COM-PR-05\]

- new_order_total — Σ(applied\[i\].qty_applied × item_price) после мутации

**Алгоритм выбора флоу:**

delta = new_order_total - original_payment_amount

delta \< 0 → partial_refund (IC-PR-06)

delta \> 0 → add_on_payment (IC-PR-07)

delta = 0 → финансовых действий нет, заказ продолжает движение

### IC-PR-06: partial_refund — Возврат переплаты (Эпик 20.1)

| **Параметр** | **Значение**                                                                          |
|--------------|---------------------------------------------------------------------------------------|
| **Условие**  | new_order_total \< original_payment_amount                                            |
| **Статус**   | ⚠️ MVP: ручной возврат через кабинет эквайринга. Post-MVP: API Бакай Банка (OQ-CK-01) |

**CRM action (MVP):**

1.  Вычислить refund_amount = original_payment_amount - new_order_total

2.  Показать менеджеру кнопку: **«Инициировать возврат \[refund_amount\] сом»**

3.  При нажатии — инструкция: «Выполнить возврат через кабинет эквайринга. Сумма: \[refund_amount\] сом. Основание: частичная отмена заказа \[CRM_Deal_ID\]»

4.  Менеджер подтверждает выполнение → CRM фиксирует в карточке: refund_amount, refund_confirmed_at, refund_confirmed_by

**Сценарий полного отказа клиента (AC 20.1.2):**

- Клиент отказывается от частичного заказа → полный refund = original_payment_amount

- CRM → \[1С\] /cancel_reserve (IC-CK-04) → резерв снят → сделка «Отменена»

### IC-PR-07: add_on_payment — Доплата при увеличении суммы (Эпик 20.2)

| **Параметр** | **Значение**                                                            |
|--------------|-------------------------------------------------------------------------|
| **Условие**  | new_order_total \> original_payment_amount                              |
| **Статус**   | ⚠️ Реализуемость зависит от документации Бакай Банка (OQ-CK-01, DEC-30) |

**CRM action:**

1.  Вычислить addon_amount = new_order_total - original_payment_amount

2.  Генерировать подчинённый счёт на addon_amount (привязка: родительская сделка → счёт на дельту)

3.  WA клиенту: «По вашему заказу изменился состав. Доплата: \[addon_amount\] сом. \[Ссылка на оплату\]»

4.  Таймаут addon_payment_timeout — если не оплачено:

    - Резерв на исходные позиции сохраняется

    - Задача менеджеру: «Клиент не оплатил доплату. Варианты: 1) Отгрузить исходный состав, 2) Полная отмена»

    - Автодействий без менеджера нет (EDGE-16)

5.  Клиент оплатил → webhook от эквайринга → CRM фиксирует доплату. Единый фискальный чек на финальную сумму — при закрытии заказа (DEC-36)
