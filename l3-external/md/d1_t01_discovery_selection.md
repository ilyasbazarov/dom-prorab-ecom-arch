# Integration Contract — Инициатива: Discovery & Selection

**Версия:** 1.1  
**Статус:** Готово к реализации (остаток: OQ-BC-03a)  
**Потребитель:** Подрядчик по Битриксу (\[CMS\] + \[CRM\])  
**Источники:** ТЗ List.kg v1.1, Data Contract Slow Track v1.1, DataContract_bitrix_catalog v1.1, Fast Track API Contract, USM v2.0 (Эпики 1–7), Глоссарий идентификаторов, Реестр решений

## Зафиксированные коммиты

| **ID**    | **Коммит**                                                                                                                                       |
|-----------|--------------------------------------------------------------------------------------------------------------------------------------------------|
| COM-DS-01 | DoD 1С-СТ-А01 и А06: **5 CSV** вместо 4 CSV (добавлен Picking_Gaps.csv)                                                                          |
| COM-DS-02 | Virtual Node для «Возврат от покупателя»: Store_From_ID = \_RETURN_IN — канон. 1С-СТ-А05 подлежит обновлению                                     |
| COM-DS-03 | Price_Promo логика везде: ветвление строго по null vs != null. Значение 0 = аномалия → логировать, трактовать как null                           |
| COM-DS-04 | Write-back BI → 1С для Is_Toxic_Offline: задача List.kg, в очереди                                                                               |
| COM-DS-05 | 1С-РТ-03 (вебхук смены статуса заказа): перенесён в **IC Processing**                                                                            |
| OQ-BC-01  | Механизм доставки BI → CMS: **FTP-pull**. BI пишет на FTP /bi_catalog/YYYY-MM-DD/, CMS опрашивает по расписанию                                  |
| OQ-BC-02  | Add_On_Only → **ACTIVE=N**. Товар без самостоятельной страницы, виден только в cross-sell/bundle через явный GUID-запрос                         |
| OQ-BC-03  | Attr\_\* — **фиксированный набор колонок** по всем категориям, NULL для неприменимых                                                             |
| OQ-DS-01  | Робот «Лист ожидания»: назначает задачу менеджеру **только при создании** новой Сделки. При дедупликации — только обновляет UF_LAST_REQUEST_DATE |
