## **БЛОК 1. Архитектурные требования к уведомлениям**

### **1.0 Принципы канала и общие определения**

**Приоритет канала:**

| **Приоритет**        | **Канал**                                | **Условие**                                                  |
|----------------------|------------------------------------------|--------------------------------------------------------------|
| 1 (Primary)          | WhatsApp Business API (HSM через Wazzup) | По умолчанию                                                 |
| 2 (Channel Fallback) | SMS                                      | При отсутствии delivered в течение таймаута, либо при failed |

**Два независимых вида Fallback:**

- **Content Fallback** — альтернативная версия текста при отсутствии переменной. Тот же канал.

- **Channel Fallback (SMS)** — отправка через SMS при недоступности WA.

**Критическая переменная** — та, без которой сообщение теряет смысл или вводит в заблуждение. Перечень — в спецификации каждого шаблона.

**Антидублирование:** на одну пару (Deal ID + код шаблона) — не более одной успешной отправки. Повторная запрещена при статусе sent/delivered/read. При возврате сделки на стадию и повторном продвижении — триггер разрешён.

### **1.2 Алгоритм отправки и Content Fallback**

ДЛЯ КАЖДОГО ТРИГГЕРА:

1\. Определить язык → UF_COMM_LANG (см. §1.3)

2\. Получить значения всех переменных

3\. ЕСЛИ все переменные заполнены:

→ Отправить \[Standard\]

4\. ЕСЛИ отсутствует КРИТИЧЕСКАЯ переменная:

→ Если Content Fallback определён → отправить \[Content Fallback\]

→ Если не определён → уведомление НЕ отправляется

→ Создать задачу менеджеру (см. ниже)

→ Записать в лог: тип = Content Fallback, список отсутствующих переменных

5\. ЕСЛИ отсутствует только НЕКРИТИЧЕСКАЯ:

→ Отправить \[Standard\] с заглушкой "—"

→ Warn в лог

**Задача при Content Fallback:**

| **Поле**      | **Значение**                                                    |
|---------------|-----------------------------------------------------------------|
| Заголовок     | Fallback-шаблон \#{код}: отсутствуют данные по заказу {Deal_ID} |
| Описание      | Список отсутствующих переменных с именами полей                 |
| Ответственный | Менеджер сделки                                                 |
| Срок          | +2ч от создания                                                 |
| Привязка      | Текущая сделка (Deal ID)                                        |

Дедупликация задач: если задача с тем же заголовком + Deal ID существует и не закрыта — обновить описание, не создавать новую.

### **1.5 Channel Fallback (SMS) — технические требования**

> **КОНФЛИКТ-1 закрыт** (Checkout TZ COM-PRE-06, Processing TZ AC-18.2): триггером SMS-fallback является отсутствие статуса **delivered** (доставлено на устройство) в течение установленного таймаута. Рекомендация v1 использовать sent отклонена.

**Логика SMS-отправки:**

ЕСЛИ WA-статус = "failed" → SMS немедленно

ЕСЛИ WA-статус "delivered" не получен за \[таймаут\] → SMS

ЕСЛИ WA-статус "delivered" получен до истечения таймаута → SMS не отправляется

**Таймауты:**

- OTP (#0): WA_TIMEOUT_OTP = **45 сек**

- Все остальные: WA_TIMEOUT_TRANSACTIONAL = **60 сек**

**SMS-технические ограничения:**

- Кириллица: 70 символов на 1-ю часть, 67 на каждую последующую

- Эмодзи в SMS — запрещены

- SMS-текст специфицируется отдельной строкой в каждом шаблоне

- Ссылки в SMS — сокращённый URL с UTM

**При SMS failed:** задача менеджеру: "Не удалось доставить уведомление (WA + SMS) по заказу {Deal_ID}. Необходим ручной контакт."

**Логирование в таймлайн сделки:**

| **Поле**                 | **Значение**                                   |
|--------------------------|------------------------------------------------|
| Дата/время               | UTC timestamp                                  |
| Канал                    | WA / SMS                                       |
| Код шаблона              | например order_confirmed_pickup                |
| Версия                   | Standard / Content Fallback / Channel Fallback |
| Статус                   | sent / delivered / read / failed               |
| Отсутствующие переменные | если Content Fallback — список полей           |

### **1.8 Обработка состава заказа (шаблоны \#1, 2, 6, 8)**

CRM-робот формирует переменную {{ORDER_ITEMS}}:

Название товара × Кол-во

Название товара × Кол-во

При превышении **1024 символов** — список обрезается, добавляется: "...и ещё N позиций".

> ⚠️ **Конфликт \#7 (частично открыт):** логика формирования {{ORDER_ITEMS}} и лимит 1024 симв. **не зафиксированы** в Processing TZ / LastMile TZ. Требуется внесение этого требования в Processing TZ до начала разработки backend-части робота. Без фиксации — гарантированная дыра на стыке CRM-робот ↔ backend.
