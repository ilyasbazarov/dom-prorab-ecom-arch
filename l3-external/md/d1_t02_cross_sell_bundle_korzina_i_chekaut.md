### IC-CS-02: \[CMS\] Рендер блока cross-sell (внутренний AC, Эпики 8–9)

| **Параметр**      | **Значение**                                             |
|-------------------|----------------------------------------------------------|
| Триггер           | Загрузка карточки товара или корзины                     |
| Источник GUIDs    | Cross_Sell_GUIDs из инфоблока (загружен через IC-DS-03)  |
| Источник остатков | Stock_Free\[Store_ID\] из CMS-кэша Fast Track (IC-DS-04) |
| Активный склад    | window.activeStoreId (DEC-22)                            |

**Алгоритм рендера:**

1.  Взять Cross_Sell_GUIDs текущего товара из инфоблока

2.  Если Cross_Sell_GUIDs пуст → **fallback** (см. ниже)

3.  Для каждого GUID применить фильтры:

    - Stock_Free\[window.activeStoreId\] \> 0 — есть на активном складе

    - Ecom_Status NOT IN (Toxic_Online, Review_Required) — не скрытые товары

    - Ecom_Status = Add_On_Only — **включать** (специально для cross-sell, ACTIVE=Y)

    - GUID ≠ текущий товар

4.  Если после фильтрации ноль результатов → **скрыть блок** (EDGE-08)

5.  Иначе → рендерить блок «С этим товаром покупают»

**Fallback (AC 9.2):**

- Условие: Cross_Sell_GUIDs пуст ИЛИ Cross_Sell_Source = category_fallback

- Запрос: товары из той же Category_L2, где:

  - Stock_Free\[window.activeStoreId\] \> 0

  - GUID != текущий GUID \[COM-CK-03\]

  - Ecom_Status NOT IN (Toxic_Online, Review_Required)

- Сортировка: Sales_30d DESC \[COM-CK-05\]. При равном Sales_30d — вторичная сортировка по Stock_Free DESC

- Лимит: топ-3. Если доступно \< 3 — выводить сколько есть, блок не скрывать

**Мониторинг (AC 9.3):**

- BI логирует долю товаров с пустым Cross_Sell_GUIDs. При \> 30% → алерт Data Engineer

### IC-CS-03: \[CMS\] Пакетное добавление в корзину — Bundle (внутренний AC, Эпик 10)

| **Параметр**        | **Значение**                                                  |
|---------------------|---------------------------------------------------------------|
| Триггер             | Клик «Добавить всё в корзину»                                 |
| Проверка Stock_Free | CMS-кэш Fast Track — **не синхронный вызов 1С** \[COM-CK-04\] |

**Алгоритм:**

1.  Собрать список: основной товар + все сопутствующие с активным чекбоксом

2.  Для каждого сопутствующего: проверить Stock_Free\[window.activeStoreId\] \> 0 в CMS-кэше

    - Stock_Free = 0 → пропустить + показать уведомление: «\[Название\] закончился на складе» (AC 10.4)

3.  Для каждого прошедшего товара: если GUID уже в корзине → qty += выбранное_qty (не дублировать) (AC 10.2)

4.  Добавить все в корзину одним запросом

5.  Показать попап «Товары добавлены»

**Умолчания (CON-05):**

- Кол-во сопутствующего товара по умолчанию = 1 Unit_Base

- Автоматический расчёт по справочнику достаточности (сколько клея на сколько плитки) — post-MVP

- Шаг инпута = 1 Unit_Base (аналогично основному товару, Эпик 4)

**Проверка Add_On_Only при добавлении в корзину:**

- При попытке добавить товар с Ecom_Status = Add_On_Only **напрямую** (не через bundle) → блокировать + сообщение: «Этот товар доступен только в комплекте с другим товаром» \[COM-CK-02\]

## ═══ СЕКЦИЯ 2: Корзина и Чекаут (Эпики 11–15) ═══

### Внутренние AC без новых интеграционных событий

#### \[CMS\] Эпик 11: Умная корзина — вес и объём

| **Параметр**    | **Значение**                                                                      |
|-----------------|-----------------------------------------------------------------------------------|
| Источник данных | Weight_kg, Volume_m3, Weight_Source из bitrix_catalog.csv (IC-DS-03)              |
| Расчёт          | Фронтенд JS: Total_Weight = Σ(Weight_kg × qty), Total_Volume = Σ(Volume_m3 × qty) |

**Алгоритм отображения:**

- Weight_kg = NULL или Weight_Source отсутствует → вклад = 0 + алерт: «Вес некоторых товаров не учтён» (AC 11.2)

- Weight_Source = category_default → алерт: «Вес рассчитан приблизительно» (AC 11.3)

- Пересчёт при изменении qty — без перезагрузки страницы (AC 11.4)

#### \[CMS\] Эпик 12: OTP-авторизация

| **Параметр**    | **Значение**                                                      |
|-----------------|-------------------------------------------------------------------|
| Первичный канал | WhatsApp Business API (HSM-шаблон, CON-10)                        |
| Fallback        | SMS-провайдер, автоматически при недоставке WA \> 45 сек (DEC-27) |
| OTP-формат      | 4 цифры                                                           |

**AC:**

- Прогрессивная задержка повторных запросов: 60 / 120 / 300 сек → блокировка 10 мин (AC 12.4)

- Rate limit: 5 запросов с номера за 60 мин → блокировка окна (AC 12.5, DEC-27)

- Новый номер → создать теневой профиль LOGIN=PHONE, PASSWORD=RANDOM (AC 12.3)

#### \[CMS\] Эпик 13: Расчёт стоимости доставки

| **Параметр**    | **Значение**                                                                                   |
|-----------------|------------------------------------------------------------------------------------------------|
| Источник данных | Total_Weight, Total_Volume из корзины (Эпик 11); is_oversize из инфоблока (ChangeRegistry \#3) |
| Внешний API     | Яндекс.Карты API (дистанция по адресу)                                                         |
| Тариф           | База + км × ставка по кузову (S/M/L/XL). Детали в отдельном ТЗ модуля доставки                 |

**Зональные правила (CON-08):**

- Бишкек → автоматический расчёт (AC 13.1)

- Чуйская область → «Стоимость уточнит менеджер» (AC 13.3)

- За пределами Чуйской → «Доставка недоступна» (AC 13.4)

- Яндекс API недоступен → поведение как Чуйская (EDGE-12, AC 13.5)

- is_oversize = true хотя бы у одного товара → кузов принудительно XL (DEC-28, AC 13.2)

**Важно (CON-06):** Стоимость доставки не включается в сумму заказа. Не проводится через кассу 1С. Отображается как «Оплата водителю на месте».

#### \[CMS\] Эпик 14: Бронирование слота

| **Параметр**                           | **Значение**                                                           |
|----------------------------------------|------------------------------------------------------------------------|
| Параметр same_day_delivery_cutoff_time | 12:00, хранится в админке Битрикс (DEC-29), изменяется без правки кода |
| Нерабочие дни                          | Инфоблок Битрикс → блокируются в Datepicker (EDGE-13)                  |
| Лимиты вместимости слотов              | Не автоматические на MVP (CON-07)                                      |

**AC:**

- Самовывоз: Datepicker показывает «Сегодня / Завтра / Послезавтра». Слот «Сегодня» блокируется если текущее время близко к открытию или закрытию склада (AC 14.2)

- Доставка: Datepicker + dropdown слотов (10:00–14:00, 14:00–18:00). Слот «Сегодня» блокируется если время \> cutoff (AC 14.4)

- Данные DATE_PICKUP, TIME_SLOT передаются в CRM при создании сделки (IC-CK-01)

|--------------------------------------------------------|----------------------------------------------------------------------------------------|
| **Событие**                                            | Создание резерва в 1С                                                                  |
| **Инициатор**                                          | \[CRM\] — автоматически (после IC-CK-02) ИЛИ вручную (менеджер нажимает «Подтверждён») |
| **Откуда**                                             | Битрикс24 CRM                                                                          |
| **Куда**                                               | \[1С\] POST /create_order                                                              |

**Триггеры:**

- **Delivery:** сразу после IC-CK-02 (payment confirmed) — автоматически

**Payload (CRM → 1С):**

{

"CRM_Deal_ID": "12345",

"Client_Phone": "+996XXXXXXXXX",

"Store_ID": "store_kokzhar_02",

"Items": \[

{

"GUID": "uuid4",

"qty": 11,

"Price_Ecom": 1000.00

}

\]

}

**Ожидаемый ответ (1С → CRM):**

| **HTTP** | **status** | **Действие CRM**                                                                                                                                          |
|----------|------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------|
| 200      | success    | Сохранить purchase_order_guid и purchase_order_numberв карточке сделки. WA клиенту: «Заказ подтверждён» (AC 15.2.3). Сделка → «В работе / Ожидает сборки» |
| 200      | partial    | Менеджер уточняет нехватку с клиентом. EDGE-02                                                                                                            |
| 200      | rejected   | Все товары OOS. Менеджер связывается с клиентом                                                                                                           |
| 200      | duplicate  | Уже создан ранее. CRM логирует, не создаёт повторно                                                                                                       |
| 400/500  | —          | CRM логирует, уведомляет менеджера                                                                                                                        |

**Логирование на стороне 1С:** каждый вызов фиксируется в Лог_Запросов_CreateOrder (1С-CK-01), включая дубли (Is_Duplicate = true при повторном CRM_Deal_ID).

### IC-CK-04: cancel_reserve_call — CRM robot вызывает /cancel_reserve в 1С

| **Параметр**  | **Значение**                                                                         |
|---------------|--------------------------------------------------------------------------------------|
| **Событие**   | Автоотмена резерва по таймауту                                                       |
| **Инициатор** | \[CRM\] Битрикс24 CRM-робот (DEC-32, DEC-CK-03)                                      |
| **Откуда**    | Битрикс24 CRM                                                                        |
| **Куда**      | \[1С\] POST /cancel_reserve                                                          |
| **Триггер**   | 72ч с момента постановки резерва + НЕ получен webhook check_printed от 1С (IC-CK-05) |

**1С-задача:** 1С-CK-02 (/cancel_reserve) — реализована в ТЗ блока Checkout.

**Payload (CRM → 1С):**

{

"CRM_Deal_ID": "12345"

}

**Ожидаемый ответ (1С → CRM):**

| **HTTP** | **status**        | **Действие CRM**                                                              |
|----------|-------------------|-------------------------------------------------------------------------------|
| 200      | cancelled         | Резерв снят. Витрина обновится через Fast Track (≤5 мин). Сделка → «Отменена» |
| 200      | already_cancelled | Уже отменён. CRM логирует, без повторных действий (идемпотентность 1С-CK-02)  |
| 200      | already_closed    | Заказ уже выполнен. CRM логирует, не меняет статус                            |
| 404      | order_not_found   | CRM логирует, уведомляет менеджера                                            |

**Post-action:** После cancelled — пары \[GUID, Store_ID\] из отменённого заказа пишутся в Очередь_Fast_Track. Стандартный 5-минутный батч обновляет витрину (DEC-CK-04).

### IC-CK-05: check_printed_webhook — 1С уведомляет CRM о пробитом чеке

| **Параметр**  | **Значение**                                                     |
|---------------|------------------------------------------------------------------|
| **Событие**   | Кассир пробил Чек ККМ для e-com заказа                           |
| **Инициатор** | \[1С\] — подписка на событие ПослеПроведения документа «Чек ККМ» |
| **Откуда**    | 1С:УТ                                                            |
| **Куда**      | \[CRM\] Битрикс24 — endpoint CRM_Webhook_Path_CheckPrinted       |

**1С-задача:** 1С-CK-03 (⚠️ OQ-CK-07: сценарий А или Б — зависит от ответов List.kg на 3 вопроса в ТЗ).

**Payload (1С → CRM):**

{

"CRM_Deal_ID": "12345",

"event": "check_printed",

"receipt_id": "КЧКМ-000456",

"timestamp": "2024-01-15T14:30:00"

}

**Ожидаемый ответ (CRM action):**

1.  Проверить по b_dom_processed_1c_events (ключ: CRM_Deal_ID + event) — идемпотентность

    - Если дубль → HTTP 200 без повторных действий

2.  Сбросить таймер 72ч-робота автоотмены резерва

3.  Перевести сделку в статус «Выполнен / Получен»

4.  Вернуть HTTP 200

**Критически:** Ошибка или недоступность CRM → 1С пишет WARN в журнал, **чек проводится в любом случае** (кассовая операция не блокируется).

**Условие отправки вебхука (не для всех чеков!):**

- Сценарий А: ДокументОснование = Заказ покупателя + УФ_CRM_Deal_ID заказа не пустой

- Сценарий Б (рекомендуемый): УФ_CRM_Deal_ID в самом «Чеке ККМ» не пустой

- Обычный розничный чек (без CRM-связки) → вебхук не отправляется

### IC-CK-06: WA-уведомления клиенту — исходящие (CRM → Client)

| **Параметр**    | **Значение**                             |
|-----------------|------------------------------------------|
| **Инициатор**   | \[CRM\] CRM-робот или триггер события    |
| **Откуда**      | Битрикс24 CRM                            |
| **Куда**        | WhatsApp Business API → клиент           |
| **Ограничение** | Только pre-approved HSM-шаблоны (CON-10) |

**Точки отправки:**
