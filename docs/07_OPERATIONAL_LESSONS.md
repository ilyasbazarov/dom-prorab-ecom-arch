# OPERATIONAL LESSONS — известные грабли инструментария

Не архитектурные решения (не ADR), а накопленные операционные факты о среде выполнения apply-скриптов. Читать перед написанием любого К-6-скрипта.

## L-01 — zsh bang-history ломает вставку блока с `!`
**Симптом:** `zsh: event not found: /usr/bin/env` при вставке apply-скрипта, начинающегося с `#!/usr/bin/env bash`.
**Причина:** apply-скрипт выполняется вставкой в уже открытый интерактивный шелл Owner'а (zsh на macOS), а не запуском файла. По умолчанию в zsh включён `histexpand` — символ `!` в интерактивной строке трактуется как обращение к истории команд, а не как текст/комментарий. Шебанг нужен только исполняемым файлам; для вставки в открытый шелл он не нужен и опасен.
**Фикс:**
1. Apply-скрипт НЕ содержит шебанг-строку.
2. Первая строка скрипта — `set +H` (отключает bang-history на сессию шелла) — страховка и от `!`, если он случайно попадёт в тело скрипта (например, внутри цитаты или комментария).

## L-02 — оборванная вставка оставляет «хвост» в буфере шелла
**Симптом:** `zsh: unknown file attribute: ^` при повторной вставке исправленного скрипта после того, как предыдущая попытка упала на середине heredoc.
**Причина:** если предыдущая вставка обрывается ошибкой (напр. L-01) до закрытия heredoc, часть текста остаётся в буфере интерактивного шелла и интерпретируется как начало новой команды (напр. `^` — zsh quick substitution).
**Фикс:** перед повторной вставкой скорректированного скрипта — сначала привести шелл к чистому промпту (Ctrl-C, при необходимости Enter на пустой строке), и только затем вставлять новый блок целиком одним куском.

## L-03 — множественные top-level heredoc-переходы в одной вставке рвут парсинг zsh
**Симптом:** `zsh: parse error near ')'` (или аналогичное) в середине вставки apply-скрипта, содержащего несколько последовательных top-level heredoc-блоков (`cat <<EOF` ... `python3 <<PYEOF` ... и т.д.), вставленных напрямую в интерактивный шелл.
**Причина:** apply-скрипт исполняется вставкой в уже открытый интерактивный шелл (не запуском файла). При большом объёме текста с несколькими независимыми heredoc-границами на верхнем уровне терминал/ZLE может разорвать поток ровно на границе между heredoc-блоками; хвост вставки интерпретируется как новые самостоятельные команды шелла Owner'а.
**Фикс:** весь К-6-скрипт оборачивается в ОДИН внешний heredoc, отдаваемый под-процессу `bash`: `bash <<'K6_SCRIPT' ... K6_SCRIPT`. Интерактивный zsh должен распознать ровно одну границу и накапливает сырой текст без попытки его разобрать; все вложенные heredoc обрабатывает уже дочерний bash в пакетном режиме. Побочный эффект в плюс: `exit 1` внутри скрипта завершает только под-процесс, а не сессию шелла Owner'а.

## L-04 — `python3 << 'HEREDOC'` с кириллицей падает вне tty
**Симптом:** `SyntaxError: Non-UTF-8 code starting with '\xd1' in file <stdin> on line N, but no encoding declared` при запуске Python 3-блока внутри apply-скрипта.
**Причина:** исходник Python читается из stdin (heredoc), а не из файла. Вне интерактивного tty (внутри вложенного под-шелла `bash <<'K6_SCRIPT'`, пайпа) CPython не всегда доверяет UTF-8-локали окружения и включает тот же механизм, что применяется к файлам без объявленной кодировки (PEP 263) — первый же кириллический байт валит компилятор.
**Фикс:** Python-код в К-6 никогда не подаётся напрямую в `python3 <<EOF`. Вместо этого он пишется heredoc'ом во временный `.py`-файл (первая строка файла — `# -*- coding: utf-8 -*-`), затем запускается `python3 /tmp/....py`. Дополнительная страховка — `export PYTHONUTF8=1` и `export PYTHONIOENCODING=utf-8` в начале скрипта.

## Правило для всех будущих apply-скриптов (К-6)
- Без шебанг-строки.
- Первая строка тела — `set +H`.
- После любого сбоя вставки — чистый промпт перед повторной попыткой (Ctrl-C + Enter), не поверх недовставленного хвоста.
- Весь скрипт — единый внешний heredoc `bash <<'K6_SCRIPT' ... K6_SCRIPT'`; никаких нескольких top-level heredoc-переходов, вставляемых напрямую в интерактивный шелл (L-03).
- Любой Python-код — только через временный `.py`-файл с `# -*- coding: utf-8 -*-` и `PYTHONUTF8=1`/`PYTHONIOENCODING=utf-8`, никогда напрямую в `python3 <<EOF` (L-04).
