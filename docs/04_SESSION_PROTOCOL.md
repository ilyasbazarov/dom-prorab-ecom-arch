# 04_SESSION_PROTOCOL — Жизненный цикл сессий и session-блок

**Статус:** STABLE (изменения через ADR). Основание: ADR-007.

## Типы сессий
| Тип | Промт | Выход |
|---|---|---|
| Business | `prompts/business.md` | решения `[BUSINESS]` + session-блок |
| Architect (режим A: решения/ревью/роудмап) | `prompts/architect.md` | ADR + патчи + session-блок |
| Architect (режим B: брифы) | `prompts/architect.md` | `briefs/<ID>.md` + session-блок |
| Executor | бриф + `prompts/executor.md` | артефакт задачи + session-блок |
| Applier | `prompts/applier.md` | ОДИН commit-скрипт |

Штатная архитекторская сессия завершается пакетом «решения + патч роудмапа + брифы под следующие задачи» — два режима в одной сессии допустимы и предпочтительны (экономия загрузки контекста).

## Ритуал старта (любая роль)
1. Получить от Owner строку `SHA сессии: <хэш>`. SHA нет → запросить, `main` не читать.
2. Загрузить обязательный контекст роли (см. `03_ROLES`) — только curl по SHA.
3. Переформулировать задачу своими словами, перечислить полученные входы.
4. Неполнота → `CONTEXT GAP: <что отсутствует и почему нельзя продолжать>`. Стоп, не догадка.

## Ритуал конца: session-блок (без него сессия не закрыта, знание потеряно)

```
=== SESSION LOG · <дата> · <тип/ID> ===

## SESSION_LOG   (→ sessions/<дата>_<id>.md)
- Задача: ...  - Сделано: ...  - Отклонения: <или "нет">

## STATE_PATCH   (→ docs/00_STATE.md)
- Фаза/задача: <старый статус> → <новый>
- Текущий фокус: ...
- Новые открытые вопросы / блокеры: <или "нет">

## NEW_DECISIONS (→ append docs/02_ADR_LOG.md; "нет" если нет)
- ADR-0XX [BUSINESS|ARCH|PROCESS]: <заголовок>
  Контекст · Решение · Последствия (полный blast radius: файл/§)
  Статус: accepted | proposed

## REGISTER_PATCHES (→ l2-registers/*; "нет" если нет)
- <файл регистра>: <операция: добавить запись / сменить статус P-xx / закрыть вопрос> + дословный контент

## NEW_CONVENTIONS (→ docs/01_CONVENTIONS.md; по умолчанию proposed)

=== END SESSION ===
```

Отличие от базиса holika: добавлена секция `REGISTER_PATCHES` — в этом проекте регистры (Problem, Contractor Questions, Open Questions, Assumptions) — основной рабочий продукт.

## Правила применения (для Applier)
- `STATE_PATCH` — применяется всегда. `NEW_DECISIONS` — только `accepted`; номер ADR = следующий по факту в файле, не из блока. `proposed` — вывести списком «требует апрува», не вносить.
- `NEW_CONVENTIONS` — только при явном «одобряю» от Owner.
- ADR-лог и регистры — append/статусные правки; существующие записи не переписываются (P-xx меняет статус, не текст доказательств — новые доказательства добавляются).
- Патч ссылается на несуществующую структуру → `СТРУКТУРНОЕ НЕСООТВЕТСТВИЕ`, стоп.
- Выход Applier — ОДИН bash-блок (К-6): записывает файлы, `git add -A && git commit && git log --format='%H' -1`.

## Правило явного патча (для Business/Architect/Executor)
Адресат патча не видит твой thinking. Каждый патч — drop-in: точный файл, точное место, тип операции (вставка/замена/append/новый файл), дословный контент. Ссылки «по аналогии» запрещены. В конце сессии — сводка «файл → операция».

## Гейт целостности
После каждого apply-коммита Owner запускает `bash tools/verify.sh`. RED → следующая сессия начинается с починки, новые задачи не стартуют.

## Definition of Done
- Executor-задача: артефакт принят по критериям брифа + session-блок применён + verify GREEN. Для скриптовых задач — только после подтверждённого Owner лога реального прогона.
- Architect-сессия: все поднятые вопросы либо решены (ADR), либо явно в Open Questions с владельцем.
- Business-сессия: каждая развилка закрыта решением или явным DEFER с датой возврата.

## Доступ (жёстко, из holika-базиса)
Только `curl` по commit-SHA: `https://raw.githubusercontent.com/ilyasbazarov/dom-prorab-ecom-arch/<SHA>/<путь>`. Web-fetch/браузер запрещён (кэш). Двойной слэш → HTTP 400. api.github.com без авторизации лимитируется — SHA даёт Owner (он у него всегда в буфере: apply-скрипт печатает его последней строкой).
