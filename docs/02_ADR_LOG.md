## ADR-008 [PROCESS] — Git-дисциплина и санкционированные исключения verify
**Дата:** 2026-07-18. **Статус:** ACCEPTED

## ADR-009 [ARCH] — Разведение клиентского референса заказа и номера документа 1С
**Дата:** 2026-07-22. **Статус:** ACCEPTED (Owner, сессия BUS-OQ01)

Gate: Business-ADR по OQ-01 (платёжная модель + подтверждение потребности в клиентском референсе) — снят, см. ADR-010.
Контекст: UF_1C_ORDER_NUMBER по дизайну существует только при status=success (d4_t02_cross_sell_bundle_korzina_i_chekaut.md@…:613,627; d1_t03_podtverzhdenie_i_ozhidanie_processing.md@…:99-101). Использование его как единственного клиентского референса оставляет клиента без идентификатора при partial/rejected (P-19).
Решение: клиентский референс заказа генерируется в CRM/CMS при создании сделки и является видимым клиенту номером; UF_1C_ORDER_NUMBER — вторичный реквизит для внутренних сверок 1С↔CRM. 1С остаётся мастером операционного документа; красные линии (CMS→1С запрещено; CRM — единственный шлюз) не затрагиваются.
Последствия (blast radius при accepted): Canonical Order Flow (Фаза 4, §идентификаторы); d4_t02 §15.2 (источник клиентского номера в обработке /create_order); интеграционный контракт (§поля сделки); problem_register P-19 → CONFIRMED + задача Bitrix.

## ADR-010 [BUSINESS] — Платёжная модель e-com и клиентский номер заказа
**Дата:** 2026-07-22. **Статус:** ACCEPTED (Owner, сессия BUS-OQ01)

Контекст. P-19 (CONFIRMED-DOC): номер документа 1С существует только при success; при partial/rejected клиенту нечего назвать в поддержке. Для доставки корпус фиксирует порядок «оплата → /create_order», т.е. нехватка наступает после списания денег. ЛК (D6 Эпик 21) требует показывать номер заказа на стадии «Ожидает подтверждения», когда документа 1С не существует. Арх-рамка ADR-009: клиентский референс в CRM/CMS не нарушает L0-инварианты, 1С остаётся мастером операционных данных.

Решение.
Платёжная модель — гибрид по типу получения, как зафиксировано в D6: доставка — 100% предоплата товарной части заказа (карта/QR), /create_order только после платёжного вебхука; самовывоз — оплата на кассе, /create_order только после подтверждения менеджера. Ратифицируется как бизнес-канон. Уточнение (Owner, 2026-07-22): стоимость доставки в предоплату не входит (CON-06) — заказ в Яндекс оформляется в режиме оплаты «наличные», клиент рассчитывается с водителем при получении (нал / p2p QR / перевод по номеру телефона).
Вводится клиентский номер заказа: присваивается системой (CRM/CMS) в момент фиксации заказа на чекауте, до платёжного события и до ответа 1С, при любом типе получения и любом исходе.
Клиентский номер — единственный идентификатор заказа во всех клиентских точках контакта (thank-you, WA/SMS, ЛК, поддержка, возвраты). Номер/GUID 1С — внутренние; связка хранится в сделке. order_id HSM-шаблонов = клиентский номер.
Коммуникация при нехватке: partial — по AC 17.1/17.3/17.4 (звонок менеджера, замена, partial/полный refund при предоплате); rejected без предоплаты — WA-отмена (AC 17.2); rejected с предоплатой — проактивный полный refund без обращения клиента, WA с номером, отменой и фактом инициированного возврата; целевой срок инициирования ≤ следующего рабочего дня [эвристика до доков Бакай]. MVP-механика возврата — ручная через кабинет эквайринга (DEC-53).
Формат номера: человекочитаемый, диктуемый по телефону, визуально отличимый от нумерации 1С. Реализация — решение Architect.

Последствия (blast radius). Финализирует P-19; активирует ADR-009 [ARCH]; вход в Canonical Order Flow (Фаза 4, l0-canon/). ТЗ: D4 чекаут/CRM (генерация и вывод номера — экстракост Bitrix-подрядчика по ADR-005, флаг «замена несделанного»), D4 t07 HSM-шаблоны (подстановка клиентского номера), Эпик 21 ЛК (источник поля «номер» = клиентский номер). 1С-контракты не затронуты. Регистры: OQ-01 → RESOLVED; новый OQ-02 (DEFER до доков Бакай Банка).

## ADR-011 [ARCH] — Структура Knowledge Repository (L0-канон) и трактовка проектных реестров решений (DEC/COM/CON)

## ADR-012 [ARCH] — Формат и генерация клиентского номера заказа
**Дата:** 2026-07-22. **Статус:** ACCEPTED (делегация ADR-010:60, сессия arch_phase4_orderflow_clientno)

Контекст. ADR-010:60 делегировал реализацию формата клиентского номера заказа Architect'у. Номер 1С (`purchase_order_number`→`UF_1C_ORDER_NUMBER`) существует только при `status=success` (d4_t02@65f2bba:613,627); при `partial`/`rejected` поля номера в ответе `/create_order` нет (d1_t03@65f2bba:99-102). HSM-шаблоны `d4_t07` привязывают клиентское «Заказ №{{1}}» к `UF_1C_ORDER_NUMBER` во всех клиентских точках, включая шаблоны нехватки/замены (d4_t07@65f2bba:348-355,365-372), где поле критическое, но пустое при partial/rejected — открытый дефект (P-19).

Решение. Каноническое поле — реквизит сделки CRM `UF_CLIENT_ORDER_NO` (строка), единственный клиентский идентификатор (INV-4). Формат: `E-YYMMDD-NNNN-C` (пример `E-260722-0043-8`): `E-` — литеральный маркер онлайн-канала (нумератор 1С его не эмитит → отличимость, ADR-010:60); `YYMMDD` — дата фиксации заказа, Asia/Bishkek; `NNNN` — сквозной дневной счётчик по онлайн-каналу, обнуление 00:00 Asia/Bishkek, zero-pad 4; `C` — контрольная цифра Luhn mod-10 по телу `YYMMDD+NNNN`. Генерация — CRM (владелец сделки), атомарно при создании сделки на чекауте, до платежа и до ответа 1С (d6_t03@65f2bba:159,180; ADR-010:57). Единственный писатель. Идемпотентность: mint привязан к чекаут-токену; ретрай создания сделки возвращает существующий номер, второй не выпускается (в духе INV-5). Присваивается при любом исходе `/create_order` и любом типе получения (INV-4).

Последствия (blast radius). Новый реквизит CRM `UF_CLIENT_ORDER_NO`; HSM `{{1}}` во всех клиентских шаблонах d4_t07 #4–#10 → rebind `UF_1C_ORDER_NUMBER` → `UF_CLIENT_ORDER_NO` (закрывает P-19 на #9/#10); ЛК «номер» (d6_t04@65f2bba:160,168-173) ← `UF_CLIENT_ORDER_NO`; l0-canon/{invariants,domain_model}.md (провенанс += ADR-012); l0-canon/canonical_order_flow.md §1. `UF_1C_ORDER_NUMBER`/`purchase_order_guid` остаются внутренними (INV-3, ADR-009). Экстракост Bitrix уже санкционирован (ADR-010, флаг «замена несделанного»). Открытый (не блокирующий) вопрос подрядчику по маске нумератора 1С — CQ-01 (DRAFT, l2-registers/contractor_questions.md).
