# CONVENTIONS

- **К-1. Доказательность.** Ни один вывод не имеет статуса CONFIRMED без цитаты из L3-снапшота с прямым побайтовым доступом (SHA + файл + строки). Пересказы третьих лиц доказательством не являются.
- **К-2. Приоритет канона.** Конфликт L0 vs L3 трактуется как дефект документации подрядчика. Изменение L0 — только через ADR.
- **К-3. Immutable снапшоты.** L3-файлы после коммита не редактируются. Новая версия документа = новый файл с датой. Diff строится машинно от снапшота.
- **К-4. Минимальная достаточность.** Артефакты (Knowledge Graph, Bounded Context Map и пр.) вводятся только при доказанной необходимости, не «по лучшим практикам».
- **К-5. Протокол KB.** Из Master KB без подтверждения используются только секции [1], [2], [9]. Остальное — тезис агента → явное подтверждение/корректировка владельца в сессии.
- **К-6. Скрипты.** Все скрипты — единый copy-paste-ready bash-блок без ручных плейсхолдеров.
- **К-7. Оракул 1С.** Статусы подтверждения: CONFIRMED-DOC (видно в тексте ТЗ) / CONFIRMED-FACT (подтверждено List.kg). Отсутствие в ТЗ = «подрядчик не обязан», а не «система так не делает».

- **К-8. Session-блок обязателен.** Любая сессия (Business/Architect/Executor) закрывается одним session-блоком по 04_SESSION_PROTOCOL. Незакрытая сессия = потерянное знание.
- **К-9. Verify-гейт.** После каждого apply-коммита Owner запускает `bash tools/verify.sh`. RED → следующая сессия начинается с починки; новые задачи не стартуют.
- **К-10. SHA сессии.** Агент работает только по commit-SHA, приложенному Owner. SHA не приложен → агент обязан запросить, чтение main запрещено.
