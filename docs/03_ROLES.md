# 03_ROLES — Роли, ответственность, биндинг моделей

**Статус:** STABLE (изменения через ADR). Основание: ADR-007.
Метод-базис: дистилляция `_METHOD` v1.0 из `ilyasbazarov/holika@ec7e6cc` + адаптация под L0–L3 и трёх внешних подрядчиков.

## Принципы
- **Роль — не чат.** Свежий чат становится ролью, загрузив её промт (`prompts/<роль>.md`) + обязательный контекст. Старые чаты ретайрятся.
- **Один файл — одна роль.** Промт роли самодостаточен.
- **Ни один чат не держит весь проект.** Знание живёт в репо; чат берёт срез.
- **Роли определяются функцией, не моделью.** Биндинг моделей — таблица ниже, меняется одной строкой без правки методологии.

## Роли

| Роль | Делает | НЕ делает | Обязательный контекст |
|---|---|---|---|
| **Business (CEO-сессия)** | принимает бизнес-решения: семантика Order Flow для клиента и денег, политика (отмены, возвраты, SLA, коммуникация), приоритеты, аппетит к косту/риску, приёмка Business Canon | не проектирует технические решения, не выбирает реализацию, не правит контракты API | `01`+`00`+`05_ROADMAP`+открытые вопросы с тегом BUSINESS (+KB твёрдые секции, +D6 по вопросу) |
| **Architect** | ADR, целостность L0, ревью регистров и предложений, роудмап; **режим B:** выпуск брифов | не пишет прод-код, не коммитит, не принимает бизнес-решений (эскалирует в Business) | `01`+`00`+`02`+`05_ROADMAP` (+MANIFEST, +регистры, +фрагменты по вопросу) |
| **Executor** | исполняет ОДИН бриф: верификация проблем против снапшотов, аналитика, скрипты, черновики задач подрядчикам | не правит L0/конвенции по своей инициативе, не принимает решений — гэп поднимает наверх | `01`+`00`+бриф (бриф несёт context-to-load) |
| **Applier** | механически конвертирует session-блок в один commit-скрипт | никакого творчества, не «улучшает» формулировки | `prompts/applier.md`+session-блок+затрагиваемые файлы |
| **Owner (человек)** | запускает сессии, исполняет скрипты, коммитит, гоняет `tools/verify.sh`, владеет апрувами | — | — |

| Роль | Модель |

Смена биндинга = правка этой таблицы (без ADR, это не изменение метода).

## Граница Business ↔ Architect
Вопрос звучит как «что правильно для клиента/денег/бизнеса» → Business.
Вопрос звучит как «как это устроить в системах, не сломав инварианты» → Architect.
Architect обязан эскалировать в Business любой выбор, меняющий клиентский опыт, движение денег или обязательства перед подрядчиками сверх бюджета. Business-решения оформляются ADR с тегом `[BUSINESS]` и являются ограничениями для Architect (входят в L0-контур).

## Эскалация
Executor → Architect: любой CONTEXT GAP, противоречие, соблазн додумать.
Architect → Business: бизнес-развилки.
Business/Architect → Owner: развилки с равновесными вариантами, платные последствия, всё необратимое.
