## BITRIX-B1-01 (P-01) — Переход «Ожидает сборки» по HTTP-коду вместо поля `status`

**Проблема.** ТЗ противоречит само себе: «Гарантия 1С» утверждает `HTTP 200` = факт
проведения с резервом ≠ 0 `[DEC-33]`, но таблица ответов того же контракта допускает при
`HTTP 200` в теле `status=partial`/`rejected` — заказа/резерва нет. Робот на HTTP-коде
переведёт сделку в «Ожидает сборки» при нехватке; кладовщик получит задание на
несуществующий заказ.

**Контекст.**
- Гарантия: `d1_t03@bb88448:91` («HTTP 200 … резерв ≠ 0 `[DEC-33]`»).
- Опровержение (тот же файл): `d1_t03@bb88448:99-101` — `200/success`→(guid,number);
  `200/partial`→«Требует уточнения» `[COM-PR-04]`; `200/rejected`→«Отменена (OOS)».
- Существующая задача: обработка ответа `/create_order` на чекауте, переход сделки по исходу.

**Изменение.**
- Триггер «Ожидает сборки» (PICKING) = `HTTP 200` И `body.status=="success"` И
  `body.purchase_order_guid != null`. HTTP-код сам по себе стадию не двигает.
- Формулировку гарантии `DEC-33` заменить: «HTTP 200 — техническое подтверждение доставки
  запроса; бизнес-результат определяется строго по полю `status` в теле ответа»
  (согласовать с владельцем 1С-контракта — `1С-PR-01/02`).

**Критерии приёмки.**
- `200/partial` и `200/rejected` не инициируют «Ожидает сборки».
- Только `200/success` + непустой `purchase_order_guid` инициирует PICKING.
- Текст гарантии не утверждает «200 = резерв поставлен».

**Трассировка:** `d1_t03@bb88448:91` vs `:99-101`. Зависимости: нет.

## BITRIX-B1-03 (P-03) — Идемпотентность по (сделка, событие) не отбрасывает устаревшие события другого типа

**Проблема.** `INF-PR-01` проверяет дубль по `(CRM_Deal_ID, event)` и при «новом» безусловно
применяет событие. Устаревшее событие ДРУГОГО типа (напр. `order_cancelled` со старой
версией, пришедшее после `order_in_progress`) применяется поверх нового — сделка откатывается
в неверный статус.

**Контекст.**
- Алгоритм: `d4_t03@bb88448:94-98` (ключ `(CRM_Deal_ID, event)`; «новый → записать,
  продолжить», без упорядоченности между типами).
- Существующая задача: `INF-PR-01`, AC-обработчика шаги 2–3.
- Требует поле `doc_version` (контракт) — эмитится 1С-задачей `1C-B1-05`.

**Изменение.**
- Курсор `last_applied_doc_version` **на сделку**. Событие с `doc_version <= курсор`
  отбрасывается НЕЗАВИСИМО ОТ ТИПА; иначе применить и обновить курсор.
- **Graceful degradation:** если `doc_version` в payload ОТСУТСТВУЕТ → текущее поведение
  (применить); если присутствует → версионное отбрасывание. Это позволяет выкатиться
  независимо от 1С и активировать версионность, когда 1С начнёт слать поле.
- Дедупликация — по `event_id` (UUID), упорядочивание — по `doc_version`: два разных
  механизма, не схлопывать. `timestamp` для упорядочивания НЕ использовать.

**Зависимость (мягкая, контрактная).** Реализуется ПАРАЛЛЕЛЬНО `1C-B1-05` по замороженному
контракту `doc_version`. Жёсткого blocked-by нет — благодаря graceful degradation Bitrix
может задеплоиться первым (в degraded-режиме).

**Критерии приёмки.**
- `order_in_progress(5)` применён; затем `order_cancelled(3)` → отброшен, сделка «В работе».
- Событие с `doc_version > курсор` любого типа применяется и двигает курсор.
- **Degraded-режим:** при отсутствии `doc_version` поведение = текущее, без ошибок.
- Дедупликация по `event_id` сохранена отдельно от версионного отбрасывания.

**Трассировка:** `d4_t03@bb88448:94-98`. Контракт: `doc_version`. Соотнесено с `1C-B1-05`.

## BITRIX-B1-06 (P-06) — Обработчик ждёт `order_confirmed`, которого 1С не отправляет

**Проблема.** `INF-PR-01` обрабатывает `order_confirmed` для перехода «В работе / Ожидает
сборки». Спецификация 1С отправляет ровно 4 события; `order_confirmed` среди них нет, для
«В работе» шлётся `order_in_progress`. Ветка `order_confirmed` мёртвая.

**Контекст.**
- Ожидание Bitrix: `d4_t03@bb88448:60,72,86,104` (`order_confirmed`).
- Спецификация 1С: `d5_t01@bb88448:1449-1456` — `order_completed/cancelled/partial/in_progress`;
  `order_confirmed` отсутствует.
- Существующая задача: `INF-PR-01` (маршрутизация по `event`).

**Изменение.**
- Переименовать `order_confirmed` → `order_in_progress` везде в `INF-PR-01`: payload-пример
  (`:72`), триггеры (`:86`), маршрутизация (`:104`), заголовок (`:60`). Действие ветки
  сохранить под именем `order_in_progress`.
- Имена событий согласовать с `BITRIX-B1-03` (ключ идемпотентности использует `event`).

**Критерии приёмки.**
- Обработчик реагирует на `order_in_progress` переходом «В работе / Ожидает сборки»; ветки
  `order_confirmed` нет.
- Набор событий = `{order_in_progress, order_cancelled, order_completed}`
  (+ `oos_at_picking` на отдельном endpoint).

**Трассировка:** `d4_t03@bb88448:60,72,86,104` vs `d5_t01@bb88448:1449-1456`. Зависимости:
имена ↔ `BITRIX-B1-03`.
