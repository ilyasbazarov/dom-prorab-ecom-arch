## КОНТРАКТ ПОЛЯ `doc_version` (Architect-owned, канон — общий для 1C-B1-05 и BITRIX-B1-03)

- Реквизит-источник 1С: `УФ_DOC_VERSION`; поле в payload вебхука: `doc_version`.
- Тип: целое неотрицательное.
- Семантика: монотонно неубывающий счётчик записей документа «Заказ покупателя»
  **в пределах одного заказа** (одного `CRM_Deal_ID`). Глобальной монотонности между
  заказами не требуется.
- Присутствие: во всех событиях статус-маппинга и в `oos_at_picking`.
- Forward-compat (см. OQ-03): при переходе на очередь исходящих событий роль монотонного
  ключа берёт порядковый номер записи очереди; имя и семантика поля `doc_version` в payload
  сохраняются — потребитель (Bitrix) не переписывается.

## 1C-B1-02 (P-02) — Подавление статус-вебхуков по `УФ_CRM_Deal_ID` заглушит легитимные события

**Проблема.** Предложение подавлять исходящий статус-вебхук по наличию `УФ_CRM_Deal_ID`
неверно: реквизит заполнен у КАЖДОГО e-com заказа и является атрибутом документа; подписка
на `ПриЗаписи` срабатывает независимо от источника. Гейт по нему заглушит легитимные
события (ручная отмена, отгрузка/выполнение, нехватка при сборке).

**Контекст (предпроверка закрыта).**
- Реквизит: `d5_t06@bb88448:30` (`УФ_CRM_Deal_ID` — ключ идемпотентности всех эндпоинтов,
  ставится единожды). Подписка: `d5_t06@bb88448:254` (`СТ_ВебхукСтатус`, `ПриЗаписи`,
  «при смене статуса», независимо от источника).
- Эхо-путь РЕАЛЕН: `/cancel_reserve` всегда инициируется CRM (`d5_t02@bb88448:183`) и меняет
  статус на «Отменён» (`d5_t02@bb88448:217-223`) → `order_cancelled` уходит обратно в CRM.
- `/update_order` статус НЕ меняет (`d5_t03@bb88448:164,204`) → эха не даёт.
- `order_completed` — **load-bearing**: CRM потребляет его (стадия «Успешно реализовано» +
  триггер WA, `d4_t03@bb88448:106`). Подавлять на источнике НЕЛЬЗЯ.
- Подавление эха УЖЕ обеспечено на стороне CRM: идемпотентность `CRM-LM-06`
  (`d5_t01@bb88448:1484`, ключ `(CRM_Deal_ID, event)`) + анти-петля INF-PR-01
  (`d4_t03@bb88448:105-106`). Эхо не приводит к неверному состоянию.

**Изменение (документационное — 0 кода 1С).**
- Запретить в ТЗ гейт/подавление исходящих вебхуков по `УФ_CRM_Deal_ID`.
- Зафиксировать: 1С отправляет статус-вебхук при каждой смене статуса; подавление эха — зона
  CRM (`CRM-LM-06` + анти-петля INF-PR-01). Дополнительной суппрессии на стороне 1С не вводится.
- Явно: `order_completed` на источнике не подавляется.

**Критерии приёмки.**
- В ТЗ нет требования подавлять вебхуки по `УФ_CRM_Deal_ID`.
- В ТЗ закреплено: подавление эха — на стороне CRM; 1С шлёт вебхук при каждой смене статуса.
- Легитимные события (ручная отмена, `oos_at_picking`, `order_completed`) уходят без изменений.

**Трассировка:** `d5_t06@bb88448:30,254`; `d5_t02@bb88448:183,217-223`; `d5_t03@bb88448:164,204`;
`d4_t03@bb88448:105-106`; `d5_t01@bb88448:1484`. **1С-разработка: ноль.** Зависимости: нет.

## 1C-B1-05 (P-05) — Нет монотонной версии документа для отбрасывания устаревших событий Bitrix'ом

**Проблема.** Корректная обработка статус-вебхуков на стороне Bitrix (`BITRIX-B1-03`)
требует монотонного признака свежести. В 1С его нет: реквизиты «Заказа покупателя» — только
`УФ_CRM_Deal_ID`, `УФ_SOURCE_STORE_ID`; payload несёт `CRM_Deal_ID, purchase_order_guid,
purchase_order_number, event, timestamp` — без версии. Вебхук асинхронный и не блокирует
запись (`d5_t01@bb88448:1481`) → доставка возможна вне порядка → упорядочивание необходимо.

**Контекст.**
- Реквизиты: `d5_t06@bb88448:30-31`. Payload: `d5_t01@bb88448:1458-1470` (версии нет;
  `timestamp` как монотонный ключ непригоден). ENABLER для `BITRIX-B1-03` (P-03).

**▶ РЕШЕНИЕ НА СЕЙЧАС (Track B, минимальное).**
- Добавить реквизит `УФ_DOC_VERSION` (см. контракт `doc_version` вверху файла).
- Инкремент при КАЖДОЙ записи документа **внутри транзакции записи, с блокировкой объекта**
  — чтобы конкурентные записи не давали коллизий (два сеанса читают N → оба пишут N+1).
- Включить значение в payload (`doc_version`) на всех событиях статус-маппинга.

**◼ ЦЕЛЕВОЕ — ОТЛОЖЕНО, ЗАФИКСИРОВАНО (OQ-03 / P-16). НЕ ДЕЛАЕМ СЕЙЧАС.**
- Транзакционная очередь исходящих событий (регистр сведений + регламентное задание):
  порядковый номер записи очереди становится монотонным ключом → снимает нужду в курсоре
  на стороне Bitrix, решает порядок доставки, гарантию доставки и подавление эха (P-02).
- `УФ_DOC_VERSION` спроектирован forward-compatible: при переходе имя/семантика поля
  `doc_version` в payload сохраняются, потребитель не переписывается.
- Решение о переходе — открытый вопрос **OQ-03** (после MVP).

**Критерии приёмки (на сейчас).**
- `УФ_DOC_VERSION` растёт при каждой записи; инкремент транзакционно-безопасен (блокировка
  объекта; конкурентные записи без коллизий).
- payload всех статус-событий содержит `doc_version`.
- Монотонность гарантирована в пределах одного заказа.

**Трассировка:** `d5_t06@bb88448:30-31`; `d5_t01@bb88448:1458-1470,1481`. Разблокирует:
`BITRIX-B1-03`. Целевое: `OQ-03` (P-16).
