# CANONICAL ORDER FLOW — L0-канон

Канонический жизненный цикл заказа e-com. Каждая P-xx и любое ТЗ верифицируются против
этого потока. Изменение — только новым ADR. Цитаты корпуса — @65f2bba (SHA сессии
наполнения); provenance обязателен: L3 → `файл@SHA:строки`; наше решение → `ADR-xxx`.
Политики (отмены/возвраты/SLA/платёжная механика) — `business_canon.md` (владелец Business);
здесь — только структура потока и точки, где политика подключается.

## §0 Красные линии (наследуются из invariants.md)
- CMS→1С запрещено; единственный путь CMS↔1С — через CRM. [INV-1]
- 1С — мастер операционной истины; клиентский номер — не операционный документ. [INV-2, INV-4]
- Семантика этого потока — Track A; не уходит подрядчикам до утверждения канона. [ADR-005]

## §1 Идентификаторы в потоке
- **Клиентский номер** `UF_CLIENT_ORDER_NO` (формат `E-YYMMDD-NNNN-C`) — присваивается CRM при
  создании сделки на чекауте, до платежа и до ответа 1С, при любом исходе. Единственный ID во
  всех клиентских точках (thank-you, WA/SMS, ЛК, поддержка, возвраты); `order_id` HSM = он. [ADR-012, INV-4]
- **Внутренние (сверка, клиенту не показываются):** `CRM_Deal_ID` (идемпотентность 1С-эндпоинтов),
  `purchase_order_guid`/`purchase_order_number`→`UF_1C_ORDER_NUMBER`, `closing_doc_guid`,
  `return_id`, `payment_id`. [INV-3, ADR-009; d5_t01@65f2bba:1451-1456]

## §2 Гейты фиксации (перед постановкой резерва) [ADR-010]
- **Доставка:** `/create_order` — ТОЛЬКО после подтверждённого платёжного вебхука
  (100% предоплата товарной части; стоимость доставки НЕ входит, оплата водителю — CON-06).
  [d6_t03@65f2bba:159,165-168,88,96]
- **Самовывоз:** `/create_order` — ТОЛЬКО после явного подтверждения менеджера; оплата на кассе.
  [d6_t03@65f2bba:180,188-190]
- Автоматического вызова `/create_order` при оформлении НЕТ ни в одном сценарии. [d6_t03@65f2bba:159,180]

## §4 Переходы (канон)
**T0 · Чекаут → фиксация.** Сделка создаётся в CRM; присваивается `UF_CLIENT_ORDER_NO` (§1);
thank-you показывает его. [ADR-012; d6_t03@65f2bba:188]

**T1 · Гейт → /create_order.** По §2. Payload `{CRM_Deal_ID, Client_Phone, Store_ID, Items[]}`.
HTTP 200 = строго факт проведения + резерв≠0 (DEC-33). Исходы: [d1_t03@65f2bba:99-102]
- `success` → сохранить `purchase_order_guid`/`UF_1C_ORDER_NUMBER`; стадия «В работе/Ожидает сборки».
- `partial` → стадия «Требует уточнения (Нехватка)»; задача менеджеру (замена/refund — политика, §7).
- `rejected` → стадия «Отменена (OOS)»; клиент — WA-отмена.
- `duplicate` (idempotent по `CRM_Deal_ID`) → лог, без повторных действий. [INV-5]
> Клиентский номер существует на всех четырёх ветках (T0 предшествует T1). [INV-4]

**T2 · События 1С→CRM.** Ровно 4: `order_completed` / `order_cancelled` / `order_partial` /
`order_in_progress`. `order_confirmed` НЕ существует (см. P-06). Идемпотентность вебхука — ключ
`order_id + event` (b_dom_processed_1c_events). [d5_t01@65f2bba:1453-1456; INV-5]

**T3 · Последняя миля.**
- Доставка: «Сборка завершена» → Яндекс claims/create → «Передан курьеру» → webhook `delivered`
  → `/close_order`. [d6_t05@65f2bba:33,35,118]
- Самовывоз: «Ожидает выдачи» → Чек ККМ (физ) `check_printed`→CRM→`/close_order`
  (idempotent) / Реализация (юр) → order_completed. [d6_t05@65f2bba:87-89,95-96]
- Закрытие → `{closing_doc_guid, closing_doc_number, closing_doc_type: check|realization}`;
  стадия «Успешно реализовано»; WA-накладная финальным составом. [d6_t05@65f2bba:119-123]

**T4 · Автоотмена резерва (самовывоз, 72ч).** CRM-робот → `/cancel_reserve` при отсутствии
`check_printed` за 72ч; 1С — идемпотентно. [d6_t03@65f2bba:199-211]

**T5 · Возвраты и рекламации — СТРОГО отдельные смарт-процессы (CON-18), объединение запрещено.**
[d6_t06@65f2bba:9,137,149]
- Возврат: `/create_return {return_id, closing_doc_guid, items[]}`; наличный — `/create_rko`.
  Идемпотентность по `return_id`. [d6_t06@65f2bba:50-58,101; INV-5]
- Рекламация: изоляция брака (Возврат+Списание в одной сессии, enum Writeoff_Reason). [d6_t06@65f2bba:161-169]
- Клиент во всех точках оперирует `UF_CLIENT_ORDER_NO` (поиск сделки — по телефону/номеру). [INV-4]

## §5 Идемпотентность внешних вызовов (наследуется из INV-5)
`CRM_Deal_ID`→эндпоинты 1С; `return_id`→create_return/create_rko; `UF_Purchase_Order_GUID`→close_order;
`payment_id`→вебхуки эквайринга; `order_id+event`→вебхуки 1С→CRM.

## §6 Инвариант клиентского номера в потоке
На всех исходах T1 (success/partial/rejected/duplicate) и обоих типах получения клиент имеет
`UF_CLIENT_ORDER_NO`, присвоенный в T0. Ни одна клиентская коммуникация не ссылается на
`UF_1C_ORDER_NUMBER`. [INV-4, ADR-012]
